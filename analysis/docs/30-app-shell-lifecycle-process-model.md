# App Shell, Lifecycle & Process Model

> Target: Meta native **WhatsApp for Windows** (`WhatsApp.Root.exe`, full-trust WinUI 3 / Windows App SDK 1.6, package version `2.2607.106.0`). This document covers the managed C# shell that boots the process, enforces single-instance, hosts the WebView2 that runs the WhatsApp Web JS bundle, wires the JS↔native bridges, and manages window / activation / lifecycle / suspend-resume.
>
> All paths in §2–§4 are relative to `decompiled_source/`. Every non-trivial claim cites `relative/path.cs:LINE` that was actually read. Inferences are explicitly marked **(inferred)**.

---

## 1. Purpose & Scope

The "App Shell" is the thin native Windows host whose job is to:

1. Start a single STA WinUI 3 process and enforce single-instance with redirection of secondary launches (`Program.cs`).
2. Construct the `App` application object, its model (`AppModel`), and its views (`ViewsController`) asynchronously off the UI thread.
3. Create a `MainWindow` (a `WinUIEx.WindowEx`) hosting one `WebView2`, navigate it to `https://web.whatsapp.com/` with a native-context query string, and lock the WebView down.
4. Register ~27 WinRT "bridge" host objects so the WhatsApp Web JS bundle can call native services (`AppModel.SetWebView`, `App.EnsureMainWindowInitialized`).
5. Handle OS activations: toast/notification, protocol (`whatsapp://`), share-target, jumplist, startup-task.
6. Manage lifecycle: show/hide to system tray, suspend/resume the web client, WNS push wakeups, controlled restarts, language/theme changes, crash/SAD recording.
7. Provide the app-wide threading/dispatcher model (`WhatsApp.Core` `Dispatcher`, `ConcurrentQueueDispatcher`, capturing dispatchers).

The actual chat/UI/business logic is **not** here — it lives in the WebView2 JS bundle. The shell is the process owner, window owner, activation router, and bridge registrar.

## 3. How It Works

### 3.1 Process entry & single-instance redirection (`Program.cs`)

`Main` is `[STAThread]` (`Program.cs:19`). Boot sequence:

```
Main:
  XamlCheckProcessRequirements()          // P/Invoke into Microsoft.ui.xaml.dll  (Program.cs:22)
  ComWrappersSupport.InitializeComWrappers()                                      // :23
  if (!DecideRedirection(args).Result)    // BLOCKING .Result on async            // :24
      Application.Start(cb)               // only the primary instance reaches here
```

`Application.Start` callback installs a `DispatcherQueueSynchronizationContext` for the current (UI) thread, then `new App()` (`Program.cs:26-30`). So the WinUI dispatcher queue is the UI `SynchronizationContext`.

**Single-instance** is via `AppInstance.FindOrRegisterForKey("61E9B470-92E6-482B-AFD8-38C1FB464FB5")` (`Program.cs:37`) — a fixed GUID instance key:

- If `appInstance.IsCurrent` → this is the primary: subscribe `appInstance.Activated += OnActivated` (`Program.cs:38-41`).
- Else → secondary: `isRedirect = true`; unless the command line contains `/pushnotification`, fetch this launch's `GetActivatedEventArgs()` and `await appInstance.RedirectActivationToAsync(...)` to hand the activation to the primary (`Program.cs:42-49`), then the secondary returns without starting WinUI.

The `/pushnotification` exception (`Program.cs:45`) means the push-background COM server launch (see §3.10) is **not** redirected — it runs as its own process.

`OnActivated` forwards every redirected activation to `App.Instance?.OnActivated(args)` (`Program.cs:54-57`).

> **(inferred)** Using `.Result` on `DecideRedirection` on the STA thread is safe here because the awaited `RedirectActivationToAsync` runs before any WinUI message pump exists; no UI `SynchronizationContext` is installed yet.

### 3.2 `App` construction (synchronous, on UI thread) (`App.cs:111-152`)

`App` is `Application`, and implements `NotificationsManager.IWindowOperationsProvider`, `SystemIntegrationsManager.IAppStateProvider`, `TrayIconManager.IApplicationOperationsProvider`, `IXamlMetadataProvider` (`App.cs:49`). The constructor runs synchronously and does *no* heavy I/O:

1. `Instance = this` (static singleton; `App.cs:113`, declared `App.cs:83`).
2. `_dispatcher.TryInitialize()` — `_dispatcher = Deployment.Current.Dispatcher` (`App.cs:51`, `App.cs:114`). See §3.7.
3. `_appLaunchMetrics.RecordAppLaunch()` (twice — `App.cs:115`, `:122`).
4. `InitializeCultureInfo()` — sets all four `CultureInfo` slots and the current thread culture from `ApplicationLanguages.PrimaryLanguageOverride` (`App.cs:116`, `:160-165`).
5. `InitializeComponent()` — loads `ms-appx:///App.xaml` (`App.cs:117`, `:742-752`).
6. SAD records + exception handler: `_sadRecords = new SadRecordsStorage(new MemoryMappedFileSadRecordPersistentStorage())` then `ExceptionsHandler{ SwallowUnhandledExceptions=true }.Subscribe(this)` (`App.cs:118-121`). See §3.9.
7. `PushNotificationBackgroundTask.Register(out _pushNotificationBackgroundRegistrationToken)` — registers the in-proc COM class factory for the push task (`App.cs:123`). See §3.10.
8. `_views = new ViewsController(this, _dispatcher, _sadRecords, _appLaunchMetrics)` (`App.cs:124`).
9. `StartModel(_sadRecords)` — kicks the **async** model build (`App.cs:125`, see §3.3).
10. Subscribes three reactions to `WhenAppInSystemTrayChanged` (`App.cs:126-138`): record SAD background transition; map tray state → `_webLifecycle.ApplyAppState(VisibilityState.SystemTray | Visible, None)`; set `AppState.IsInSystemTray`.
11. `_appThemeManager = new ApplicationThemeManager(this)` (`App.cs:139`).
12. `_nativeAppStateBridgeController = new NativeAppStateBridgeController(this, _appLaunchMetrics)` (`App.cs:140`).
13. Registers `SystemEvents.SessionSwitch += onSessionSwitch` on the events thread; on `SessionUnlock` → `_webLifecycle.ForceResume()` (`App.cs:143-146`, `:167-173`).

Pre-login bridge fields are constructed eagerly here: `_webLifecycle = new WebLifecycleController()` (`App.cs:55`).

### 3.3 Async model bootstrap (`StartModel` / `AppModel.Initialize`)

`StartModel` (`App.cs:175-213`) is `async void`. It **hops off the UI thread** first:

```csharp
await new ConcurrentQueueDispatcher().ContinueIn("StartModel", ...);   // App.cs:177
AppModel model = new AppModel(sadRecords);                             // :178
await model.Initialize();                                              // :181 (try/catch)
```

On native-storage init failure (`App.cs:183-191`): `FailuresService.Investigate(...)`; if `SqliteException` with a fatal HResult (`IsFatalSqliteError` checks SQLite codes 8,11,13,14,18,19,20,21,23,26 via `Sqlite.HRForError`, `App.cs:197-212`) → `model.Logout()`; then `Restart(ControlledRestartTypes.CorruptNonDbSettings, …, restartToTray:false)`.

On success: `_modelSource.TrySetResult(model)` resolves the `TaskCompletionSource<AppModel>` (`App.cs:63`, `:192`) that every later consumer awaits; then `model.Warmup()`, `_webLifecycle.Warmup()`, builds `MayHaveMessagesNotificationManager`, and subscribes to background-task push events (`App.cs:193-196`).

`AppModel` is the DI/composition root. Its constructor (`AppModel.cs:109-161`) wires:
- VoIP web bridge `_voipRoot = new VoipWinRTWebBridge()` and assigns all `VoipSignaling.*` delegate callbacks (`AppModel.cs:114-136`).
- `_loginSessionManager = new LoginSessionManager(new ConcurrentQueueDispatcher())` (`AppModel.cs:137`).
- Controllers: contacts, abprops, client-key, server-enc-key-salt, tc-token (`AppModel.cs:138-143`).
- `_sqliteDb = new SqliteDb(_loginSessionManager, "genericStorage.db")` + `_sqliteBridge` (`AppModel.cs:144-145`).
- Preferences (`AppDataContainerStorage("web_preferences")`), WAM, media services, app-activation service, sharesheet (`AppModel.cs:146-155`).

`AppModel.Initialize` (`AppModel.cs:163-176`): initializes the logger, `await _loginSessionManager.Initialize()` → `SessionData`, then initializes abprops/contacts/media services from it, starts periodic contact sync, and sets `IsStartedRegistered = sessionData.ClientKey != null` (the "is this account logged in" flag, `AppModel.cs:175`).

`AppModel.Warmup` (`AppModel.cs:178-190`) starts WAM, contacts, the VoIP worker (`VoipManaged.Worker.DispatchAsTask(...)`), checks for a previous crash (`CrashReporter`), and toggles the push background task based on the notification-banner preference.

### 3.4 Main window initialization (`App.EnsureMainWindowInitialized`) (`App.cs:259-341`)

Idempotent (guards on `MainWindow != null`, `App.cs:261`). Runs on the UI dispatcher (`await _dispatcher.ContinueIn(...)`, `App.cs:265`). Steps:

1. `MainWindow = new MainWindow()` (`App.cs:266`).
2. RTL: set/clear `ExtendedWindowStyle.LayoutRtl` from `CultureInfo.CurrentUICulture.TextInfo.IsRightToLeft` (`App.cs:267-275`).
3. `NativeWindowHelper.SetNativeWindowEventHandlers(MainWindow, QueryEndSessionHandler, PowerResumeHandler)` — OS session-end + power-resume hooks (`App.cs:276`; handlers at `:343-351`). `QueryEndSession` records `ForcedRestartByOperatingSystem`; `PowerResume` → `_webLifecycle.ForceResume()`. Mechanism (`NativeWindowHelper.cs:28-83`): it **subclasses the HWND's WndProc via `SetWindowLongPtr(hwnd, GWLP_WNDPROC=-4, …)`** (not the comctl `SetWindowSubclass` API), captures the original proc, and chains every message through `CallWindowProc`. The replacement proc acts on exactly two messages — `WM_QUERYENDSESSION (17)` → `queryEndSessionHandler()` and `WM_POWERBROADCAST (536)` with `wParam == PBT_APMRESUMEAUTOMATIC (18)` → `powerResumeHandler()` (`:44-51`); `WM_ENDSESSION (22)`/`WM_DESTROY (2)`/`WM_CLOSE (16)` are declared as consts but unhandled (dead `_ = 22; _ = 2; _ = 16;`, `:52-54`). 32/64-bit `Get/SetWindowLong(Ptr)` are branched on `IntPtr.Size`; the delegate is pinned with `GC.KeepAlive` (`:82`).
4. Icon, `AppWindowRegistry.Instance.TrackWindow`, `Closed += OnWindowClosed`, DPI tracking (`App.cs:277-280`).
5. `_mainWindowPositionManager` restores + tracks window position (`App.cs:281-283`; §3.6).
6. Guarded by `_views.WindowView == null` (one-time UI/model wiring, `App.cs:285-340`):
   - `_views.SetUp(MainWindow)` → builds `WindowView`, subscribes window Activated/VisibilityChanged, attaches the VoIP session (`ViewsController.cs:94-113`).
   - `AppModel model = await _modelSource.Task` — **blocks the UI flow until the async model is ready** (`App.cs:292`).
   - `_views.TrayIcon.Initialize()`, `PeriodicJobScheduler.Instance.Schedule()`, `_views.SetModel(model)` (`App.cs:293-295`).
   - Determines `shouldStartInTray = activatedArgs.Kind == StartupTask || cmdline contains "/nowindow"` (`App.cs:304`); records launch metrics (`:305-306`).
   - `CoreWebView2 webView = await MainWindow.GetWebViewCore(shouldStartInTray)` (`App.cs:307`; §3.5).
   - If `model.IsStartedRegistered != true` → `ClearBrowsingData("no login session")` (`App.cs:308-311`).
   - Builds `SystemIntegrationsManager`, `BuildQueryParameters()`, `WebContentUpdateManager` (`App.cs:312-314`).
   - **Pre-model bridges** added directly here (`App.cs:315-318`): `ConnectionBridge`→`LifecycleBridge(_webLifecycle)`, `TouchpadFix`, `NativeAppStateBridge`, `WebUpdateBridge`.
   - `model.SetWebView(webView, systemIntegrationsManager, MainWindow.ZoomBrowserExtension)` — registers the **~23 model bridges** (`App.cs:319`; §3.8).
   - `HangReporter.StartMonitoring`, `_webLifecycle.SetWebView`, set `MainWindow.ViewModel`, `_updateManager.StartUpdates()`, `webView.AlwaysUseArgumentsForDocumentRequested(args)`, `MainWindow.Start(args)` (navigates the WebView) (`App.cs:321-327`).
   - Show or hide based on `shouldStartInTray` (`App.cs:328-335`).
   - Start theme tracking, expiration check, language-override subscription, `OnMainAppStarted()` (`App.cs:336-339`).

`BuildQueryParameters` (`App.cs:378-403`) is the native→web handshake context: `windows=1`, `windowsBuild` (`PackageInfo.Version.ToStringHybrid()`), `windowsBuildType`, `bridgeError=1`, `launchContext`, `osBuild`, and conditionally `windows_offline=1` when offline support is on and not data-connected. The WhatsApp Web JS reads these to detect it is inside the native Windows shell — **confirmed in the bundle**: `WAWebEnvironment.isWindows` is the gate the bundle uses to set the analytics subplatform to `"win_hybrid"` (else `"web"`) and to enable the hybrid-bridge surface (`grep waweb-source-bundle` finds `r("WAWebEnvironment").isWindows?l="win_hybrid"…`, `windowsBuild`, `launchContext`, `windows_offline` as recognized query keys).

### 3.5 WebView2 environment + lockdown (`MainWindow`)

`MainWindow` is a `WinUIEx.WindowEx` implementing `IComponentConnector` (`MainWindow.cs:29`). Constructor (`MainWindow.cs:438-460`):
- `ExtendsContentIntoTitleBar = true`, `MinHeight=600`, `MinWidth=800` (`MainWindow.cs:441-443`).
- `WebView.CoreWebView2Initialized += OnCoreWebView2Initialized` (`:444`).
- Builds `AdvancedWebView2NavigationStatusMonitor` over `WebView2NavigationStatusMonitor`, observed on UI dispatcher → `OnNavigationStatusChanged` (`:445-446`).
- `ZoomBrowserExtension.Bootstrap()`, `SharesheetManager`, `JumpListController`, memory-optimization flags from `AbProps.Props.WindowsGracefulDegradationVersion` (`:447-451`).
- `Closed` handler: if `_restartToTrayOnClose`, cancel close and `RestartApp(WebView2Error, …, restartToTray:true)` (`:452-459`).

`GetWebViewCore(bool startInBackground)` (`MainWindow.cs:462-514`) lazily creates a `CoreWebView2Environment` with options:
- `AreBrowserExtensionsEnabled = true`, `Language = CultureInfo.CurrentCulture.Name` (`:466-469`).
- `DisableDataReportingToMS()` (`:471`).
- `StartBackgrounded()` if launching to tray (`:472-475`); `StartGpuDisabled()` if `AbProps.Props.Webview2DisableGpuAcceleration` (`:476-479`).
- On low-memory devices, appends Chromium flags from the memory-optimization bitset (`MemoryOptimizationFlags` enum `:31-38`): `--disable-features=msSmartScreenProtection,msWebView2EnableTrackingPrevention`, `--js-flags=--scavenger_max_new_space_capacity_mb=8`, `--msWebView2SimulateMemoryPressureWhenInactive=1` (`:480-508`).
- User-data folder = `ApplicationData.Current.LocalCacheFolder.Path` (`:509`).
- `await WebView.EnsureCoreWebView2Async(_environment)` then `await _coreInitializationSource.Task` (set in `OnCoreWebView2Initialized`) (`:511-513`).

`OnCoreWebView2Initialized` (`MainWindow.cs:534-570`) locks down the WebView: disables browser accelerator keys, status bar, reputation checking, built-in error page, swipe nav, script dialogs, zoom control, pinch zoom (`:538-546`); wires `DOMContentLoaded`, `NewWindowRequested`, `ProcessFailed`, `NewBrowserVersionAvailable`, `ContextMenuRequested`, `PermissionRequested`, `ScriptDialogOpening`, `NavigationStarting → ExternalLinkNavigation.OnNavigationStarting` (`:548-555`); starts navigation tracking; forces persistent storage permission; resolves `_coreInitializationSource` (`:556-558`).

- **Permissions**: every `PermissionRequested` is auto-`Allow`ed (`MainWindow.cs:723-735`).
- **External links**: `NewWindowRequested` → `Launcher.LaunchUriAsync` in the OS, `args.Handled=true` (`:674-693`).
- **Script dialogs**: auto-`Accept()` (`:737-748`).

`Start(args)` (`MainWindow.cs:516-532`): if env is `OnDemand` open devtools (`if (GetCurrentEnvironmentType()==WaWebEnvironmentType.OnDemand) coreWebView.OpenDevToolsWindow()`, `:519-521`); resolves URL from `WaWebEnvironmentProvider.Instance.GetUrl()`; adds header `X-WA-WebView2-Version` = package version; enables web-resource-requested events for the sharesheet; optionally creates `CacheManager` for offline support; then `coreWebView.Navigate(url, args)` (`:531`).

> **DevTools are enabled in this retail build.** `OnCoreWebView2Initialized` sets `core.Settings.AreDevToolsEnabled = IsDevEnvironment()` (`MainWindow.cs:547`), and `IsDevEnvironment()` is hardcoded `return true;` (`MainWindow.cs:669-672`). This is **not** a debug/alpha artifact: the build is Production/Release (`BuildDetails.CurrentReleaseChannel => Production`, `CurrentBuildConfiguration => Release`, so `IsDebug => false`; `BuildDetails.cs:9,17,19`). So WebView2 DevTools (F12 / right-click → Inspect) are reachable in shipped WhatsApp-for-Windows; the `OnDemand` env additionally force-opens the DevTools window on navigate.

**Process-failure resilience** (`OnCoreWebView2ProcessFailed`, `MainWindow.cs:585-621`):
- `BrowserProcessExited`: ping the WebView; if dead set `_restartToTrayOnClose`; if also not visible, restart to tray; show fatal-error screen and poll-ping until alive (`:601-608`, `:636-652`).
- `RenderProcessExited` / `RenderProcessUnresponsive` / `FrameRenderProcessExited`: `WebView.Reload()` (`:609-613`).
- `NewBrowserVersionAvailable` → ask-to-update dialog → restart `WebView2Update` (`:654-667`).

### 3.6 Window position persistence (`MainWindowPositionManager.cs`)

Keys in `ApplicationData.Current.LocalSettings`: `window.position.X/Y`, `window.size.Width/Height`, `window.state` (`:15-23`). `TryToRestorePreviousWindowPosition` (gated on OS build ≥ 19041) resizes/moves/sets `WindowState`, and `EnsureIsVisible` re-clamps the window onto the host monitor's working area if <50% overlaps (`:105-142`). `StartTrackingWindowPositionChange` combines `WhenStateChanged`/`WhenPositionChanged`/`WhenSizeChanged`, `Throttle(1s)`, and only persists position+size when state is `Normal` (`:63-103`).

### 3.7 Navigation status state machine (splash/error overlays)

`WebView2NavigationStatusMonitor` (`:9-109`) tracks per-`NavigationId` start/complete and emits `NavigationStatus { Success, Failed, InProgress, NoConnection }` via a `BehaviorSubject` (`:11-27`). On completion of the latest navigation it maps `CoreWebView2WebErrorStatus` (`CannotConnect/Disconnected/ConnectionAborted/ConnectionReset/HostNameNotResolved → NoConnection`, else `Failed`) or `Success` (`:81-96`).

`AdvancedWebView2NavigationStatusMonitor` (`:8-57`) layers retry/backoff debouncing: a `Failed` after `InProgress` is delayed `2 * failedCounter` seconds (Rx `DelayOrdered`), counter resets on `Success` (`:26-38`); an `InProgress` lasting > **90 s** is upgraded to `Failed` via `_isInProgressTooLongSubject` (`:39-55`).

`MainWindow.OnNavigationStatusChanged` (`:695-713`) toggles `BrowserLoadingSplashScreen` (visible while `InProgress`) and `BrowserNavigationErrorScreen` (visible on `Failed`/`NoConnection`, with `NoConnection`→`NoConnection` error type else `UnexpectedNavigationError`); after >1 completed navigations it calls `ClearAppLaunchTimestamp` (launch-metric end marker).

### 3.8 JS↔native bridge registration (the DI surface for the web client)

The bridge wiring is the core JS↔native contract. Two registration sites:

**Pre-model (in `App.EnsureMainWindowInitialized`, `App.cs:315-318`)** — 4 bridges via `webView.AddWinRTBridge(name, instance)`:
`ConnectionBridge`(`LifecycleBridge`), `TouchpadFix`, `NativeAppStateBridge`, `WebUpdateBridge`.

**Model bridges (in `AppModel.SetWebView`, `AppModel.cs:209-242`)** — a shared `DispatchAdapter` is created (`:217`) and ~23 bridges registered, each as `webView.AddWinRTBridge(name, new XBridge(controller), dispatchAdapter)`:

`VoipBridge`, `VoipSignalingBridge`, `WamBridge`, `AbPropsBridge`, `PreferencesBridge`, `RateAppBridge`, `ScalingControlBridge`, `ClientKeyBridge`, `PicturesBridge`, `SQLiteBridge`, `MediaFilesBridge`, `MediaTranscodingBridge`, `SharesheetBridge`, `AppActivationBridge`, `TcTokenBridge`, `SeamlessMigrationBridge`, `SystemIntegrationsBridge`, `LinksPreviewBridge`, `BrowserExtensionsBridge`, `AdvBridge`, `ContactsBridge`/`PopulatedContactsBridge`, `DebugFeaturesBridge`, `ServerEncKeySaltBridge` (`AppModel.cs:218-241`).

`AddWinRTBridge` (`WinRTAdapterExtensions.cs:13-21`) is a thin wrapper over `CoreWebView2.AddHostObjectToScript(name, dispatchAdapter.WrapObject(bridge))`. JS reaches each as `window.chrome.webview.hostObjects.<name>`. `WrapObject` runs each adaptee through `DispatchAdapter.WrapObject(unwrapped, dispatchAdapter)` (`:8-11`).

**Confirmed in the bundle (JS-side contract).** Cross-checking `waweb-source-bundle/` shows the WhatsApp Web client accesses these exact host objects: every registered bridge name appears as `chrome.webview.hostObjects.<Name>` (grep finds `hostObjects.AbPropsBridge`, `AdvBridge`, `AppActivationBridge`, `BrowserExtensionsBridge`, `ClientKeyBridge`, `ConnectionBridge`, `ContactsBridge`/`PopulatedContactsBridge`, `DebugFeaturesBridge`, `LinksPreviewBridge`, `MediaFilesBridge`, `MediaTranscodingBridge`, `NativeAppStateBridge`, `PicturesBridge`, `PreferencesBridge`, `RateAppBridge`, `RingtoneBridge`, `ScalingControlBridge`, `SeamlessMigrationBridge`, `ServerEncKeySaltBridge`, `SharesheetBridge`, `SQLiteBridge`, `SystemIntegrationsBridge`, `TouchpadFix`). The bundle configures the WebView2 host-object proxy globally: `hostObjects.options.defaultSyncProxy=!0`, `hostObjects.options.forceAsyncMethodMatches=[/Async$/,/AsyncWithSpeller$/]`, `hostObjects.options.ignoreMemberNotFoundError=!0` — i.e. calls are **synchronous by default**, methods whose names end in `Async`/`AsyncWithSpeller` are forced to the async (promise) proxy, and missing-member access is tolerated. Latency-sensitive bridges are read through the explicit synchronous proxy `chrome.webview.hostObjects.sync.<Name>` (confirmed for `ContactsBridge`, `PopulatedContactsBridge`, `NativeAppStateBridge`). So the `*Async` method-name convention on the C# `IXBridgeToNative` interfaces is the actual JS sync/async selector, and the §3.8 registration list is the host-object surface the bundle expects 1:1.

`ContactsBridge` is registered under name `PopulatedContactsBridge` if contacts were already synced, else `ContactsBridge` (`AppModel.cs:238-239`) — the name itself signals warm-cache state to the JS.

**Legacy string-IPC fallback** (`AppModel.OnWebMessageReceived` + `InvokeBridgeMethodFromMessage`, `AppModel.cs:249-415`, marked `[Obsolete("Use WinRT bridge instead")]`): JS posts a comma-delimited `method,arg1,...,transactionId` string via `WebMessageReceived`; native reflects the method onto `VoipWinRTWebBridge`, then `SqliteBridge`, then the `_bridgeTypesToInstances` map, URL-decodes/parses args by reflected parameter type (handles `string[]` `|`-delimited and `string[][]` `]|[`-delimited, `AppModel.cs:345-362`), awaits any returned `Task`, JSON-serializes the result and posts back `transactionId,NativeResponseToJS,0,<json>` (`:397-400`) or `…,-1,` on error (`:393-396`). A special method `CompatibilityGetSupportedNativeIpcs` returns the reflected method→arity map (`:264-266`, `:244-247`).

### 3.9 Activation routing & deeplinks

Two activation entry points converge: `OnLaunched` (cold launch, `App.cs:235-257`) and `OnActivated` (redirected/secondary activations from `Program.OnActivated`, `App.cs:544-578`).

`OnLaunched`:
- Skips entirely if cmdline has `/pushnotification` (push COM-server process; `App.cs:238`).
- `await EnsureMainWindowInitialized()` (`:240`), then dispatches by `GetActivatedEventArgs().Kind`:
  - `AppNotification` → `_views.Notifications.HandleNotificationActivation((AppNotificationActivatedEventArgs)data)` (`:242-245`).

#### 3.9.1 Sharesheet file injection into the SPA (`SharesheetManager`)

The deeplink above (`whatsapp://send?...&source=sharesheet`) carries only the *text*/`jid` of a share; the OS-shared **files** (`StorageFile`s) are not URI-encodable, so they are streamed into the web client out-of-band via **synthetic in-page URLs** served over `CoreWebView2.WebResourceRequested`. This is the mechanism the share path's `attachment_uris` query value actually points at, and the concrete sub-mechanism behind the §3.5 claim that `MainWindow.Start` "enables web-resource interception for the sharesheet". Implementation: `SharesheetManager` (`decompiled/WhatsApp.Root/WhatsApp.SystemIntegrations/SharesheetManager.cs`), an `internal class SharesheetManager(WebView2 webView)` (`:15`), constructed once per window as `_sharesheetManager = new SharesheetManager(WebView)` in the `MainWindow` ctor (`MainWindow.cs:378`, `:449`).

**Register (native side mints URLs).** `RegisterFiles(IEnumerable<StorageFile> files)` (`SharesheetManager.cs:21-37`): **clears** the prior `_unconsumedFiles` dictionary (`:28` — each new share replaces the last; there is no accumulation across shares), then for each non-null file inserts it under a freshly-minted key `"/sharesheet/" + Guid.NewGuid()` (prefix const `SharesheetUriPrefix = "/sharesheet/"`, `:17`; key built at `:33`) and returns the list of keys (`:36`). A null `files` collection logs and returns empty (`:23-27`). The keys are **relative paths**, not absolute URLs — they become URLs only once resolved against the page origin (`https://web.whatsapp.com/sharesheet/<guid>`) inside the WebView. Reached from the shell via `MainWindow.RegisterFilesForSharesheet → _sharesheetManager.RegisterFiles` (`MainWindow.cs:572-575`), itself called from `App.CreateShareDeeplinkAsync` when the share contains `StandardDataFormats.StorageItems` (`App.cs:667-671`); the returned keys are joined comma-separated into the `attachment_uris` query param of the `whatsapp://send` deeplink (`App.cs:685-688`). So the bundle receives the synthetic paths inside the deeplink and fetches each one as an ordinary in-page resource.

**Intercept (filter is global).** `MainWindow.Start` (the navigation entry point, §3.5) wires the handler *before* navigating: `coreWebView.EnableWebResourceRequestedEvents(); coreWebView.WebResourceRequested += _sharesheetManager.SharesheetWebResourceRequested;` (`MainWindow.cs:525-526`). `EnableWebResourceRequestedEvents` (`WhatsApp.Tools/WebViewExtensions.cs:275-286`) registers a **wildcard** filter `core.AddWebResourceRequestedFilter("*", CoreWebView2WebResourceContext.All)` (`:279`) — i.e. *every* web-resource request the page makes flows through the handler, and `SharesheetManager` itself does the path-prefix discrimination (not the filter). (Note: this is the only `WebResourceRequested` subscription in the shell; `BinaryTransparency.cs:85` also calls `CreateWebResourceResponse` but for an unrelated 204 response — see below.)

**Bundle-side consumption (confirmed).** The synthetic keys ride in the `whatsapp://send` deeplink's `attachment_uris` query value; the WhatsApp Web JS reads it and **splits on `","`** into an array (`…"attachment_uris": n.attachmentUris = e.split(",")…`, `waweb-source-bundle/`), gating downstream work on `n.attachmentUris != null && n.attachmentUris.length > 0`, and only on the `source=="sharesheet"` branch. Each resulting relative `/sharesheet/<guid>` resolves against the page origin and is fetched as an ordinary in-page resource, which the global `WebResourceRequested` handler above serves once from the `StorageFile` stream. (The single-use `Remove`-after-serve and the back-to-back-share `Clear()` race remain runtime-only edges.)

**Adjacent `WebResourceRequested` consumer — BinaryTransparency (`BinaryTransparency.cs`).** A separate, Chrome-CodeVerify mechanism shares the navigation lifecycle. `BootstrapBinaryTransparencyExtensionFolder` (`:45-72`) copies a `CodeVerify/Chrome` extension into `LocalCacheFolder`, minting a fresh 32-byte URL-safe-base64 nonce; the extension is added to the WebView2 profile only when `BuildDetails.IsDebug && IsProductionWebEnv()` (`:37-43` / `AddBrowserExtensionToWV`), so in this Release build the extension-install path is inert. The live half is `BinaryTransparencyWebResourceRequested` (`:74-127`): it intercepts requests to `…/bt_validation_result/<nonce>/<STATE>` (matching on the nonce), answers each with a **204 No Content** (`CreateWebResourceResponse(null, 204, …)`, `:85` — the "unrelated 204" noted above), records the `VALID`/`IGNORE`/`RISK`/failure outcome via `FailuresService.Count`, and on `RISK`/failure shows a reload/ignore dialog (`IgnoreEsc|IgnoreOverlayClick`). `App.TriggerBinaryTransparency()` (`App.cs:467-470`) is a **no-op** in this build (`_ = MainWindow?.WebView == null;`, a discarded comparison). Separately, **`BugNub`** debug reporting (`BugNub.cs`) is reached only from the debug tray menu — `App.OpenBugNub` (`App.cs:499-508`) awaits the model, shows the window, then `BugNub.PromptBugReport(webView, BugCategory.General, SaveJSLogs)`. None of these three (BinaryTransparency interceptor, `TriggerBinaryTransparency`, `OpenBugNub`) affect the normal boot/activation lifecycle: the first is a passive validation sink, the latter two are inert/user-triggered.

**Serve (stream the file as the HTTP response).** `SharesheetWebResourceRequested` (`SharesheetManager.cs:39-48`) is an `async void` event handler that just `try/await HandleSharesheetWebResourceRequestedAsync(args)` and swallows exceptions. `HandleSharesheetWebResourceRequestedAsync` (`:50-95`):
1. Parses `args.Request.Uri`; on parse failure logs and returns (`:52-56`).
2. **Discriminates by prefix**: if `uri.AbsolutePath` does not start with `/sharesheet/` (case-insensitive) **or** the path is not a key in `_unconsumedFiles`, it returns without touching `args.Response` — letting WebView2 fetch the resource normally (`:59-62`). This is what makes the wildcard filter harmless for all non-sharesheet traffic.
3. For a matching key: takes a `Deferral` (the response is produced asynchronously, `:63`), opens the file read-only `await file.OpenAsync(FileAccessMode.Read)` → `IRandomAccessStream content` (`:66`), builds two response headers — `Content-Type: <file.ContentType>` and `Content-Disposition: attachment; filename="<file.Name>"` (`:67-80`) — and sets `args.Response = webView.CoreWebView2.Environment.CreateWebResourceResponse(content, 200, "OK", <headers>)` (`:81`). The response **body is the file's own stream**, so the file bytes never round-trip through a data-URI or a temp copy.
4. **Single-use**: on success it `_unconsumedFiles.Remove(uri.AbsolutePath)` (`:82`) — each synthetic URL is consumed exactly once (hence "unconsumed" files); a second fetch of the same GUID falls through to the normal-resource path and 404s at the origin. **(inferred)**
5. On exception: `args.Response = CreateErrorResponse(500, "Internal Server Error")` (a `text/plain` 500 built via `CreateWebResourceResponse`, `:87`, `:97-101`) and `FailuresService.Investigate(...)` (`:88`); the `Deferral` is always completed in `finally` (`:92`).

Net effect: OS share files appear to the WhatsApp Web JS bundle as fetchable URLs under the page's own origin, served on demand with the correct MIME type and filename, with no filesystem path exposed to the renderer and no separate native upload step — the bundle treats them like any attachment it picked from a file input. This is **distinct from** the `whatsapp://send` deeplink/protocol activation documented in §3.9 and docs 01/31 (which carries only text/`jid` and the list of synthetic URLs): those docs name the interception but not this register-files-then-serve-stream half. Verified against source: `SharesheetManager.cs` (full file), `MainWindow.cs:378,449,525-526,572-575`, `WebViewExtensions.cs:275-286`, `App.cs:667-688`.

### 3.13 Threading & dispatcher model (`WhatsApp.Core`)

The shell is **not** a free-threaded program; it uses explicit named dispatchers and an await-on-dispatcher pattern.

- **`IThreadDispatcher`** (`IThreadDispatcher.cs`): `bool IsCurrent` + `void Dispatch(state, SendOrPostCallback)`.
- **UI dispatcher `Dispatcher`** (`Dispatcher.cs`): wraps the WinUI `SynchronizationContext`. Before `Initialize()` is called on the UI thread, `Dispatch` queues actions into `_queuedActions` (`:22-49`); `Initialize` captures `SynchronizationContext.Current`, names the thread "UI thread", and flushes the queue (`:51-72`). `IsCurrent` compares `_threadId` to `Environment.CurrentManagedThreadId`; `IsReady` = context captured. Exposed as `Deployment.Current.Dispatcher` (`Deployment.cs:68`).
- **`ConcurrentQueueDispatcher`** (`ConcurrentQueueDispatcher.cs`): a `SynchronizationContext` + `IThreadDispatcher` implementing a **serial** queue drained on thread-pool threads. `Dispatch` enqueues a `WorkItem`; `InvokeQueueItems` sets itself as the current sync-context, runs items in order, and `Interlocked.Exchange`/`CompareExchange` on `_threadId` so `IsCurrent` is correct during draining (`:39-99`). Each new model/login/bridge component gets its own instance for serial execution (e.g. `LoginSessionManager`, `SqliteBridge`, `WebLifecycleController`, `WebContentUpdateManager`).
- **`SingleThreadedDispatcher`** (`SingleThreadedDispatcher.cs`): a dedicated `Thread` draining a `BlockingCollection` — used where one persistent OS thread is required.
- **`Deployment.ThreadPool`** (`Deployment.cs:70-79`): a `LimitedThreadPool` of 10 `ConcurrentQueueDispatcher`s round-robined by `Interlocked.Increment` (`:9-34`), unless `Experiment.LimitThreadPool` forces a raw `ThreadPoolDispatcher`.
- **Capturing dispatchers**: `ICapturingThreadDispatcher<TContext>` lets a worker thread own a native context object. `VoipManaged.Worker` is a `VoipDispatcher : ICapturingThreadDispatcher<IVoip>` (`VoipManaged.cs:45`) — all VoIP native calls hop onto it via `await VoipManaged.Worker.ContinueIn(...)` (e.g. `ViewsController.cs:107,344,398`).
- **Await ergonomics** (`DispatcherExtensions.cs`): `dispatcher.ContinueIn()` returns an awaitable that completes synchronously if already `IsCurrent`, else posts the continuation onto the dispatcher (`:264-268`, `:23-45`); `task.ConfigureAwait(dispatcher)` resumes a `Task` on a specific dispatcher (`:270-274`, `:47-85`); plus `Dispatch`, `InvokeOrDispatch`, `InvokeSynchronous`, `DispatchAsTask` (`:199-262`). All carry `[CallerMemberName]`/`[CallerFilePath]` for the dispatch profiler. This `await someDispatcher.ContinueIn(...)` idiom is pervasive across the shell (App, ViewsController, WebLifecycleController, WebContentUpdateManager).
- `Dispatchers.CreateSerial` / `Profile` provide creation + optional CPU/latency profiling (`Dispatchers.cs:18-37`); `Dispatchers.CpuWatchdog` underpins hang reporting.

### 3.15 WAM client telemetry subsystem (native event queue + serializer + rotation)

WAM ("WhatsApp Analytics Metrics") is the client-side analytics pipeline. The native half is backed by the `WamBridge` host object (§3.8); its C# implementation is the **producer→serialize→persist→rotate→hand-to-JS** chain below. Note the direction: native **does not upload**; it drains the event buffers, serializes/normalizes them, and pushes JSON to the **JS bundle**, which owns the actual server upload. There are two distinct event buffers (native ring buffer + a managed channel) joined by `WamController`.

**Event universe — `Wam.cs` (`decompiled/WhatsApp.Core/WhatsApp/Wam.cs`).** A single static class `Wam` (`Wam.cs:6`) holds:
- `Wam.Enum.*` — a vast field-code/enum universe; each enum value carries a `[Display(Name=…)]` snake-case wire name (e.g. `AboutItemAction`, `AppLaunchType`, `TsSurface`) (`Wam.cs:8` opens `public static class Enum`; first enum at `:10`). This file is ~2.2 MB — the bulk is this enum catalog.
- `Wam.Event.*` — one nested `WamEvent` subclass per event type (`Wam.cs:28445` opens `public static class Event`). Each event is a POCO of nullable properties plus three members: `GetCode()` returns the integer **event id** (e.g. `AccountReportAutomaticSettingChanged.GetCode()==5148u` `Wam.cs:28461`; `AppLaunch.GetCode()==1094u` `Wam.cs:30461`), `Reset()` nulls all fields, and `SerializeFields(IWriter)` emits each set field by **numeric field code** via `fields.Change(code, value)` (e.g. `AppLaunch.SerializeFields` writes codes 22,23,2,7,3,… `Wam.cs:30464-30482`).
- `Wam.IWriter` (`Wam.cs:82192-82201`) — the serialize contract: `ChangeValueToNull(int code)`, `ChangeValue(int code, double)`, `ChangeValue(int code, long)`, `ChangeValue(int code, string)`. The `Change(this IWriter, int, T?)` extension overloads (`Wam.cs:82405-82456`) translate nullable CLR values to the four primitives: a `bool?` becomes a long `1/0`, a null becomes `ChangeValueToNull`, enums go through `EnumToLong<T>` which casts to `(int)` (`Wam.cs:82458-82467`). There are also typed setter helpers for the common envelope fields (`SetAbKey`/`SetAppBuild`/`SetAppVersion`/… `Wam.cs:82203+`).
- `Wam.Storage` — a process-wide `WamStorage` singleton (`Wam.cs:82210`).

**`WamEvent` base (`decompiled/WhatsApp.Core/WhatsApp/WamEvent.cs`).** Abstract: `uint GetCode()` + `void SerializeFields(Wam.IWriter)` (`WamEvent.cs:5-7`). This is the entire contract every `Wam.Event.*` implements.

**Binary serializer — `WamEventsSerializer` (`decompiled/WhatsApp.Core/WhatsApp/WamEventsSerializer.cs`), `Wam.IWriter`.** A length-prefixed binary format keyed by a `LineType` int discriminator: `EventId=0, EventWeight=1, IntField=256, LongField=512, DoubleField=768, StringField=1024, NullField=1280, EventStopFlag=65536` (`WamEventsSerializer.cs:10-20`). `CurrentSchemaVersion = 1` (`:22`).
- `Write(WamEvent, version, weight)` frames one event: `Start(code,version,weight)` writes `EventId, code, version(=1), EventWeight, weight` and seeds `_fieldsCount=2` (`:177-185`); then `event.SerializeFields(this)` writes each field as `<LineType><code><value>` (strings via `WriteNormalizedString`, `:157-163`); then `Stop()` writes `EventStopFlag, _fieldsCount` (`:187-191`).
- `WriteNormalizedString` (`decompiled/WhatsApp.Core/WhatsApp/BinaryWriterExtensions.cs:13-18`) encodes with `Encoding.Default` (length-prefixed) — note the *reader* decodes strings with UTF-8 (`WamEventsSerializer.cs:82`), an asymmetry to preserve when porting.
- `Deserialize(byte[])` (`:30-127`) is the inverse, yielding `Dictionary<object,object?>` per event keyed `"id"`/`"weight"`/field-code; it validates the version (`>1` throws), validates the trailing field count matches, and treats `NaN`/null as field removal (`:46-48`, `:63-66`, `:103-126`). This is the path `WamController` uses to turn the **native** ring buffer's bytes back into dictionaries.

**Per-event storage + privacy gate + sampling — `WamStorage` (`decompiled/WhatsApp.Core/WhatsApp/WamStorage.cs`).** Holds an **unbounded** `Channel<Dictionary<object,object?>>` (`:35`), drained on a dedicated `ConcurrentQueueDispatcher{ Id = FieldStats }` (`:39-42`).
- `Save(WamEvent ev, int? samplingInterval, string? version)` (`:54-75`): gated by **three** conditions ANDed — `IsPrivacyCheckedEvent(ev)`, `samplingInterval >= 0`, and sampling (`samplingInterval <= 1` always passes, else `_random.Next(samplingInterval) == 0`, i.e. 1-in-N) (`:61`). When admitted it hops to the FieldStats dispatcher, builds a dict seeded `{id: code, weight: samplingInterval}`, runs the event through an inner `Writer` that writes directly into the dict (`:12-33`), and `WriteAsync`s it to the channel (`:64-73`).
- `IsPrivacyCheckedEvent` (`:77-89`): membership test against a **hardcoded allow-set** `CreateCheckedSet()` of ~95 `Wam.Event.*` types (`:91-183`, e.g. `AppLaunch`, `Call`, `MessageSend`, `MessageReceive`, `MediaUpload2`, `TsBitArray`, `UserActivity…`, the `MDAppState…`/`Adv…` families). **Events not in this set are silently dropped** (and recorded once in `_loggedEvents`) — a static privacy whitelist baked into the client.
- `GetReader()` exposes the `ChannelReader` so `WamController` can drain it.

**Controller / rotation / bridge — `WamController` (`decompiled/WhatsApp.Root/WhatsApp/WamController.cs`).** Implements **both** `IWamBridgeToNative` (the bridge's JS-facing side, see §3.8) and `WhatsAppNative.IWamGlobalCallbacks` (the native ring buffer's callback). Constructed `new WamController(Wam.Storage)` and started via `_wam.Start()` from `AppModel` (`AppModel.cs:153`, `:180`; `AppModel.cs:220` registers `new WamBridge(_wam)` under name `WamBridge`).
- `Start()` (`WamController.cs:30-49`): hops to its serial dispatcher, `SetDeviceInfo()`, `WamGlobalBuffer.SetCallbacks(this)` to receive native rotation callbacks, drains the native buffer once (`Save(WamGlobalBuffer.Rotate())`), then loops forever on a **1-second** cadence: `await Task.WhenAll(reader.WaitToReadAsync(), Task.Delay(1s))` then `Save(_storage.GetReader().ReadEverythingBuffered())` — i.e. batch-drain the managed channel every second.
- `SetDeviceInfo()` (`:51-68`) pushes device facts into the **native** buffer: physical-memory MB (`SetMemClass`), logical CPU cores (`SetNumCpu`), `Major.Minor.Build` OS version (`SetOsVersion`) from `DeviceStatus.Instance`/`MachineSpec.Instance`.
- `IWamGlobalCallbacks.OnRotateRequested(bool isSending)` (`:70-74`): the native buffer asks the managed side to flush; controller re-drains via `Save(WamGlobalBuffer.Rotate())`. This is the **upload-rotation** trigger.
- `Save(byte[])` (`:87-100`) deserializes the native bytes via `WamEventsSerializer.Deserialize`; `Save(IEnumerable<Dictionary>)` (`:102-113`) JSON-serializes (Newtonsoft) and either pushes to JS (`_web.NotifyEventsSaved(json)`) or **queues** the JSON string if JS hasn't subscribed yet (`_queue`).
- `IWamBridgeToNative.Subscribe(IWamBridgeToWeb web)` (`:76-85`): JS registers its callback; the controller replays the buffered `_queue` then keeps it as `_web` for live delivery. Bridge contract: `IWamBridgeToNative.Subscribe` (`decompiled/WhatsApp.Root/WinRTAdapter/IWamBridgeToNative.cs:12`), `IWamBridgeToWeb.NotifyEventsSaved(string eventsJson)` (`IWamBridgeToWeb.cs:12`); the projected `WamBridge` also exposes an `EventsSaved` event (`WamBridge.cs:94-104`).

**`WamGlobalBuffer` is native (`decompiled/WhatsAppNativeProjection/WhatsAppNative/WamGlobalBuffer.cs`).** It is a WinRT projection of `WhatsAppNative.WamGlobalBuffer` (activation `ActivationFactory.Get("WhatsAppNative.WamGlobalBuffer")`, `:57`). `Rotate()` returns the serialized event bytes from the **native** ring buffer (`:94-97`); `SetCallbacks`/`SetMemClass`/`SetNumCpu`/`SetOsVersion` are static statics-interface calls (`:104-122`). The actual buffer storage and the binary layout it emits live in the native C++/Rust component (`WhatsAppNative`), consumed by the managed `WamEventsSerializer.Deserialize`. So WAM events arrive from **two** producers: native code logging into `WamGlobalBuffer`, and managed C# logging into `Wam.Storage` (`WamStorage`); `WamController` unifies both streams to JSON for the JS uploader.

**Native side mechanism — read from `WhatsAppNative.dll` strings [native-binary].** `objdump`/`rabin2` confirm the only exports are `DllCanUnloadNow` + `DllGetActivationFactory` (pure WinRT component; no named `Rotate`/`Submit` export), and the class symbols `.?AVWamGlobalBuffer@WhatsAppNative@@` / `.?AV__WamGlobalBufferActivationFactory@WhatsAppNative@@` / `.?AVWamWriter@WhatsAppNative@@` are compiled in. The buffer's internal C++ API is revealed by the embedded string table: `LogBuffer_Rotate`, `LogBuffer_GetRotatedPath`, `LogBuffer_SubmitRotated`, `LogBuffer_Submit`, `LogBuffer_SubmitUnlocked`, `LogBuffer_CatFile`. This shows `WamGlobalBuffer` is a **rotating on-disk log buffer**, not a pure in-memory ring: the managed `Rotate()` projection maps to native `LogBuffer_Rotate` (atomically swap the active buffer/file and hand back the rotated segment's serialized bytes), and `OnRotateRequested` is the native producer signalling that a rotation segment is ready to flush. The byte layout the managed `WamEventsSerializer.Deserialize` consumes is what `LogBuffer_*` emits; its exact on-buffer encoding still needs a decompiler pass (§6 Q8). The native producer surface is large — `strings` shows the `WAM_ENUM_CALL_SETUP_ERROR_TYPE_*` family and `wa_ts_logger_register`/`wa_ts_logger_deregister`/`wa_ts_logger_update_pid_by_register_id` lambdas, i.e. the VoIP/`wacall` time-series logger is a confirmed native event source feeding this buffer.

### 3.16 Time-spent / unified-session usage analytics (`WhatsApp.Models`)

A second analytics subsystem, distinct from WAM's queue, that *produces* time-spent `Wam.Event.*` events and a server-shared "unified session id". Doc 17 §UnifiedSession documents the **wire stanza** (`<ib><unified_session id>`) and doc 13 lists the **Smax family**; this section documents the native producer.

**`UnifiedSessionManager` (`decompiled/WhatsApp.Models/WhatsApp.Models.TimeSpendManager/UnifiedSessionManager.cs`), `IUnifiedSessionManager`.** Generates and shares a unified session id:
- Enabled only when `IConfig.UnifiedSessionVersion == 2` (`IsUnifiedSessionEnabled`, `:96-99`); TS-event logging gated separately by `IConfig.IsUnifiedSessionLogTsEventEnabled` (`:38`).
- On construction it subscribes to `IConnectionService.WhenIsConnectedChanged` filtered to `isConnected==true` → `OnAppConnected` (`:45-49`).
- `OnAppConnected` (`:64-72`): on the **first** connect it flips `_isAppFullyRegistered` (Interlocked) and `GenerateSessionId()`, then always `ShareSessionId()`.
- `GenerateSessionId` (`:88-94`): the id is **not random** — it is a *time bucket*: `((serverTimeUtc.ToUnixMilli() + 3 days) % 1 week).ToString()` (constants `Ms3Days=259200000`, `MsPerWeek=604800000`, `:24-26`, `:92`). So all clients in the same weekly window derive the same bucket id; it intentionally rotates weekly and on foreground.
- `ShareSessionId` (`:74-80`): when enabled and id non-null, calls `IConnectionService.ShareUnifiedSessionId(_sessionId)` — **this is the producer of the `UnifiedSession.Outgoing/ShareRequest` `<ib><unified_session id>` stanza cataloged in doc 17 §UnifiedSession** (id validated 1..64 chars at the Smax layer).
- `StartTrackForegroundEvent(IAppForegroundTracker)` (`:51-57`): subscribes to `WhenAppInForegroundChanged` (a `(bool IsAppInForeground, int WindowId)` observable, `IAppForegroundTracker.cs:7`); each foreground transition `OnAppForeground` regenerates the id and re-shares it (`:82-86`).
- `IUnifiedSessionManager` (the consumed interface, `IUnifiedSessionManager.cs`) exposes only `IsUnifiedSessionLogTsEventEnabled` + `GetSessionId()`; the recorders read the id to stamp TS events.

**Time-spent recorders (`WhatsApp.Models.TimeSpendManager` + `WhatsAppCommon`).** Orchestrated by `TimeSpentManager` (`decompiled/WhatsApp.Models/WhatsAppCommon/TimeSpentManager.cs`): holds a list of `ITimeSpentSessionDataRecorder`s built from a `TimeSpentFieldStatsRecordOption` flags bitset (`CreateSessionDataRecorders`/`TryCreateSessionDataRecorder`, `:89-117`), all driven on a serial `TimeSpentWorker` dispatcher (`:39`). `RecordEventNow(now, AppEventKind, windowId)` and `LogNavigationTo(TsSurface,…)` fan an `AppEvent` to every recorder (`:65-76`, `:119-125`). The `IWamEventProcessor` sink (`:11-16`) has `Process` (sampled) and `ProcessWithoutSampling` paths into the WAM pipeline of §3.15.
- **`AppEvent`/`AppEventKind`** (`WhatsAppCommon/AppEvent.cs`, `AppEventKind.cs`): the unit of input — `CreatedAt`, `Kind` (`Foregrounded, UserAction, Backgrounded, Suspended, Deactivated, EnterSurface`), `WindowId`, optional `TsSurface`, `NavigationTrackingId`, `NavigationSurfaceMetadata`. `NavigationSurfaceMetadata` is a record of `(int? GroupSize, TypeOfGroupEnum? GroupType)` (`NavigationSurfaceMetadata.cs:5`).
- **`TimeSpentBitArrayDataRecorder`** (`WhatsApp.Models.TimeSpendManager/TimeSpentBitArrayDataRecorder.cs`): records a **64-second** activity bitmap (`BitArrayLength=64`, `_bitArraySessionDuration=64s`, `:10-18`). ### 3.17 Account / build expiration gate (`WhatsApp.Models.Expiration`)

The client self-enforces a build lifetime ("update WhatsApp / this version will stop working"). Pure-logic core + a shell-side UI driver.

**`ExpirationStateReader` (`decompiled/WhatsApp.Models/WhatsApp.Models.Expiration/ExpirationStateReader.cs`).** `GetCurrentExpirationState(this IConfig)` (`:18-45`) returns `(DisplayExpirationState, daysLeft)`:
- `daysLeft` = `0` if `IConfig.IsUpgradeRequestedByServer()`, else `ceil((GetExpirationDate() - UtcNow).TotalDays)` (`:20`).
- `<= 0` → `DisplayExpired`; `> 7` → `DisplayNothing` and reset `DaysLeftAtLastExpiryAlert = 30`; `(0,7]` and `< DaysLeftAtLastExpiryAlert` → `DisplayWarning` if `>1` day left else `DisplayNearlyExpired`; otherwise `DisplayNothing` (`:22-43`). The `DaysLeftAtLastExpiryAlert` book-keeping throttles re-nagging to once per day-decrement.
- **`IsUpdateAvailable` is dead input inside the warning window.** The warning branch reads `bool flag = config.IsUpdateAvailable; if (!flag) flag = true;` (`ExpirationStateReader.cs:33-37`), forcing `flag` unconditionally `true`, so the `DisplayNothing` else of `flag ? … : DisplayNothing` (`:38`) is **dead code** — the client always shows the warning/near-expiry dialog inside the 7-day window regardless of whether a Store update is actually available. This ships (Release build, `BuildDetails.cs:9,17`); it reads as a deliberate force-true (the negation `if(!flag) flag=true` is the opposite of the intuitive `flag=false`).
- `DisplayExpirationState` enum: `DisplayNothing, DisplayWarning, DisplayNearlyExpired, DisplayExpired` (`DisplayExpirationState.cs:3-9`).

**Config adapter — `ExpirationStateAdapter` (`decompiled/WhatsApp.Root/WhatsApp.SystemIntegrations/ExpirationStateAdapter.cs`)** binds the reader to `Settings`:
- `IsUpgradeRequestedByServer()` = `settings.LoginFailedUpgradeRequiredVersion == PackageInfo.Version.ToString()` — i.e. the **server told this exact build to upgrade** on login failure (`:22-25`).
- `GetExpirationDate()` = `settings.ServerExpirationOverride` if set, else `BuildDetails.BuildTime + (60 + settings.JitterDays)` days — a **60-day build lifetime** plus a per-install jitter, overridable by the server (`:27-37`).
- `IsUpdateAvailable` is passed in from a Store check.

**Shell driver — `App.StartExpirationStateCheck` (`decompiled/WhatsApp.Root/WhatsApp.SystemIntegrations/AppExpirationCheckExtensions.cs`)**, invoked late in main-window init (`App.cs:337`, `this.StartExpirationStateCheck(MainWindow.WebView, model.Settings)`):
- `CheckUpdates()` queries the Store (`StoreContext.GetAppAndOptionalStorePackageUpdatesAsync().Count > 0`, `:60-76`) for `IsUpdateAvailable`, then computes the state (`:15`).
- `DisplayNothing` → return; `DisplayWarning`/`DisplayNearlyExpired` → an *"update required in N days"* dialog, dismissible, `shouldCloseApp=false`; `DisplayExpired` → *"app expired"* dialog with `DialogBehavior.IgnoreEsc|IgnoreOverlayClick` and `shouldCloseApp=true` (a hard lockout) (`:26-43`).
- Records `settings.DaysLeftAtLastExpiryAlert = num` (`:42`); the "Update" button launches the Store URI (`PackageInfo.AppStoreUri`) and `application.Close()` on success; an expired dialog dismissal also `application.Close()`s (`:44-57`).

1. **[CANNOT RESOLVE STATICALLY]** **Native `WhatsAppNativeInit.Setup()` body** — the projection is confirmed (`WhatsAppNativeInit.cs:129-132`), and `Setup()` is a pure WinRT ABI thunk: `__IWhatsAppNativeInitPublicNonVirtualsMethods.Setup(_objRef…)` (`WhatsAppNativeInit.cs:129-132`).
