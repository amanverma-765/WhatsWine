# 90. Linux / Electron Porting Roadmap

> Target source: Meta **native** WhatsApp for Windows (`WhatsApp.Root.exe`, WinUI 3 / Windows App SDK 1.6, v2.2607.106.0).
> Port target: `whatsapp-desktop/` (Electron Forge + Vite + TypeScript scaffold).
> This is a **synthesis** document. It builds on the per-mechanism reverse-engineering docs already in `docs/` and does **not** re-derive their `path.cs:LINE` evidence; every claim here is traceable to a cited sibling doc (e.g. "doc 11 §3.7"), and through that doc to the decompiled C#. Where this doc transcribes a numeric constant, the number is copied verbatim from the cited doc's transcription of source — not paraphrased.
> Per-doc evidence index: `01` build/packaging, `10` transport, `11` Noise, `12` binary-node, `13` Smax, `14` XMPP/IQ, `15` send, `16` receive/offline, `17` stanza families, `20` Signal e2e, `21` auth/pairing, `30` app-shell, `31` WebView2 bridge, `32` storage, `33` protobuf, `40` media, `41` VoIP, `42` notifications.

## 1. Purpose & Scope

This document converts the eighteen per-mechanism port-mapping analyses into a single **phased, buildable roadmap** for re-hosting the native Windows WhatsApp client on Linux as an Electron app. It answers, for each milestone:

1. **Windows mechanism** — what the native client does (with the sibling-doc anchor).
2. **Recommended Node/Electron/native stack** — the concrete library/API to use.
3. **waweb reuse** — what comes for free from the `waweb-source-bundle/` JS bundle, vs. what must be re-implemented natively.
4. **Effort & risk** — relative sizing and the failure modes to watch.

### 1.0 The one architectural decision that governs everything

The native Windows app is a **hybrid**: a thin native C#/C++/Rust shell hosting the **same WhatsApp Web JS bundle** inside WebView2, with native capabilities exposed to JS as `window.chrome.webview.hostObjects.<Name>` bridge objects. Critically, **the native transport handles only `iq`/`success`/`failure` post-login; all rich stanzas (`message`/`receipt`/`notification`/`presence`/`chatstate`) and the entire Signal e2e engine, message UI, offline drain, and receipts run in the JS bundle** (doc 10 §3.8, doc 30 §5, doc 31 §5).

This forces a strategic fork that recurs in every milestone below:

- **Strategy A — "Shell reuse" (recommended default).** Electron `BrowserWindow` loads `https://web.whatsapp.com/?windows=1&...`; you re-implement only the **native host-object bridges** the bundle calls. The bundle brings its own Noise/WebSocket transport, FunXMPP codec, Signal engine, message/receipt/offline machinery, and most protobufs. Native work shrinks to crypto/storage/media/VoIP host objects + the bridge IPC surface (doc 01 §5, doc 31 §5).
- **Strategy B — "Headless native client."** Terminate the protocol in Node (no WebView for networking). You must re-implement the whole FunXMPP stack, Noise handshake, Signal e2e, message pipeline, offline drain. This is the bulk of a from-scratch client and is **not** copyable wholesale from this C# shell, because the native FunXMPP layer here is **migration-scoped and incomplete** — it lacks `message`/`receipt`/`presence` handling (doc 15 §5, doc 14 §5). Baileys (`@whiskeysockets/baileys`) / whatsmeow are the better references for Strategy B.

**This roadmap is structured for Strategy A** (lowest risk, mirrors the Windows architecture), and flags inside each milestone what changes if you go Strategy B. The milestones M1–M3 contain a self-contained Node Noise/transport/Signal track that is **optional under Strategy A** (the bundle does it) but **mandatory under Strategy B**.

### Out of scope
Per-mechanism wire details (transcribed in the sibling docs), the WhatsApp account-safety/ToS implications of a third-party client, and the instruction-level native C++/Rust engine bodies (the legacy Ghidra dumps were empty, but radare2 statically recovered the native-crypto surface — the X25519 Montgomery constant `a24=121665`, SHA-512 IV, and `Curve25519::{Derive,Sign,Verify,GenKeyPair}` are byte-evidenced in `WhatsAppNative.dll`, doc 96; only the per-instruction Sign/Verify body and the `IVoip` engine internals remain un-read, doc 41 §4).

## 3. How It Works — The Phased Roadmap

Dependency order (each milestone assumes the prior ones):

```
M0 scaffold + IPC bridge        ─┐
M1 transport + Noise            ─┤ (M1–M3 optional under Strategy A; mandatory under B)
M2 auth / pairing               ─┤
M3 Signal e2e                   ─┘
M4 messaging (send/receive/offline)
M5 storage (SQLite + keystore)
M6 media (transcode + mmg)
M7 VoIP (calling)
M8 notifications + push + tray + activation
```

Under **Strategy A** the critical path is essentially **M0 → M5 → (M6/M7/M8 in parallel)**: once the bridge surface (M0) and the storage/keystore host objects (M5) exist, the bundle's own transport/Noise/Signal/messaging (M1–M4) lights up for free, and M6/M7/M8 are the remaining native services. Under **Strategy B**, the path is strictly sequential M1 → M2 → M3 → M4.

### M0 — Scaffold & the host-object IPC bridge

**Windows mechanism.** A full-trust WinUI 3 host opens one `MainWindow` with a single `WebView2` navigated to `https://web.whatsapp.com/` with a **native-context query string** the bundle keys off — `windows=1`, `windowsBuild`, `windowsBuildType`, `bridgeError=1`, `launchContext`, `osBuild` (doc 31 §5.1, doc 01 §5 "Native-context query string", doc 30 §5). Native capabilities are injected via `AddHostObjectToScript`, so the bundle calls them at `window.chrome.webview.hostObjects.<Name>` with WebView2-specific proxy semantics: `options.defaultSyncProxy = true`, `forceAsyncMethodMatches = [/Async$/, /AsyncWithSpeller$/]`, `options.ignoreMemberNotFoundError = true`, and a `hostObjects.sync.<Name>` synchronous variant (doc 31 §5.2). There is also a **legacy comma-string IPC** fallback (`split(",")` / `NativeResponseToJS`, negotiated via `CompatibilityGetSupportedNativeIpcs`) that the bundle still ships (doc 31 §5.2).

**Recommended stack.**
- `BrowserWindow` (`minWidth:800, minHeight:600`, `titleBarStyle:'hidden'` to mirror `ExtendsContentIntoTitleBar`) + `session.fromPartition('persist:wa')`; `app.setPath('userData', …)` for the WebView2 user-data-folder analogue (doc 30 §5).
- `loadURL(buildUrl())` and re-append the native query string in `will-navigate`/`will-redirect`; **`windows=1` is load-bearing** — without it the bundle falls back to plain web and never calls the bridges (doc 31 §5.1, doc 01 §5).
- Set the `X-WA-WebView2-Version` (or equivalent) header via `session.webRequest.onBeforeSendHeaders` (doc 31 §5.1).
- **Preload shim of `window.chrome.webview.hostObjects`** that reproduces the Proxy contract over Electron IPC: sync methods via `ipcRenderer.sendSync` (blocks the renderer — acceptable, the WebView2 sync proxy also blocks), `*Async`/`*AsyncWithSpeller` via `ipcRenderer.invoke`, plus a callback registry so native can call `*ToWeb` events back into the bundle (`webContents.send` re-emitted on `window`). Reproduce `ignoreMemberNotFoundError` and the `/Async$/` carve-out (doc 31 §5.2, doc 30 §5). Also shim `chrome.webview.postMessage` for the legacy comma-IPC path (`method,args…,txId` → reply `txId,NativeResponseToJS,0,<json>`); `bridgeError=1` tells the bundle to expect it (doc 31 §5.2).
- WebView lockdown maps directly: `setWindowOpenHandler`→`shell.openExternal`; `session.setPermissionRequestHandler(cb=>cb(true))` (mic/cam/notifications/clipboard); block accelerators via `before-input-event`; `setVisualZoomLevelLimits`; auto-accept dialogs; reskin context menu via `webContents.on('context-menu')` (doc 30 §5, doc 31 §5.1).

**waweb reuse.** The renderer payload is reused unchanged; the entire chat/UI/business layer is platform-agnostic. M0 delivers **only** the host: a Chromium window + the bridge namespace shim. The full bridge list to wire (each → an `ipcMain.handle`/`webContents.send` pair): `SQLiteBridge`, `ClientKeyBridge`, `ServerEncKeySaltBridge`, `VoipBridge`, `VoipSignalingBridge`, `MediaFilesBridge`, `MediaTranscodingBridge`, `ContactsBridge`/`PopulatedContactsBridge`, `WamBridge`, `ConnectionBridge`/lifecycle, `NativeAppStateBridge`, `WebUpdateBridge`, `AdvBridge`, `TcTokenBridge`, `AbPropsBridge`, `PreferencesBridge`, `PicturesBridge`, `LinksPreviewBridge`, `SeamlessMigrationBridge`, `SystemIntegrationsBridge`, `SharesheetBridge`, `AppActivationBridge`, plus the thin/pure-JS ones (`RateAppBridge`, `ScalingControlBridge`, `BrowserExtensionsBridge`, `DebugFeaturesBridge`, `TouchpadFix`) (doc 31 §5.2, doc 30 §5, AppxManifest bridge registry).

**Effort & risk.** Effort: **Medium** (the bridge shim is fiddly but mechanical). Risk: **High-fidelity required** — the sync-proxy + `ignoreMemberNotFoundError` + `/Async$/` semantics are WebView2-specific; subtle mismatches break the bundle quietly (doc 31 §5.4 item 1). This milestone is the **core porting surface** and gates Strategy A entirely. Also reproduce `WebContentUpdateManager`'s `/spin?subplatform=win_hybrid&revision=…` health-poll + reload loop with a Linux `subplatform` value (doc 30 §5, doc 31 §5.2/§5.4). DevTools are enabled in this build (`IsDevEnvironment()==true`) — gate by build flag (doc 31 §5.4 item 4). Drop WebView2-install / `NewBrowserVersionAvailable` / Code Verify binary-transparency paths (doc 01 §5, doc 31 §5.4).

### M1 — Transport & Noise handshake  *(optional under Strategy A; mandatory under Strategy B)*

**Windows mechanism.** A `ConnectionManager` state machine dials **raw TCP** (no OS TLS) to the chat bootstrap host with **port cycling `443 → 5222 → 80`** (`int[] Ports = { 443, 5222, 80 }`), pinned to the last-good port across restarts; port 80 wraps the binary stream in an **HTTP/1.1 chunked tunnel** (`ChunkedHttpSocket`) (doc 10 §3.1–3.4). *Re-verified this session (resolves §6 item 9, static facts):* the tunnel CONNECT line is verbatim `"POST /chat HTTP/1.1\r\nHost: c.whatsapp.net\r\nUser-Agent: Mozilla/5.0 (compatible; WAChat/1.2; +http://www.whatsapp.com/contact)\r\nTransfer-Encoding: chunked\r\n\r\n"` (`ChunkedHttpSocket.cs:214`, chunked-response parse at `:87-108`) — copyable, but whether current servers still accept this legacy path is unverified and safe to skip initially (443 covers nearly all networks). The bootstrap **DNS host is `g.whatsapp.net`** (plus `g-fallback.whatsapp.net` and `e1..e16.whatsapp.net`); `s.whatsapp.net` is the *JID/stanza* domain, **not** the dialed host (doc 10 §3.10, anchor correction). Over the socket sits a **3-byte big-endian length frame** layer, then a **Noise handshake**:

- Three variants, ASCII protocol names as the transcript seed (doc 11 §3.1, doc 10 §3.6): `Noise_XX_25519_AESGCM_SHA256` (fresh, the live path), `Noise_IK_25519_AESGCM_SHA256` (resume; **dead code in this build** — `serverStaticPublic` is always `null`, doc 11 §3.11), `Noise_XXfallback_25519_AESGCM_SHA256` (36 bytes, hashed).
- Wire prologue before the first Noise message: optional edge header `ED\0\x01` = `{0x45,0x44,0x00,0x01}` + 3-byte-len + blob, then the version header **`57 41 06 03`** = ASCII `"WA"`, protocol/Noise version **`6`**, token-dictionary version **`3`** (`WAPDefaultTokenDictionary.DictionaryVersion = 3`) (doc 11 §3.4, doc 10 §3.6). *Re-verified this session (resolves §6 item 5):* the header is built as `byte[4]{ 87, 65, 6, 0 }` with the third byte overwritten at runtime by `obj[3] = (byte)FunXMPP.Dictionary.GetDictionaryVersion()` (`HandshakeHandler.cs:47-48`); the **third byte is literally the dictionary version** (3 here), not an independently negotiated field — pin it to your matched token table.
- Symmetric state: `MixHash(d): h = SHA256(h‖d)`; `MixKey(dh): ck,k = HKDF-SHA256(salt=ck, ikm=dh, L=64)` then `n=0`; `Split(): write,read = HKDF-SHA256(salt=ck, ikm="", L=64)` → first 32 = client→server **write** key, next 32 = server→client **read** key (doc 11 §3.3).
- Handshake AEAD: AES-**256**-GCM, **12-byte big-endian** nonce (4 zero bytes + 64-bit BE counter), **AAD = current transcript hash `h`**, tag (16 B) **appended** to ciphertext; `decrypt` returns **null** on auth failure (doc 11 §3.7).
- Server cert: a `CertChain` protobuf validated against a **pinned 32-byte Curve25519 root** `14 23 75 57 4D 0A 58 71 66 AA E7 1E BE 51 64 37 C4 A2 8B 73 E3 69 5C 6C E1 F7 F9 54 5D A8 EE 6B` (issuer `"WhatsAppLongTerm1"`, serial `0`); leaf key must byte-equal the handshake server-static; **time checks disabled for WA6** (doc 11 §3.8).
- Post-login transport frames: same 3-byte frame, but AES-256-GCM with **AAD = null**, **independent** `writeNonce`/`readNonce` 12-byte BE counters starting at 0 and never reset; first plaintext byte is a compression flag (`0x02` = zlib-deflate, `0x00` = plain), 32 MiB pre-encryption cap (doc 11 §3.9, doc 10 §3.8). *Re-verified this session (resolves §6 item 2):* the nonce is unambiguously **big-endian** — `WAProtocol.LongToByteArray(counter, 12)` writes MSB→LSB into the low 8 bytes (`WAProtocol.cs:53-66`), and is the sole nonce source for both the handshake cipher (`HandshakeCipher.cs:71,78`) and the transport `writeNonce++`/`readNonce++` (`StanzaWriter.cs:76`, `EncryptedBytesReceiver.cs:26`). Canonical Noise/most JS libs are little-endian → you desync after frame 0 if you copy them.
- Liveness has **no native heartbeat** — it is purely the socket **read timeout**: `LoginTimeout = 30 s` during handshake, `ForegroundPingTimeout = 100 s` after login; the actual `w:p` ping IQ to `s.whatsapp.net` is **scheduled/sent from the JS bundle**, not native (doc 10 §3.10). Reconnect backoff is `min(Fibonacci(n+1), 3600) × jitter`, jitter ∈ [0.75, 1.25), clamped to 4 s in background (doc 10 §3.10).

**Recommended stack.**
- `node:net` `Socket` for raw TCP (no `node:tls` — Noise replaces TLS); `socket.setTimeout(ms)` to replicate the read-idle `CancelAfter`; an early-send queue flushed on `'connect'`; Node 20+ `autoSelectFamily` for happy-eyeballs, `node:dns` `resolve4`/`resolve6` + a static fallback IP list (doc 10 §5).
- Crypto: `@noble/curves` (`x25519`) + `@noble/hashes` (`sha256`, `hkdf`, `hmac`) + Node `crypto.createCipheriv('aes-256-gcm')`. **Hand-port the state machine from doc 11 §3.6–3.7** rather than using a stock Noise library (doc 11 §5, doc 10 §5).
- zlib via Node `zlib.deflate`/`inflate` (**zlib-wrapped, not `deflateRaw`** — SharpZipLib emits zlib headers; verify against a captured frame) (doc 11 §5, doc 10 §5).
- Cert chain: `protobufjs`/`ts-proto` for `CertChain`/`NoiseCertificate`; embed the pinned root verbatim; XEdDSA verify via `@noble/curves` or `@signalapp/libsignal-client` (doc 11 §5).

**waweb reuse.** Under **Strategy A this entire milestone is skipped** — the JS bundle runs its own Noise + transport (over WebSocket when it behaves as plain web.whatsapp.com), and the native shell only proxies it for background/push reasons (doc 11 §5, doc 21 §5). *Re-verified this session (resolves §6 item 10):* the bundle ships its own transport literal **`wss://web.whatsapp.com/ws/chat`** plus a WANoise handshake module (`waweb-source-bundle` grep), so the renderer payload is **self-sufficient for networking** as plain web.whatsapp.com — Strategy A's core assumption is now evidence-backed, not inferred (the only residual is the operational risk that WhatsApp could server-gate the web transport for a non-browser origin). The waweb bundle is **not** a reusable source for a *headless* Noise port (its transport is browser-WebSocket-based, not this native raw-TCP Noise channel) — so Strategy B must hand-port M1 (doc 11 §5 "Cannot reuse from JS bundle").

**Effort & risk.** Strategy A: **None** (skipped). Strategy B: **High** effort, **highest crypto risk** in the project. The named correctness traps (doc 11 §5, doc 10 §5): (1) **big-endian** nonce (canonical Noise/most JS libs are little-endian → desync after frame 0); (2) HKDF salt/IKM orientation (`Perform(L,key,salt)` ⇒ `HKDF(IKM=key, salt=salt)`); (3) **zlib-wrapped** (not raw) deflate; (4) **AAD = transcript hash during handshake but null for transport frames**; (5) the pinned root + leaf-key-equals-server-static check; (6) **ping scheduling lives in JS** — a headless port must add its own `<iq xmlns="w:p" to="s.whatsapp.net" type="get">` loop at < 100 s or the read timeout kills the socket (doc 10 §5 gaps). Native dependency: only **Curve25519** (X25519 + XEdDSA) is native in this layer; everything else (GCM, SHA-256, HKDF, deflate) is managed and trivially portable (doc 11 §4).

### M2 — Authentication & companion pairing  *(JS-resident under Strategy A)*

**Windows mechanism.** Two flows (doc 21 §3): **(A)** the native **Noise login** carrying the `ClientPayload` blob (the device authenticates by proving possession of the static key over the Noise channel); **(B)** **companion pairing** (QR / phone-code) which derives the account `ClientKey`, signs ADV (account/device) material with Curve25519, and is **implemented in the JS bundle** — the native side only vends `AdvBridge.Verify`, `ClientKeyBridge` (Set/Clear/Get), and `ServerEncKeySaltBridge` (doc 21 §3.2, §5). The `ClientPayload` login envelope (doc 11 §3.10, doc 21 §3.1) carries: `Username` = phone# as uint64 (field 1), `Passive = true`, `Pull = true`, `PushName`, `Device`, `connect_reason = USER_ACTIVATED`, `LidDbMigrated`, and a `UserAgent` with **`platform = WINDOWS = 13`** (enum re-confirmed this session 0-indexed `ANDROID(0), IOS(1), … SMB_ANDROID(10), KAIOS(11), SMB_IOS(12), WINDOWS(13), WEB(14)…` at `ClientPayload.cs:61-82`, set at `HandshakeHandler.cs:241`; the bundle independently keys off Windows via the `win_hybrid` subplatform — `WAWebEnvironment.isWindows` → `"win_hybrid"`, so both surfaces must stay consistent — see §6 item 7), `release_channel = RELEASE`, app version split from `a.b.c.d`, `Mcc/Mnc = "000"`, OS/manufacturer/device/locale fields. `ClearClientKey` triggers a full app relaunch (`App.Restart(UserLogout)`) (doc 21 §5).

**Recommended stack.**
- `AdvBridge.Verify` → `@noble/curves` (XEdDSA) or `@signalapp/libsignal-client`, exposed via `contextBridge`; preserve base64-in/out and swallow-exception→`false` semantics. **Lowest-risk option: drop native verify and let the bundle use its own `WAWebCryptoCurve25519VerifySignature`** — validate the bundle is self-sufficient first (doc 21 §5, doc 20 §5).
- `ClientKeyBridge`/`ServerEncKeySaltBridge` → `contextBridge` methods over an encrypted KV store; `ClearClientKey` → `app.relaunch(); app.exit()` (doc 21 §5, doc 20 §5). At-rest protection: see M5.
- `ClientPayload`/`HandshakeMessage` → `protobufjs`/`ts-proto`; **the single most load-bearing protobuf in the shell** (doc 33 §5). Keep `WINDOWS=13` or pick an appropriate platform — see risk.

**waweb reuse.** **Huge** — pairing, QR/code, ADV signatures, and ClientKey derivation are already portable JS. Strategy A's pairing needs **zero new crypto**: the bundle does it; the port just vends `AdvBridge`/`ClientKeyBridge`/`ServerEncKeySaltBridge` host objects (doc 21 §5). The only place native-equivalent crypto is unavoidable is the **flow-A Noise login** *and only if* you also replace the bundle's networking (M1, Strategy B) — the bundle's own WebSocket transport otherwise covers it (doc 21 §5, inference).

**Effort & risk.** Strategy A: **Low** (three thin host objects + relaunch wiring). Strategy B: **Medium-High** (the `ClientPayload` builder + persisting the Noise static keypair and cached server-static for IK resume). Risks (doc 21 §5): (1) impersonating `platform = WINDOWS` from Linux is a **fingerprinting risk** and may trigger Windows-specific server feature gating or break on server changes; (2) the `ClientKey` is the portable account login secret — storing it without a keyring is a real downgrade from DPAPI (gate on `safeStorage.isEncryptionAvailable()`); (3) if you re-implement flow A, persist the cached server-static or you always pay full `XX` instead of `IK` resume.

### M3 — Signal protocol / end-to-end encryption  *(JS-resident under Strategy A)*

**Windows mechanism.** The **Signal engine is entirely JS** in this hybrid — X3DH, the double ratchet, sender-keys, and message AEAD all run in the WebView bundle; the native side only provides `WhatsAppNative.Curve25519` (`Derive`/`Sign`/`Verify`/`GenKeyPair`) used by `AdvBridge.Verify`, the Noise handshake, and cert-verify, plus the key-material host objects (doc 20 §3, §5). Persisted Signal state (`SessionStructure`, `SenderKeyRecordStructure`, pre-keys, identity) is serialized/stored **by JS** (in `genericStorage.db`) using protobuf schemas that are **upstream-libsignal-compatible** (field tags match, doc 20 §3.1–3.3, §5). The pre-key upload IQ `<iq xmlns="encrypt">` uses load-bearing field sizes (4/32/3/32/32/64 bytes, `<type>=0x05`) (doc 20 §3.4). HKDF/HMAC are managed RFC-5869 with salt defaulting to 32 zero bytes (doc 20 §3, doc 41 §3.6).

**Recommended stack.**
- Curve25519 host object → **`@signalapp/libsignal-client`** (ships the exact `Curve.calculateSignature`/`verifySignature` = **XEdDSA**, bit-compatible), or `@noble/curves` (`x25519` + XEdDSA). **Do not substitute tweetnacl Ed25519** — XEdDSA ≠ plain Ed25519 (doc 20 §5, doc 15 §5).
- Key-material host objects (`ClientKey`, `ServerEncKeySalt`) → two `contextBridge` methods over the encrypted store from M5 (doc 20 §5).
- HKDF/HMAC/random → Node `crypto.hkdf`/`hkdfSync` (sha256) or `@noble/hashes`; `crypto.randomBytes` (doc 20 §5).
- (Strategy B only) full session/sender-key/pre-key persistence → `@signalapp/libsignal-client`'s `SessionRecord`/`SenderKeyRecord`/`PreKeyRecord`, which read/write the **identical** protobuf schema documented in doc 20 §3 (doc 20 §5, doc 33 §5).

**waweb reuse.** **Maximal.** For the Signal layer the native surface shrinks to **~4 tiny host methods** (`AdvBridge.verify`, `ClientKey.set/get/clear`, `ServerEncKeySalt.set/get/clear`) plus the encrypted blob store. The cryptographically delicate X3DH/ratchet/sender-keys/message-AEAD is carried entirely by the unmodified bundle — provided the curve host object is **XEdDSA-exact** (doc 20 §5).

**Effort & risk.** Strategy A: **Low** (4 host methods). Risks (doc 20 §5): (1) **XEdDSA exactness is the #1 correctness risk** — prefer `@signalapp/libsignal-client` to guarantee bit-compat (the native curve is byte-confirmed X25519+XEdDSA via radare2, doc 96, so libsignal is provably the right primitive); (2) the bundle expects the `chrome.webview.hostObjects.<Name>` shape (async-returning, base64 string args) — emulate it precisely (M0); (3) at-rest protection of `ClientKey`/identity DB downgrades from DPAPI to `safeStorage`/libsecret, changing the Linux trust boundary (M5). Native dependency: **only Curve25519** — its X25519+XEdDSA scheme is now byte-evidenced in `WhatsAppNative.dll` via radare2 (X25519 `a24=121665`, SHA-512 IV, and `Curve25519::{Derive,Sign,Verify,GenKeyPair}` all statically present; doc 96), so `@signalapp/libsignal-client` is a confirmed bit-compatible substitute and the only un-read residual is the per-instruction Sign clamping/nonce, which is moot for the port (doc 20 §4–§5, doc 96).

### M4 — Messaging: send / receive / offline drain  *(JS-resident under Strategy A)*

**Windows mechanism.** The **live message pipeline is JS in WebView2** (doc 15 §5, doc 16 §5). Outbound: the JS bundle builds `Message`/`WebMessageInfo` protobufs, Signal-encrypts to `msg`/`pkmsg` (or `skmsg` for groups), FunXMPP-encodes, and sends over its own transport; the native FunXMPP send path (`BinTreeNodeWriter`, `StanzaWriter`) exists but is **migration-scoped and does not handle `message`/`receipt`/`presence`** (doc 15 §3.5–3.7, §5). Inbound: `BinTreeNodeReader` parses the FunXMPP tree; the **offline drain** is an `<ib>` sequence (`offline`/`offline_preview` with an `offline` count 0–1024, `offline="1"` backlog markers, terminated by `priority_offline_complete`), with **aggregate-receipt batching** to avoid hammering the server on reconnect — but the real offline/receipt/sync-state semantics live in the **minified JS** (doc 16 §3.7–3.8, §5). Binary node encoding uses the FunXMPP token dictionary (236 primary + 4×256 secondary, `DictionaryVersion = 3`) and IQ correlation via `IqRequestsTracker` (uppercase-hex incrementing id, one-shot pop) (doc 12 §3, doc 14 §3). The Smax framework (510 generated families) is the IQ grammar spec but is **largely unused by native** — message/receipt/group traffic is JS; native originates only a few low-level IQs (Pings, PreKeys, DirtyBits) (doc 13 §5, doc 17 §5).

**Recommended stack.**
- Strategy A: **nothing new** — the bundle owns send/receive/offline/receipts end-to-end; M4 is satisfied once M0+M5 exist.
- Strategy B (headless): port `BinTreeNodeWriter`/`Reader` + token tables (copy `WAPDefaultTokenDictionary.cs` **verbatim**, pin `DictionaryVersion = 3`) or **reuse Baileys' `WABinary` codec** (byte-for-byte the same format, but **diff its token table** against this build's — versions drift) (doc 12 §5, doc 15 §5, doc 16 §5). `protobufjs`/Baileys protos for `MessageKey`/`WebMessageInfo`/`Message`. Signal encrypt via `@signalapp/libsignal-client`. IQ correlation → `Map<string,{resolve,reject}>`, **promisified with per-IQ timeouts** the original lacks (doc 14 §5). Offline drain → implement the `<ib>` type dispatch, `offline_preview` count, `priority_offline_complete` terminator, and aggregate-receipt batching (doc 16 §5). Smax runtime → ~400 lines of TS validators returning a `Result<T,SmaxError>`; port only the families the shell originates natively (Pings, PreKeys, DirtyBits, optionally Presence/Chatstate) (doc 13 §5, doc 17 §5).

**waweb reuse.** **Decisive in favor of reuse.** Doc 16 §5 states the decompiled evidence "strongly favors reuse" — the real message/offline/receipt logic is already JS and already runs over a WebSocket; loading WhatsApp Web in Electron makes most of M4 "come for free." The native FunXMPP stack here is explicitly **not a complete chat client to copy from** (doc 15 §5 gap 2).

**Effort & risk.** Strategy A: **Low** (config/glue only). Strategy B: **Very High** — this is the bulk of a from-scratch client; Baileys/whatsmeow are better references than this incomplete C# layer. Risks (doc 12/14/15/16 §5): token-dictionary **version drift** silently corrupts decode; attribute **JID typing is lost on receive** (re-derive `KVType` before re-encoding outbound); a **single GCM failure desyncs the nonce** (treat any auth failure as fatal-to-stream and reconnect); `JID_FB(246)` may arrive even though this client rejects it; attribute **order must be preserved** on the wire (use ordered arrays, not plain objects); no native dependency in this layer (pure TS).

### M5 — Local storage: SQLite + keystore

**Windows mechanism.** `WhatsAppNative.Sqlite` is an **encrypted SQLite engine** with a **custom AES+HMAC page codec — not stock SQLCipher** (no `cipher_version`/`kdf_iter`/`PRAGMA key` markers anywhere in the DLL, doc 94/96) — using WAL, `synchronous=FULL`, `secure_delete=ON` PRAGMAs (doc 32 §3, §5). The JS bundle drives the main message DB (`genericStorage.db`) entirely through one host object, `SQLiteBridge.ExecuteSqlite(string[][]) → SqliteResult[]` (the JS owns the schema and queries) (doc 32 §5). Native-owned typed stores have their own schemas: `contacts` (v23), `abprops` (v1), `mediaDownloads` (v2), `contactsState` (v1), `session.db`, settings KV (doc 32 §5). *At-rest — now fully traced (doc 94 §2, §6 item 6).* A **live on-disk dump** confirms all **seven** native DBs are page-encrypted: none starts with `SQLite format 3\0`, and **every file is an exact multiple of 4096 bytes with a high-entropy 16-byte prefix** ⇒ a **custom AES+HMAC page codec (NOT stock SQLCipher)** with 4096-byte pages + a per-DB 16-byte salt prefix (the page-layout resembles SQLCipher's, but there are no `cipher_version`/`kdf_iter`/`PRAGMA key` markers — doc 94/96). The page-cipher secret per DB is: **`session.db`** ← `EnsureFixedSecretSize(ProtectionUtils.ProtectBytes(StaticKeyBytes))` = **DPAPI `DataProtectionProvider("LOCAL=user")`** over the hardcoded 16-byte `StaticKeyBytes` (`LoginSessionManager.cs:28-32,66`; `ProtectionUtils.cs:18`) — **user-bound**; **`nativeSettings.db`** ← `EncryptionUtils.EncryptBytes(ClientKey, StaticKeyBytes)` = **AES-CBC-PKCS7** under **PBKDF2-SHA256(password=ClientKey, salt=`SystemIdentification.GetSystemIdForPublisher().Id`, 10000 iters → 32 B)**, IV also PBKDF2-derived (`LoginSessionManager.cs:169-170`; `EncryptionUtils.cs:42-54`) — **machine-bound**; the **five data DBs** (`contacts`/`contactsState`/`abprops`/`genericStorage`/`mediaDownloads`) ← `LegacyDbSecret` = `SecureRandomBytes(32)` generated at login and stored in `nativeSettings.db` (`LoginSessionManager.cs:97-98`). The chain roots in two machine/user-bound secrets (DPAPI master key + `GetSystemIdForPublisher`), **neither recoverable from an app-folder dump → the native DBs are not decryptable offline**. The exact page-cipher KDF iter / HMAC variant stays inside the statically-linked SQLite codec in `WhatsAppNative.dll` (no `sqlcipher`/`PRAGMA key`/`kdf_iter` strings; `bcrypt.dll` imported for RNG only) — the one residual, but moot for a port (see below). Account isolation: `sessions/{SHA1(ClientKey)}/` with a single-account on-disk **wipe-inactive** policy (`WipeInactiveSessionFolders`, `LoginSessionManager.cs:113-135`) and a `LegacyDbSecret` regenerated on login (doc 32 §5, doc 21 §5). Background maintenance: passive checkpoint 2 h, truncate 12 h, integrity check 7 d, optimize 24 h (doc 32 §5). **Critically (doc 94 §3, doc 95):** the live dump proves these native (custom-codec) encrypted DBs are only a **cache/mirror** — the cryptographic source of truth (Signal identity/prekeys/sessions/sender-keys, the Noise `ClientKey`, and the SyncD app-state — the five live SyncD collections being `critical_block`/`critical_unblock_low`/`regular`/`regular_high`/`regular_low`) lives **in plaintext** in the WebView2 IndexedDB (`LocalCache/EBWebView/.../https_web.whatsapp.com_0.indexeddb.leveldb`, Chromium LevelDB), so the machine-bound page codec adds little confidentiality beyond OS file ACLs.

**Recommended stack.**
- **`better-sqlite3` built against SQLCipher** (or `@journeyapps/sqlcipher`) for a **fresh** encrypted store; apply the same WAL/`synchronous=FULL`/`secure_delete` PRAGMAs. (Note: stock SQLCipher will **not** read the original Windows DBs — those use a custom AES+HMAC codec, not stock SQLCipher params — so this is for a new at-rest store, not for migrating existing files; doc 32 §5, doc 94/96.)
- `SQLiteBridge.ExecuteSqlite` → keep the **exact contract**: `ipcMain.handle('execute-sqlite', (string[][]) => SqliteResult[])` forwarding to a better-sqlite3 worker; the `LastInsertedRowId`/`RowsAffected`/`Rows`/`ColumnNames`/`Error` field names and the 8-keyword classifier are **load-bearing for the JS side** (doc 32 §5).
- At-rest master key → **Electron `safeStorage`** (libsecret/KWallet) replacing DPAPI; PBKDF2/AES-CBC via Node `crypto`; substitute the `SystemIdForPublisher` salt with a stable per-install id (`/etc/machine-id` + app salt, or a random id stored in `safeStorage`) (doc 32 §5, doc 21 §5).
- Account dir → `app.getPath('userData')/sessions/{sha1(ClientKey)}/` via Node `crypto.createHash('sha1')` + `fs`; replicate the single-account wipe carefully (it is destructive) (doc 32 §5).
- Logout → `app.relaunch(); app.exit()` (mirrors `App.Restart(UserLogout)`). Background maintenance → Node timers + `pragma('wal_checkpoint(TRUNCATE)')`/`integrity_check`/`ANALYZE` (doc 32 §5).
- The native typed stores (contacts/abprops/media-downloads/session) → re-create those `CREATE TABLE`s with a light mapper (`kysely`/`drizzle`); the reflection ORM need not be faithfully ported — only the schemas matter (doc 32 §5).

**waweb reuse.** The entire `genericStorage.db` schema, all queries, and the data model are **JS and reused unchanged** provided the `SQLiteBridge.ExecuteSqlite` contract is reproduced; only the four native typed-store schemas must be re-implemented (doc 32 §5).

**Effort & risk.** Effort: **Medium**. Risks (doc 32 §5, doc 94): (1) the at-rest key chain is now **fully traced**, but it is blocked by **machine/user binding** (DPAPI `LOCAL=user` + `GetSystemIdForPublisher`), not by an unknown cipher → reading *existing* Windows DBs from a copied app folder is infeasible offline; **a fresh-install reimplementation is the safe path** (the exact page-cipher KDF iter/HMAC is the only un-disassembled residual, and it is moot once you reimplement); (2) **no exact DPAPI / `SystemIdForPublisher` equivalent** → re-anchoring on `safeStorage`/`machine-id` makes old encrypted DBs unreadable (acceptable for a new client) and, on headless/no-keyring hosts, `safeStorage` falls back to weaker protection (document it; gate on `isEncryptionAvailable()`); (3) the single-account wipe is destructive — replicate carefully. **Strategy-A simplification (live-data-driven, doc 94 §4):** a renderer-hosted port may **skip the native custom-codec/DPAPI chain entirely** — the WA-Web bundle persists the authoritative Signal/app-state store in the renderer's own Chromium IndexedDB (plaintext on disk, exactly like the native client's WebView). `SQLiteBridge`/`genericStorage.db` and the four native typed stores then become an *optional* mirror, not a hard gate; the real hardening task shifts to protecting the Electron `userData` dir (file ACLs + optional `safeStorage`), since anyone with read access to it already has the keys. **This milestone remains the practical gate for Strategy A only insofar as the bundle calls `SQLiteBridge` for native-context persistence** — the IndexedDB-backed Signal store works without it.

### M6 — Media pipeline: transcode + mmg up/download

**Windows mechanism.** Four native services (doc 40 §3, §5): a `MediaTranscodingService` (shared-buffer transcode: source buffer → temp file → transcode → result buffer) with a Native transcoder + a Media-Foundation `UwpTranscodeWrapper` VGA fallback; a `CodecDetector` (sample-rate gates, force-transcode HEVC/AV1/E-AC3/multi-track, AAC/MP3/Opus bounds); a `VideoFrameGrabber` (3-tier thumbnailing); `MediaDownloadManager` (via `CoreWebView2.DownloadStarting`) + `MediaFilesService` (foldering, sanitize, Mark-of-the-Web). Transforms cap the **longest edge at 960 px (480 px low-mem)**, output **H.264 + AAC in MP4 with faststart**, AAC passthrough, special GIF handling ("WA animated gif" moov tag). Hash verify is SHA-256 vs a base64 hash; downloads recorded in `mediaDownloads.db` (`CompletedDownloads2`, PK `(FileHash, Extension)`). **The media encryption + mmg HTTP protocol (the 577 `mediaKey` refs, `WhatsApp * Keys` HKDF) is all JS** (doc 40 §5).

**Recommended stack.**
- **ffmpeg** (`fluent-ffmpeg` / `@ffmpeg-installer/ffmpeg`) in the main process or a `utilityProcess`; the whole Native-vs-UWP dual path collapses to one ffmpeg invocation. Reproduce the **960 px / 480 px-low-mem** cap and `-c:v libx264 -c:a aac -movflags +faststart` (doc 40 §5).
- `CodecDetector` → `ffprobe -show_streams -print_format json`; re-implement the classification table; demux/remux via `ffmpeg -map … -c copy`; thumbnails via `ffmpeg -ss 0 -frames:v 1` (doc 40 §5).
- Download → Electron `session.on('will-download')` + `DownloadItem.setSavePath()` (direct analog of `DownloadStarting`); foldering with Node `fs`/`path`, weekly bucket `transfers/<year>-<week>`; **drop Mark-of-the-Web** (Windows-only ADS) (doc 40 §5).
- Shared-buffer handoff (`PostSharedBufferToScript`) → there is **no Electron zero-copy equivalent**; use `MessageChannelMain`/`postMessage` with transferable `ArrayBuffer`, or keep media bytes in main and pass only file paths (matches the temp-file pattern already used internally) (doc 40 §5 gaps).
- `mediaDownloads.db` → better-sqlite3 (M5), keep the `(FileHash, Extension)` PK; SHA-256 verify via Node `crypto`. Clipboard image copy → Electron `clipboard.writeImage()`. Known-folder mapping → Electron `app.getPath('pictures'|'downloads')` → XDG dirs (doc 40 §5).

**waweb reuse.** **Biggest single win in M6:** media encryption and the mmg up/download HTTP protocol run **unchanged in JS** inside the renderer — the port only replaces the four native transcode/download services (doc 40 §5).

**Effort & risk.** Effort: **Medium**. Risks (doc 40 §5 gaps): no zero-copy shared buffer (use temp files / transferable ArrayBuffers); must add `-movflags +faststart` and possibly replicate container fixups (`CheckAndRepair`); **GIF→MP4 with the WA animated-gif moov marker** must be reproduced (verify exact tag against JS/protocol); the `MediaTooLarge` size gate and codec sample-rate windows are **server-driven** — pull real limits from the bundle/ab-props rather than hardcoding. No special crypto here (it's in JS).

### M7 — VoIP calling

**Windows mechanism.** The hybrid split is **favorable** (doc 41 §5): the JS bundle owns signaling transport, the Signal `<enc>` call-key crypto, and the call UI/model; the hard native part is the **`IVoip` engine** (GUID `FF8DC465-7F0B-35AB-B334-6A3CB9A6A88F`, 74 members) doing ICE/TURN/SRTP/Opus/video, with up-calls via `IVoipCallbacks` (doc 41 §3, §4). The call cipher key arrives as a Signal `<enc v="2" type="pkmsg">` (32 bytes), decrypted by JS, then native derives media keys via `Axolotl.CallKeysFromCipherKeyV2` (doc 41 §3.6):
- **`HKDF-SHA256(IKM=cipherKey, salt=zero[32], info=UTF8(peerJid), L=46)` → split `okm[0..30]` = 30-byte SRTP block, `okm[30..46]` = 16-byte P2P key** (doc 41 §3.6, transcribed verbatim).
- **Secure SSRC:** `HKDF-SHA256(IKM=UTF8(callId), salt=LE32(ssrcTag), info=UTF8(identifier), L=4)` → read as **LE uint32** (doc 41 §3.6).
WhatsApp's media path is **not standard WebRTC SDP** — it is a custom `te2`/relay/SRTP-with-derived-keys scheme (HBH-SRTP over relay), so off-the-shelf libwebrtc/DTLS-SRTP **does not interoperate** without re-implementing the transport (doc 41 §5). The `<relay>/<te2>` content packs IP:port + flags (note a `faceb00c` marker) and is **only partially decoded** in the RE (doc 41 §5, §6).

*Engine identity re-confirmed this session (advances §6 item 4):* native strings positively identify the `IVoip` engine as **Google WebRTC + Meta `wacall` foundation over PJSIP/PJNATH** — hundreds of `webrtc::` symbols (AEC3/AGC/NetEq/`AudioDecoderOpusImpl`), PJSIP socket+STUN builders (`pj_sockaddr_parse`, `add_stun_attr_message_integrity_s`, `add_stun_attr_xor_address_s`), relay election (`elected_relay_idx`, `get_relay_idx_and_proto_from_relay_name`, `relay_measured_c2r_rtt_list`), HBH-SRTP transport suites `AES_CM_128_HMAC_SHA1_80`/`_32` (`sfu.enable_hbh_srtp`), and SFrame call-media E2EE (`wa_sframe_cipher_aes_impl.cc`). *On `te2` (resolves §6 item 3):* the captured `<relay>` (`android/decoded-offer-stanza.json`) wraps `<participant>` + 182-byte `<token id=…>` credentials; the `te2` blob byte layout is **decoded and RESOLVED** (doc 41 §3.5/§6 item 1, via the web bundle's own `te2` rewriter `research/waweb-unmin/DXbR99V6DKN.js:3063-3081`). It is a **plain length-discriminated packed `addr(4|16) ‖ port(2)` blob — 6 bytes = IPv4, 18 bytes = IPv6 — with NO trailing flag bytes and NO warp framing**. The captured `2a032880f3150136faceb00c000000b10d96` is a plain **18-byte IPv6** `te2`: `addr(16)` = `2a032880f3150136faceb00c000000b1` (a Meta `2a03:2880::/32` address) ‖ `port(2)` = `0x0D96` = 3478 (canonical STUN/relay port). The `faceb00c` substring sits at **byte offset 8, *inside* the 16-byte IPv6 address** — it is a **placeholder mask** the client overwrites with the real relay IPv6 from its datacenter table, **not** an appended cookie or a separate header field (which is why the bundle's `length===18` branch blindly replaces all 16 address bytes). **Separately**, at the STUN/P2P *packet-framing* layer (not inside the `te2` blob), `faceb00c` is also the WhatsApp **"warp" magic cookie** validated per inbound datagram (native `has_warp_header`, `enable_check_for_warp_header`, `Error: Dropping incoming P2P message (wrong magic cookie 0x%X)`); the warp cookie is a runtime packet artifact, not a `te2` field form (doc 41 §6 item 1).

**Recommended stack.**
- Host objects `VoipBridge`/`VoipSignalingBridge` → `ipcMain.handle('voip:*')` + typed `contextBridge`; mirror the exact method set so the **unmodified bundle** keeps working (doc 41 §5).
- Marshaling/JID-parse/base64-WAP decode (`VoipWebCore`/`VoipWinRTWebBridge`) → pure TS reusing the M4 `BinTreeNodeReader/Writer` + token dictionary port (doc 41 §5).
- **Media engine** → `pion/webrtc` (Go) or libwebrtc via a native addon, or `werift`/`@roamhq/wrtc` (TS). Two routes for the non-standard transport: (a) re-implement WA's media/relay framing on a UDP/ICE library (large), or (b) port/FFI the original native engine (doc 41 §5).
- **SRTP/P2P/SSRC key derivation** → `@noble/hashes` HKDF-SHA256, **exact** per the formulas above (`hkdf(sha256, cipherKey, zeros(32), utf8(jid), 46)` slice `[0..30]`/`[30..46]`; SSRC `hkdf(sha256, utf8(callId), le32(ssrcTag), utf8(identifier), 4)` → LE uint32) (doc 41 §5).
- Devices/permissions → `navigator.mediaDevices` / `getUserMedia` in the renderer (Chromium handles PipeWire/PulseAudio + V4L2); screen share → `desktopCapturer` + the `xdg-desktop-portal` PipeWire screencast (Wayland) (doc 41 §5).
- Incoming-call UX → a custom always-on-top focus-stealing `BrowserWindow` for the ring UI + in-renderer `<audio loop>` ringtone (Linux has no OS incoming-call scenario); shipped sounds `ringtone_02.m4a`, `hangup_01` (doc 41 §5, doc 42 §5).

**waweb reuse.** The **entire signaling transport, the Signal `<enc>` call crypto, device-JID/PN↔LID resolution, and the call UI/model are already in the bundle** — keep all of it; the port supplies only (1) the host-object IPC surface and (2) a media engine (doc 41 §5).

**Effort & risk.** Effort: **High** (largest single native build in the project). Risks, in order (doc 41 §5): (1) **the media engine** — WA's relay/HBH-SRTP scheme is non-standard, so stock WebRTC won't interoperate without re-implementing the transport; (2) **implementing the (already-decoded) `te2`/HBH-SRTP/SFrame framing** on your UDP/ICE library — the `te2` byte layout is RESOLVED (plain `addr(4|16) ‖ port(2)`, 18-byte IPv6 / 6-byte IPv4, no trailing flags, no warp framing; doc 41 §3.5/§6 item 1), so this is a build task, not an open decode; (3) Linux incoming-call UX. **Smallest viable first cut: audio-only 1:1.** Native dependency: the `IVoip` engine is positively identified as Google WebRTC + Meta `wacall` over PJSIP/PJNATH from native strings (interface + stack confirmed; only the per-instruction engine body is un-disassembled) (doc 41 §4) — so route (a) re-implementation is the only fully-specified path, and only the key-derivation (Axolotl) is confirmed-exact.

### M8 — Notifications, push, tray & activation

**Windows mechanism.** The **load-bearing Windows-specific design is suspend/resume-on-push** (doc 42 §5, doc 30 §5): a **WNS push channel** (`WnsToken{uri, "uwp_public"}`) lets the OS **wake a terminated app**; an out-of-proc COM background task (`WhatsApp.Root.exe /pushnotification`) and a `MayHaveMessages` throttled generic toast (24 h window) cover the suspended case; `WebLifecycleController` runs the token/ack/suspend handshake. Foreground notifications use `AppNotificationManager` toasts (avatar circle-crop, quick-reply textbox, context menu, custom per-chat tones from `.m4a/.wma`), a numeric **taskbar badge** (`BadgeUpdateManager`), and a persistent looping **`ToastScenario.IncomingCall`** toast (doc 42 §5). Activation flows through a toast-activator COM server + `whatsapp://` protocol + single-instance redirection, with `ToastLaunchArgs` `action=…&key=…` query strings (doc 42 §5, doc 30 §5, doc 01 §5).

**Recommended stack.**
- **Replace suspend/resume-on-push with a persistent-connection model.** Keep one lightweight resident main process (or systemd user service / `--hidden` autostart via `app.setLoginItemSettings({openAtLogin, args:['--hidden']})`) holding the connection; this deletes ~60% of the subsystem — `WebLifecycleController` token/ack/suspend, the BG-task COM server, and `MayHaveMessages` all become moot (doc 42 §5, doc 30 §5). *Re-verified this session (resolves §6 item 8):* the Windows suspend/memory machinery is `CoreWebView2.TrySuspendAsync()`/`Resume()` and `CoreWebView2MemoryUsageTargetLevel.Low` (`WebViewExtensions.cs:66,88,110`), orchestrated by `WebLifecycleController.cs:111-155`. These are **WebView2-only APIs with no Chromium/Electron analogue**, so the persistent-connection model is the only port answer; accept that tray-minimized CPU/RAM will differ from the Windows suspend behavior. Feed the bundle's WNS-token registration an empty/N-A token (the bundle already handles "no native push" in web mode) (doc 42 §5).
- Message toasts → Electron `Notification`; pre-crop the avatar PNG yourself (no `HintCrop`). **Quick-reply text input is not portable** — supported on macOS (`hasReply`) and some Freedesktop servers via the `inline-reply` cap, **not GNOME Shell by default**; detect the cap and fall back to "click → open chat + focus composer" (doc 42 §5).
- Custom per-chat tones → not reliably supported by Freedesktop notifications; play decoupled via a renderer `<audio>`/Web Audio element; re-encode `.wma` assets to `.ogg`/`.mp3` (doc 42 §5).
- Incoming-call toast → a **custom always-on-top focus-stealing `BrowserWindow`** with in-renderer looping ringtone (Discord/Slack pattern); no portable `IncomingCall` scenario (doc 42 §5).
- Activation → `app.requestSingleInstanceLock()` + `second-instance` (replaces `AppInstance.FindOrRegisterForKey`); `app.setAsDefaultProtocolClient('whatsapp')` + a `.desktop` `MimeType=x-scheme-handler/whatsapp`; `ToastLaunchArgs` becomes a plain JS object in the notification `click` closure (delete the query encoder) (doc 42 §5, doc 30 §5, doc 01 §5).
- Badge → `app.setBadgeCount(n)` (Unity/KDE launcher; otherwise overlay on the tray icon). Tray → Electron `Tray` + `Menu`; on Linux needs `AppIndicator`/`StatusNotifierItem` (GNOME requires an extension); intercept `'close'` to hide-to-tray (doc 42 §5, doc 30 §5, doc 01 §5). Settings gates (`WhenShowNotificationBanner`, `WhenShowTaskbarBadge`, `IsCallNotificationsMuted`, `IsCallRingtoneMuted`) are **web preferences the bundle already owns** (`"WindowsNotificationBannerSetting"`, etc.) — the port only consumes them (doc 42 §5).

**waweb reuse.** The notification **settings keys/storage and the "no native push" graceful path are already in the bundle**; the native side only consumes the gates and renders toasts (doc 42 §5).

**Effort & risk.** Effort: **Medium** (most of it deletes; the custom incoming-call window is the main build). Risks (doc 42 §5): (1) **WNS has no Linux equivalent** — the persistent-connection model is the only realistic answer, and you lose OS-driven wake-from-terminated; (2) quick-reply / per-chat tones / incoming-call scenario are **not portable** to Freedesktop notifications; (3) tray on **GNOME** needs an extension; (4) badge works only under Unity/KDE. None of this is crypto- or wire-critical, so it can lag the core path.

### 3.x Cross-cutting items

- **Protobuf authoring (spans M1–M7).** You almost never author protos in Strategy A — the bundle carries its own. The native shell owns **only** `ClientPayload` + `HandshakeMessage` (M1/M2, highest-priority artifact) and the ADV signature flow (M2); everything else (e2e `Message`, `HistorySync`, `Syncd*`, settings) is consumed in JS (doc 33 §5). Recover exact `.proto` text for Google-codec types by base64-decoding the `FileDescriptor` blobs in each `*Reflection.cs` and running `protoc --decode=google.protobuf.FileDescriptorProto`; SilentOrbit-only types (`ClientPayload`, `HistorySync`) have no descriptor — read field numbers off the tag-byte constants in the `.cs` (doc 33 §5). Use a single `protobufjs`/`ts-proto` generator (the Windows SilentOrbit+Google duplication is a build artifact) (doc 33 §5).
- **Network reachability (M1/M8).** A simple "is data connected" boolean + change event drives the 2 s-throttled reconnect; use Electron `net.isOnline()`/`powerMonitor` or NetworkManager D-Bus (doc 10 §5).
- **Dispatcher model.** The C# serial dispatchers collapse to ordinary `async`/promise queues on Node's single thread; only the VoIP native engine needs a real `Worker`/addon (doc 30 §5).

## 5. Linux/Electron Port Mapping — Consolidated Summary

### 5.1 Effort & risk matrix (Strategy A, the recommended path)

| Milestone | Native work under Strategy A | waweb reuse | Effort | Risk |
|---|---|---|---|---|
| **M0** scaffold + bridge | host window + `hostObjects` shim + ~25 bridge stubs | renderer payload as-is | Medium | **High** (sync-proxy fidelity gates everything) |
| **M1** transport + Noise | **none** (bundle's own transport) | full | None | n/a (Strategy A) |
| **M2** auth / pairing | 3 thin host objects + relaunch | pairing/ADV/ClientKey crypto | Low | Medium (platform fingerprint, key at-rest) |
| **M3** Signal e2e | ~4 curve/key host methods | full Signal engine | Low | **High** (XEdDSA exactness) |
| **M4** messaging | none (glue/config) | full send/recv/offline | Low | Low |
| **M5** storage | custom-codec/SQLCipher DB + keystore + 4 typed schemas | `genericStorage.db` schema/queries | Medium | Medium (no DPAPI/`SystemId`; gates Strategy A) |
| **M6** media | ffmpeg services + download routing | media crypto + mmg HTTP | Medium | Medium (faststart, GIF tag, shared buffer) |
| **M7** VoIP | host objects + **media engine** + key deriv | signaling + `<enc>` crypto + UI | **High** | **High** (non-standard relay/HBH-SRTP; `te2` byte layout itself is decoded) |
| **M8** notifications | persistent-conn model + toasts + tray + custom ring window | settings keys + no-push path | Medium | Medium (WNS gap, quick-reply, GNOME tray) |

### 5.2 Recommended build order

1. **M0 + M5** first — together they unlock Strategy A: once the bridge surface and the storage/keystore host objects exist, the bundle's transport/Noise/Signal/messaging (M1–M4) come for free.
2. **M2/M3 host objects** (Curve25519 verify, ClientKey, ServerEncKeySalt) — small, and required for login/pairing to complete.
3. **M6, M7, M8 in parallel** — independent native services; sequence by product priority (text+media before calls is typical). M7 is the long pole.
4. **M1 (headless Noise) only if you abandon Strategy A** for a no-WebView client — then it becomes the sequential foundation for B's M2→M3→M4.

### 5.3 The reuse thesis, stated once

Across **every** milestone the dominant finding is identical: **the cryptographically and protocol-delicate logic already lives in the reusable `waweb-source-bundle` JS** (Noise, Signal, FunXMPP, message/receipt/offline, media crypto+mmg, VoIP signaling, most protobufs, notification settings). The Windows native shell is a **thin re-host of that bundle plus native host objects**, and the Linux port's real work is to **re-create those host objects on Electron** (`contextBridge`/IPC mirroring `window.chrome.webview.hostObjects.<Name>`) plus the four native capability stacks that have no pure-JS substitute: **encrypted SQLite (M5), media transcode (M6), the VoIP engine (M7), and OS notifications/tray (M8)**. Going headless (Strategy B) discards this leverage and is a substantially larger, higher-risk build for which Baileys/whatsmeow — not this incomplete C# layer — are the right references.
