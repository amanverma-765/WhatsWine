## 1. Purpose & Scope

This document covers the **shell↔web seam** of the hybrid app: how the native WinUI 3 process hosts a single Chromium **WebView2**, navigates it to WhatsApp Web, and exposes the native C#/C++/Rust capabilities to the WhatsApp Web JS bundle running inside that WebView2.

Concretely, in scope:

1. **WebView2 environment + control creation** — `CoreWebView2Environment`, settings lockdown, permission/dialog/context-menu/new-window policy, process-failure recovery, navigation status tracking, splash/error overlays (`MainWindow`).
2. **Navigation to the bundle** — which origin is loaded (`WaWebEnvironmentProvider`), the native-context query string (`App.BuildQueryParameters`), the custom version header, document-request rewriting, and the soft/hard reload / web-content-update loop (`WebContentUpdateManager`).
3. **The two JS↔native IPC channels:**
   - **Primary: WinRT host objects** vended via `CoreWebView2.AddHostObjectToScript`, each native controller wrapped by a generated `WinRTAdapter.*Bridge` (CsWinRT projection) through a shared `DispatchAdapter`. ~27 bridges. This is the `window.chrome.webview.hostObjects.<Name>` surface the bundle calls.
   - **Legacy/fallback: comma-delimited string IPC** over `CoreWebView2.WebMessageReceived` / `PostWebMessageAsString` (`AppModel.InvokeBridgeMethodFromMessage`, `[Obsolete]`).
4. **The bridge interface model** — the generated `I<X>BridgeToNative` (JS→native), `I<X>BridgeToWeb` (native→JS callbacks via `Subscribe`), `I<X>BridgeFactory` triple, plus the projected value types (`WnsToken`, etc.) and their ABI vtable marshalling.
5. **The `WhatsApp.WebView2` assembly** — despite the name, the **localization-resources** project for the WinUI chrome, not the WebView host.

**Out of scope** (covered by sibling docs): the wire/Noise/Smax protocol, Signal crypto, SQLite storage internals, VoIP engine internals, push/toast plumbing — referenced here only where they cross the bridge.

## 3. How It Works

### 3.1 Process bootstrap → single WebView2

`Program.Main` is `[STAThread]`: it calls `XamlCheckProcessRequirements()`, `ComWrappersSupport.InitializeComWrappers()`, then enforces single-instance via `AppInstance.FindOrRegisterForKey("61E9B470-92E6-482B-AFD8-38C1FB464FB5")`; a secondary instance `RedirectActivationToAsync`’s its args to the primary **unless** launched with `/pushnotification`, and only the primary calls `Application.Start(... new App())`.
`Program.cs:19-52` — note the redirect carve-out: `if (!Enumerable.Contains(commandLineArgs, "/pushnotification"))` (`Program.cs:45`).

`App` is the `Application` subclass and `IXamlMetadataProvider`. The window + WebView are created lazily in `EnsureMainWindowInitialized()` (`App.cs:259`), which:
- `new MainWindow()` (`App.cs:266`),
- sets RTL window style from `CultureInfo.CurrentUICulture.TextInfo.IsRightToLeft` (`App.cs:268-275`),
- `await MainWindow.GetWebViewCore(shouldStartInTray)` (`App.cs:307`) — `shouldStartInTray` is true when activation kind is `StartupTask` or `/nowindow` is present (`App.cs:304`),
- registers bridges, navigates, shows/hides.

`MainWindow` is a `WinUIEx.WindowEx`; its XAML (`ms-appx:///MainWindow.xaml`, loaded via `InitializeComponent`, `MainWindow.cs:798-806`) hosts a `Grid AppHostContainer` containing a single `WebView2 WebView`, a `Grid BrowserLoadingSplashScreen`, and a `BrowserNavigationErrorScreenView BrowserNavigationErrorScreen` (`MainWindow.cs:391-400`, wired in `Connect`, `MainWindow.cs:810-833`). `ExtendsContentIntoTitleBar = true`, `MinHeight=600`, `MinWidth=800` (`MainWindow.cs:441-443`).

Event handlers wired (`:548-555`): `DOMContentLoaded` (no-op, `:715`), `NewWindowRequested`, `ProcessFailed`, `NewBrowserVersionAvailable`, `ContextMenuRequested`, `PermissionRequested`, `ScriptDialogOpening`, `NavigationStarting += ExternalLinkNavigation.OnNavigationStarting`. - **In-frame vs bounce-to-OS policy** (`ExternalLinkNavigation.OnNavigationStarting`, `WhatsApp.Tools/ExternalLinkNavigation.cs:9-22`, full body read): a top-level navigation is **cancelled and handed to the OS browser** (`args.Cancel = true; await Launcher.LaunchUriAsync(uri)`) iff *all* hold — (1) the target host differs from the current `sender.Source` host, (2) the current source is a web scheme (`uri2.IsWebSchema()`), (3) `NavigationKind == NewDocument` (top-level document load, not a sub-frame/reload), and (4) the target host is **not** the WhatsApp Web origin host (`WaWebEnvironmentProvider.GetUrl()`), with a single dev-only carve-out allowing `facebook.com` when env type is `Development`. So same-origin navigations and sub-resource/in-frame loads proceed in the WebView; any cross-origin *new-document* navigation (e.g. a clicked external link that would replace the SPA) is bounced to the default browser. (This is the in-frame allow-list the §6 list previously left unread.)

- **Permissions**: every `PermissionRequested` is auto-`Allow`ed (`OnPermissionRequested`, `MainWindow.cs:723-735`: `args.State = Allow; args.Handled = true`). (Mic/cam/notifications/clipboard all granted silently.)
- **Script dialogs**: every dialog is auto-`Accept()`ed (`OnScriptDialogOpening`, `:737-748`).
- **New windows**: any `window.open`/target=\_blank is opened in the **OS default browser** via `Launcher.LaunchUriAsync(uri)` and `args.Handled = true` (`OnNewWindowRequested`, `:674-693`) — no child WebView is ever created.
- **Persistent storage**: `ForcePersistentStoragePermission` (`WebViewExtensions.cs:242-256`, gated by `AbProps.Props.WinHybridForcePersistentStoragePermission`) issues a **CDP** call `core.CallDevToolsProtocolMethodAsync("Browser.setPermission", {permission:{name:"persistent-storage"}, setting:"granted"})`.

`GetUrl()` → `GetUrlForEnvironment(GetCurrentEnvironmentType())`. Mapping (`:78-88`):
- `Production` → **`https://web.whatsapp.com/`**
- `Development` → `https://dev-web.whatsapp.com/`

### 3.5 Navigation + native-context query string (`MainWindow.Start`, `App.BuildQueryParameters`)

`App` builds the query dict the bundle reads to detect it is inside the native shell (`BuildQueryParameters`, `App.cs:378-403`):
```
windows        = "1"
windowsBuild   = PackageInfo.Version.ToStringHybrid()
windowsBuildType = BuildDetails.GetBuildOrChannel()
bridgeError    = "1"
launchContext  = <_appLaunchMetrics.LaunchContext ?? "unknown">
osBuild        = "<OsVersion.Build>"
windows_offline = "1"   // only if Webview2EnableOfflineSupport && !IsDataConnected
```
This dict is then used three ways:
1. `_updateManager = new WebContentUpdateManager(webView, dict)` (`App.cs:314`).
2. `webView.AlwaysUseArgumentsForDocumentRequested(dict)` (`App.cs:325`) — re-applies the params on every top-level document request/navigation whose path is `/` (`WebViewExtensions.cs:314-370`).
3. `MainWindow.Start(dict)` (`App.cs:326`).

`MainWindow.Start` (`MainWindow.cs:516-532`):
- if env type is `OnDemand` → `OpenDevToolsWindow()`,
- `coreWebView.AddHeaderForEachRequest("X-WA-WebView2-Version", PackageInfo.Version.ToStringHybrid())` — sets a per-request header via a `WebResourceRequested` handler that calls `headers.SetHeader(...)` (`WebViewExtensions.cs:183-240`),
- `EnableWebResourceRequestedEvents()` (filter `"*"`, `CoreWebView2WebResourceContext.All`, `WebViewExtensions.cs:275-286`),
- hooks the sharesheet `WebResourceRequested`,
- optionally creates a `CacheManager` for offline support,
- `coreWebView.Navigate(url, args)` — the extension `Navigate(uri, dict)` merges the dict into the query string with `HttpUtility.ParseQueryString`/`UriBuilder` (`WebViewExtensions.cs:288-312`).

So the loaded URL is effectively `https://web.whatsapp.com/?windows=1&windowsBuild=...&bridgeError=1&launchContext=...&osBuild=...`.

### 3.6 Navigation status state machine (splash + error overlay)

`WebView2NavigationStatusMonitor` tracks a `BehaviorSubject<NavigationStatus>` over `{Success, Failed, InProgress, NoConnection}` (`WebView2NavigationStatusMonitor.cs:11-27`). It keeps a `Dictionary<ulong, NavigationStartingEventArgs>` of in-flight navigations and a `_lastNavigationId`; on `NavigationCompleted` for the latest id it maps `WebErrorStatus` → `NoConnection` for `CannotConnect/Disconnected/ConnectionAborted/ConnectionReset/HostNameNotResolved`, else `Failed`, else `Success` (`:85-93`). `OperationCanceled` completions are ignored (`:68`).

`AdvancedWebView2NavigationStatusMonitor` decorates it (`AdvancedWebView2NavigationStatusMonitor.cs`): it adds an exponential-ish delay (`2 * _navigationFailedCounter++` seconds) before surfacing an `InProgress→Failed` transition (`:29-32`), and flips status to `Failed` if a navigation stays `InProgress` for **90 s** (`Task.Delay(90s)` guarded by a `_revision` token, `:42-49`).

`MainWindow.OnNavigationStatusChanged` (`MainWindow.cs:695-713`) drives the UI:
- `InProgress` → show `BrowserLoadingSplashScreen`.
- `Failed`/`NoConnection` → show `BrowserNavigationErrorScreen` with `ErrorType.UnexpectedNavigationError` or `ErrorType.NoConnection`.
- The error screen’s buttons map to: Close→`CloseApp`, **Reconnect→`WebView.CoreWebView2.Stop(); WebView.Reload()`** (`:755-759`), Restart→`RestartApp(WebView2Error,...)` (`:761-764`).

### 3.7 Context menu reskin (`WebView2ContextMenuHelper`)

Default menus are kept enabled but intercepted in `ContextMenuRequested`. `RemoveUnsupportedContextMenuOperations` keeps only an allow-list (`cut/copy/copyImage/paste/pasteAndMatchStyle/spellcheck/selectAll/redo/undo`) and strips everything else — `inspectElement, webCapture, share, print, saveAs, reload, forward, back, emoji, copyImageLocation, saveImageAs, copyLinkToHighlight` are all `false`, and the writing-direction command id `41120` is dropped (`WebView2ContextMenuHelper.cs:21-44, 186-219`). If a `spellcheck` item is present the helper bails and lets WebView2 draw its own menu; otherwise it builds a WinUI `MenuFlyout` (Material3-styled, RTL-aware) mirroring the items, mapping `ShortcutKeyDescription` strings into `KeyboardAccelerator`s, and sets `args.SelectedCommandId` on click (`:46-115`).

### 3.8 Process-failure recovery (`OnCoreWebView2ProcessFailed`, `MainWindow.cs:585-621`)

- `BrowserProcessExited` → ping the web client; if dead, `_restartToTrayOnClose = true` and (if not visible) `RestartApp(WebView2Error, restartToTray:true)`, and show the fatal-error overlay that polls `PingWebView()` every 5 s until it recovers (`:636-652`).
- `RenderProcessExited` / `RenderProcessUnresponsive` / `FrameRenderProcessExited` → `WebView.Reload()`.
`_restartToTrayOnClose` is also consulted in the window `Closed` handler to trigger a tray-restart instead of exiting (`MainWindow.cs:452-459`). `NewBrowserVersionAvailable` prompts an update dialog → `RestartApp(WebView2Update)` (`:654-667`).

### 3.9 The primary IPC channel: WinRT host objects

**Registration glue** (`WinRTAdapterExtensions.cs`):
```csharp
public static void AddWinRTBridge(this CoreWebView2 webView2, string name, object bridge, DispatchAdapter da)
    => webView2.AddHostObjectToScript(name, da.WrapObject(bridge));          // shared adapter
public static void AddWinRTBridge(this CoreWebView2 webView2, string name, object bridge)
    => webView2.AddHostObjectToScript(name, new DispatchAdapter().WrapObject(bridge));  // own adapter
```
`WrapObject(bridge)` ≡ `dispatchAdapter.WrapObject(bridge, dispatchAdapter)` (`WinRTAdapterExtensions.cs:8-11`). `DispatchAdapter.WrapObject` is itself a projected native call into `WinRTAdapter.dll` (`DispatchAdapter.cs:146-149`, RCW over `ICoreWebView2DispatchAdapter`), i.e. the wrapping/marshalling policy lives in native `WinRTAdapter.dll`, not C#. **The native side is confirmed by `strings` on `x64/WinRTAdapter.dll`**: the C++/WinRT `winrt::WinRTAdapter::implementation::DispatchAdapter` type implements `Microsoft::Web::WebView2::Core::ICoreWebView2DispatchAdapter` (the `WrapObject` ABI vtable); the per-call async result is `wv2winrt_impl::DispatchAsyncResult` implementing `ICoreWebView2PrivateDispatchAsyncResult`/`…2`/`…FinishedHandler`/`…Info`, and the host-object member surface is the private `wv2winrt_impl::DispatchBase` over `ICoreWebView2PrivateDispatchContainer` (1–4). Threading uses `CoCreateFreeThreadedMarshaler`. So the member-resolution / sync-vs-async (`forceAsyncMethodMatches`) decision and the actual call marshalling are Microsoft's private **wv2winrt** dispatch engine embedded in `WinRTAdapter.dll` — provenance confirmed, but the byte-level logic is not decompiled here.

**Critical proxy config (set once by the bundle, `U2j2EhR17gV.js`):**
```js
e.hostObjects.options.defaultSyncProxy = !0;
e.hostObjects.options.forceAsyncMethodMatches = [/Async$/, /AsyncWithSpeller$/];
e.hostObjects.options.ignoreMemberNotFoundError = !0;
```
Implications for a port: **calls are synchronous by default** (`defaultSyncProxy=true`), so JS does `hostObjects.sync.<Bridge>.<Method>(...)` and gets a value back inline; only methods whose names end in `Async`/`AsyncWithSpeller` return promises; and unknown-member access must **not throw** (`ignoreMemberNotFoundError=true`). This is the WebView2 host-object proxy contract WhatsApp Web relies on — a Node/Electron port must replicate these exact semantics.

> **Where this proxy actually lives (confirmed this session, [native-binary]).** These three `options.*` literals are **not present in any binary in the package** — 0 hits in `x64/WinRTAdapter.dll` and 0 in `x64/Microsoft.Web.WebView2.Core*.dll`; across all of `decompiled_source/` they appear **only** in the JS bundles. ### 3.10 The legacy/fallback IPC channel (comma-delimited string IPC)

Registered in `AppModel.SetWebView` via `_webView.WebMessageReceived += OnWebMessageReceived` (`AppModel.cs:212`). `OnWebMessageReceived` → `InvokeBridgeMethodFromMessage(text)` (`AppModel.cs:403-415`), which is `[Obsolete("Use WinRT bridge instead")]` (`AppModel.cs:249`).

### 3.15 AbProps — the app-wide feature-flag / A-B gate and its exposure round-trip

This is the runtime mechanism the rest of these docs keep referencing as `AbProps.Props.X` (the WebView2-env GPU/background/memory flags in §3.2–3.3 are themselves `AbProps.Props.*` reads). **Doc 32 covers only the on-disk `abprops.db` schema/migration; this section covers how a flag is resolved in memory and how *reading* one is logged back to the JS A/B framework.** All flag *values* arrive from the JS side over `AbPropsBridge`; the native side is a typed cache plus an exposure ledger.

**The static accessor (`WhatsApp.AbProps/WhatsApp/AbProps.cs`).** `AbProps` is a process-global holding a single immutable `AbPropsValues` snapshot inside a Reactive `BehaviorSubject<AbPropsValues>` seeded with a default `new AbPropsValues()` (`AbProps.cs:9-11`). - **Config-code decode → typed fields.** The live ctor `AbPropsValues(string version, IReadOnlyDictionary<int,(string?,string?)> configs, IObserver<string>? exposureObserver = null)` (`:3727-3742`) stores `version`, the observer, and the `configs` map (keyed by int config-code, value = `(rawValue, expoKey)`), then iterates every config calling `TrySet((AbPropertyCode)code, rawValue)`; each failure bumps a private `_errors` counter (`:3737-3740`, surfaced via `GetErrorsCount()` `:3749`). `TrySet` is the 229-arm decoder (`:3767+`): bool flags parse with `XmlConvert.ToBoolean` swallowing `Argument/Format` exceptions, int flags with `int.TryParse`, string/json kept raw (e.g. `:3771-3876`). - **Exposure-on-read.** Every public getter calls `OnExposure(code)` **before** returning its field, e.g. `AiGroupCallMaxVersionByCountry.get { OnExposure(ai_group_call_max_version_by_country); return _field; }` (`:824-835`; **229 such getters**). `OnExposure` (`:3759-3765`) is the gate: *iff* an observer is attached **and** this code is present in `_configs` **and** its `expoKey` (`value.Item2`) is non-empty, it fires `_exposureObserver.OnNext(expoKey)`. ### 3.16 Native URL / link-preview fetcher (`LinksPreviewBridge` → `LinkPreviewProvider`)

This is **not a passthrough** — it is a real native HTTP client + HTML scraper + image decoder. When the JS client needs an inline link preview (the unfurl card under a pasted URL), it calls the single async bridge method and native does the whole fetch/parse/thumbnail round-trip off-process.

**Bridge surface.** The host object is registered post-login as `LinksPreviewBridge` wrapping a fresh `LinkPreviewProvider`: `webView.AddWinRTBridge("LinksPreviewBridge", new LinksPreviewBridge(new LinkPreviewProvider()), dispatchAdapter)` (`AppModel.cs:235`, shared `DispatchAdapter`). The contract is a single method:
```csharp
// WinRTAdapter/ILinksPreviewBridgeToNative.cs:11-14  (GUID 5E1B10BC-…)
public interface ILinksPreviewBridgeToNative {
    IAsyncOperation<string> GetPreviewAsync(string link);
}
```
The `Async` suffix means JS gets a **promise** back (matches `forceAsyncMethodMatches = [/Async$/]` from §3.9). The RCW shim `WinRTAdapter/LinksPreviewBridge.cs` is the usual marshalling stub (`GetPreviewAsync` forwards to `ILinksPreviewBridgeToNativeMethods.GetPreviewAsync`, `:190-193`; `ActivationFactory.Get("WinRTAdapter.LinksPreviewBridge")`, `:82`). t.getPreviewAsync = asyncToGenerator(function*(e){
  var t = yield this.$1.getPreviewAsync(e);   // e = link string
  return t == null || t === "" ? null : JSON.parse(t)

So the native return contract is consumed precisely: `""`/null (hard-error or empty) → JS `null`; otherwise `JSON.parse` of the `{title, description, thumbnail}` object (the base64-encoded `byte[]` thumbnail survives as a JS string in the parsed JSON). This matches the native `Schema` shapes in the pipeline below.

**The fetch/parse/thumbnail pipeline (`WhatsApp.Bridge/LinkPreviewProvider.cs`).** `GetPreviewAsync(string link)` (`:30-59`):
1. Creates a `CancellationTokenSource` with **`CancelAfter(5000)`** — a hard **5 s** budget covering *both* the HTML fetch and the thumbnail fetch (`:35-36`).
2. `await LinkPreviewUtils.FetchWebPageMetadata(link, string.Empty, cancel.Token)` (`:37`) — HTTP-GETs the page and scrapes its `<head>`.
3. **Parse-failure fallback:** if metadata is `null`, returns `JsonSerializer.Serialize(new Schema { Title = new Uri(link).Host, Description = link })` — i.e. `{title: <host>, description: <link>}` with **no thumbnail** (`:38-45`).
4. On success, `await metadata.LoadThumbnailAsync(cancel.Token)` (`:46`) — second HTTP GET + decode, then returns `{title, description, thumbnail}` where **`thumbnail` is a raw `byte[]`** (`:47-52`).
5. Any exception → `FailuresService.Investigate(...)` and returns **`string.Empty`** (`:54-58`).

The returned JSON shape is the private `Schema` class (`:18-28`): `{"title": string?, "description": string?, "thumbnail": byte[]?}` (`System.Text.Json` serializes `byte[]` as base64). So JS receives `{title, description, thumbnail}` (full success), `{title:host, description:link}` (parse fail), or `""` (hard error).

**HTML scrape (`WhatsApp.Tools/LinkPreviewUtils.cs`).** The async path used by the bridge is `FetchWebPageMetadata(uri, hostName, token)` (`:73-93`) → `TryReadHeadContent` → `ParseHead`:
- **`TryReadHeadContent(url, hostName, token)`** (`:209-242`): `HttpUtils.Get(url, hostName, GetUrlPreviewUserAgent(), HttpUtils.Options.None)` (`:214`), copies the stream into a `MemoryStream` (1 KiB chunks, honoring `token`), reads it all, then extracts only the `<head>…</head>` via the precompiled PCRE `headRegex = "<head[^>]*>([\s\S]*)</head>"` (`:37,220-224`). ### 3.17 Offline web-resource cache (`CacheManager`)

A native HTTP cache layered *on the WebView2 itself* so the WhatsApp Web SPA can load while offline. It is **not** a generic browser cache — it caches only the SPA's own HTML documents and serves them back **only when the device is offline**.

**Creation (gated, optional).** Built during `MainWindow.Start` only if the AbProp is on and it doesn't already exist: `if (AbProps.Props.Webview2EnableOfflineSupport && _cacheManager == null) _cacheManager = CacheManager.Create(coreWebView)` (`MainWindow.cs:527-530`; field `_cacheManager` at `:388`). The ctor (`CacheManager.cs:28-34`) sets the cache root to **`ApplicationData.Current.LocalFolder.Path\WebView2Cache`** and subscribes to two `CoreWebView2` events: `WebResourceRequested` (serve path) and `WebResourceResponseReceived` (persist path). This pairs with the `windows_offline=1` query param the shell adds when `Webview2EnableOfflineSupport && !IsDataConnected` (`App.BuildQueryParameters`, §3.5; const `WindowsOffline = "windows_offline"`, `CacheManager.cs:20`; 1 hit in the bundle).

**Persist path — `OnWebResourceResponseReceived`** (`:113-137`). On every response, **only when online** (`NetworkStateMonitor.Instance.State.IsDataConnected`) and `StatusCode == 200`, it reads the `Content-Type` header (default `"unknown"`) and, if `ShouldCache(uri, contentType)`, fires `CacheResource(...)`. `ShouldCache(uri, contentType)` (`:139-144`) returns true iff the URI **starts with the WhatsApp Web origin** (`WaWebEnvironmentProvider.Instance.GetUrl()`) **and** the content is `text/html` (when contentType is null it defaults to cacheable). So **only same-origin HTML documents are cached** — not JS/CSS/media.
- **`CacheResource`** (`:146-171`): `cacheKey = GetCacheKey(uri)`, pulls the body via `e.Response.GetContentAsync()`, ensures the folder, writes the bytes to `WebView2Cache\<cacheKey>`, then writes a sidecar `WebView2Cache\<cacheKey>.meta` = `JsonSerializer.Serialize(new CacheMetadata(uri, contentType, DateTime.UtcNow, DateTime.UtcNow.AddHours(168.0)))`.
- **Cache key** = lowercase hex SHA-256 of the URI **with the `windows_offline` query param stripped** (`GetCacheKey`, `:173-178`; `RemoveQueryParameter`, `:82-103`). Stripping `windows_offline` is essential so the online URL (no flag) and the offline URL (`windows_offline=1`) hash to the **same** key — i.e. what was saved online is found offline.
- **TTL = 168 h (7 days)** for HTML (`HtmlTtlHours = 168.0`, `:22`), encoded as `ExpiresTimestamp` in the sidecar.

**Sidecar schema (`CacheMetadata.cs`).** Primary-ctor record: `{ string Uri, string ContentType, DateTime CachedTimestamp, DateTime ExpiresTimestamp }` (`CacheMetadata.cs:5-14`).

**Serve path — `OnWebResourceRequested`** (`:41-80`). **Network-gated:** if `NetworkStateMonitor.Instance.State.IsDataConnected` **or** `!ShouldCache(uri, null)`, it returns immediately and lets the live request proceed (`:46-51`). Only when **offline** and the URI is a same-origin HTML doc does it intercept: takes a `Deferral`, computes the cache key, and looks for `<cacheKey>` + `<cacheKey>.meta`. If either is missing it bails (live request, which will fail offline). It deserializes the `.meta`, and **iff `metadata.ExpiresTimestamp >= DateTime.UtcNow`** (not expired) it synthesizes a response: `e.Response = _core.Environment.CreateWebResourceResponse(content, 200, "OK", GetCorsHeaders(metadata.ContentType))` where the content is the cached file opened read-only (`:65-69`). CORS headers = `"Content-Type: <ct>\nAccess-Control-Allow-Origin: *\n"` (`GetCorsHeaders`, `:180-183`). The `Deferral` is always completed in `finally` (`:76-79`).

> **Net behaviour:** while online, every 200-OK same-origin HTML doc is mirrored to `LocalFolder\WebView2Cache\<sha256(url−windows_offline)>` with a 7-day TTL sidecar; while offline, requests for those same docs are answered from disk with a synthetic `200 OK` (CORS-open) until the sidecar expires. JS/CSS/media are **not** cached here (they rely on the WebView2's own HTTP cache in `LocalCacheFolder`, §3.2). This is why the shell flips `windows_offline=1` and re-navigates: it lets the cached shell boot, and the bundle can detect offline mode from the query param.
>
> **Live-disk reality [live-appdata] (docs 94/95):** the captured AppData tree of the real logged-in install has **no `WebView2Cache` folder under `LocalState`** and **no Chromium HTTP `Cache`/`Cache_Data` dir under `EBWebView/Default/`** — i.e. on this install the `CacheManager` HTML-mirror was never populated, consistent with the offline path being gated by `Webview2EnableOfflineSupport` (likely off/unexercised in this build). The offline-shell mechanism is fully described in code but **empirically dormant** here; whether a populated cold cache actually renders offline remains a runtime question (§6 Q10).

> **Port note (Electron):** implement in `session.webRequest` / a `protocol`-interceptor or `session.protocol.handle`. On `onCompleted`/response for `startsWith(webOrigin) && content-type text/html && online`, persist body + `{uri,contentType,cachedTs,expiresTs=+168h}` keyed by `sha256(stripQuery(url,'windows_offline'))` under `userData/WebView2Cache`. On request, when `net.isOnline()===false` and the key exists and not expired, respond `200` from disk with `Access-Control-Allow-Origin: *`. Electron's own HTTP cache covers the JS/CSS/media tier, matching the WebView2 split.

### 3.19 WAM telemetry / analytics pipeline (Fieldstats) — `WamBridge` → `WamController`

This is the **metrics half** of the JS↔native contract — the inverse direction of most bridges. Here **native is the *aggregator/transport* and JS is the *consumer/uploader*.** Native (and native subsystems: VoIP, crypto, media, crash) emit structured WAM ("WhatsApp Analytics Metrics", a.k.a. **Fieldstats**) events into a buffer; `WamController` periodically drains the buffer, deserializes it, re-serializes to JSON, and pushes the events **up to the JS layer** via `WamBridge.NotifyEventsSaved(json)`. The JS WhatsApp-Web client owns the actual WAM upload to Meta's servers (the native side never talks to the WAM endpoint). Doc 31 previously only name-dropped `WamBridge` in the bridge-enumeration lists; this section documents the whole subsystem.

**The event model (`WhatsApp.Core/WhatsApp/Wam.cs`, 82,466 lines, machine-generated).**
- **`WamEvent` abstract base** (`WhatsApp.Core/WhatsApp/WamEvent.cs:3-8`): just two abstract members — `uint GetCode()` (the event's numeric WAM id) and `void SerializeFields(Wam.IWriter fields)`.
- **The concrete events** live as nested classes under `Wam.Event` (`Wam.cs:28445` `public static class Event`). Each is a `class <Name> : WamEvent` with nullable auto-properties, a `Reset()`, a `GetCode()` returning the WAM event id, and a `SerializeFields` that writes each property under a **per-event field code**. Confirmed example `Wam.Event.AddressbookSync` (`Wam.cs:28631-28661`): props `AddressbookSyncErrorCode`/`AddressbookSyncIsReg`/`AddressbookSyncResultType`/`AddressbookSyncT`, `GetCode() => 480u`, and `SerializeFields` does `fields.Change(2, AddressbookSyncErrorCode); fields.Change(1, AddressbookSyncIsReg); fields.Change(3, EnumToLong(AddressbookSyncResultType)); fields.Change(4, AddressbookSyncT);` — i.e. **field code → value**, enums coerced via `EnumToLong`. Another: `Wam.Event.AddressingModeMismatch` `GetCode() => 4750u` (`:28684-28696`). The task brief's "~1241 classes" is the upper bound across builds; in *this* dump `grep -c "class \w+ : WamEvent"` = **894** concrete `Wam.Event.*` subclasses (`Wam.cs`).
- **`Wam.IWriter`** (`Wam.cs:82192-82201`) is the serialization sink each event writes into: `ChangeValueToNull(int code)`, `ChangeValue(int code, double)`, `ChangeValue(int code, long)`, `ChangeValue(int code, string)`. The strongly-typed `Change(this IWriter, int code, T? value)` extension overloads (`Wam.cs:82405-82464`) adapt nullable `bool`/`long`/`double`/`string`/`enum`: **`bool?`→`long` 1/0** (`:82405-82415`), null→`ChangeValueToNull`, and `EnumToLong<T>` casts an enum to its underlying `(int)` (`:82459-82464`). There are also ~hundreds of named `Set*` helpers on common attributes (e.g. `SetAbKey` ⇒ code 18, `SetAppBuild` ⇒ code 14, `Wam.cs:82205-82217`).

**Emission API + privacy/sampling gate (`Wam.cs` + `WhatsApp.Core/WhatsApp/WamStorage.cs`).** A subsystem fires an event via the extension `WamEvent.SaveEvent(samplingInterval=1, version=null)` (fire-and-forget `async void`, `Wam.cs:82395-82398`) or its awaitable twin `SaveEventAsync` (`:82400-82403`); both delegate to the process-global `Wam.Storage` (`= new WamStorage()`, `Wam.cs:82203`) `.Save(ev, samplingInterval, version)`. `WamStorage.Save` (`WamStorage.cs:54-75`) is the **filter**:
1. **Privacy gate** — `IsPrivacyCheckedEvent(ev)` (`:77-89`) returns true only if the event's runtime `Type` is in a hardcoded allow-list `CreateCheckedSet()` of ~130 types (`:91-183`, e.g. `AppLaunch`, `Call`, `MediaUpload2`, `MessageSend`, `E2EMessageRecv`, `CrashLog`, the `MDAppState*`/`Adv*` families, `Voip*`). Events **not** in the set are dropped (and recorded once in `_loggedEvents`). So only privacy-reviewed events are ever buffered.
2. **Sampling gate** — `samplingInterval >= 0 && (samplingInterval <= 1 || _random.Next(samplingInterval) == 0)` (`:61`): `1` = always keep; `N>1` = keep with probability `1/N`; `<0` = drop.
3. On pass, it hops to a serial `ConcurrentQueueDispatcher { Id = Log.DispatcherId.FieldStats }` (`:39-42`), builds a `Dictionary<object,object?>{ "id"=GetCode(), "weight"=samplingInterval }`, runs `ev.SerializeFields(new Writer(dictionary))` (a `Wam.IWriter` that writes straight into the dict — `Writer`, `:12-33`), and `await _buffer.Writer.WriteAsync(dictionary)` into an **unbounded `Channel<Dictionary<object,object?>>`** (`:35`). `GetReader()` (`:48-51`) exposes the `ChannelReader`.

**The custom binary wire format (`WhatsApp.Core/WhatsApp/WamEventsSerializer.cs`).** This is a second, *binary* `Wam.IWriter` (it's also what the **native** global buffer uses). `CurrentSchemaVersion = 1` (`:22`). It writes via a `BinaryWriter` over a `MemoryStream`, **`LineType`-tagged** records (enum `:10-20`): `EventId=0`, `EventWeight=1`, `IntField=256`, `LongField=512`, `DoubleField=768`, `StringField=1024`, `NullField=1280`, `EventStopFlag=65536`. `Write(WamEvent, version, weight)` (`:170-175`) frames one event: `Start` writes `EventId, code(u32), version=1, EventWeight, weight` and seeds `_fieldsCount=2` (`:177-185`); each `ChangeValue(code, …)` writes its tag + code + payload (strings via `WriteNormalizedString`, `:157-163`); `Stop` writes `EventStopFlag, _fieldsCount` (`:187-191`). The matching **`Deserialize(byte[])`** (`:30-127`, an `IEnumerable<Dictionary<object,object?>>` state machine) reads it back: rejects `version > 1`, validates the trailing field-count against `EventStopFlag`, and yields one dict per event keyed by `"id"`/`"weight"`/field-code; **NaN doubles and null values delete the key** (`AddDouble`/`AddValue`, `:103-126`). This deserializer is what `WamController` runs over the *native* buffer.

**The native global buffer (`WhatsAppNativeProjection/WhatsAppNative/WamGlobalBuffer.cs` + `IWamGlobalCallbacks.cs`) + the `WamWriter` record-builder.** Native C++/Rust subsystems write their own WAM events into a **`WhatsAppNative.dll` global buffer** (projected as the static class `WamGlobalBuffer`). Its statics: `byte[] Rotate()` (swap-and-return the current serialized buffer, `:94-97`), `SetCallbacks(IWamGlobalCallbacks)` (`:104-107`), and device-info setters `SetMemClass(long)`/`SetNumCpu(long)`/`SetOsVersion(string)` (`:109-122`). The per-record writer is a separate projected WinRT class **`WamWriter`** (`WhatsAppNativeProjection/WhatsAppNative/WamWriter.cs`; native `WamWriter@WhatsAppNative` + `__WamWriterActivationFactory`, confirmed by `strings x64/WhatsAppNative.dll`): its API (`__IWamWriterPublicNonVirtuals.cs:13-29`) is `Start(uint eventCode, int weight)` → `ChangeValue(int code, long|double|string)` / `ChangeValueToNull(int code)` → `GetBytes()` → `Stop()` — a native subsystem opens a record with event-id+weight, sets type-tagged field codes, and the finished record accrues in `WamGlobalBuffer`. (This is the native mirror of the C#-side `Wam.cs` `SerializeFields`/`GetCode` shape and of Baileys' `[code, type]` field maps — see §6 Q14.) `IWamGlobalCallbacks` (`IWamGlobalCallbacks.cs:10-13`, GUID `D7E4A482-…`) has a single `OnRotateRequested(bool isSending)` — native tells C# "I want my buffer drained now". The `Rotate()` payload is the `WamEventsSerializer` binary format above, hence `WamController` deserializes it with `WamEventsSerializer.Deserialize`.

### 3.20 Trusted-Contact token subsystem (TcToken) — `TcTokenBridge` → `TcTokenController` / `TrustedContactManager`

WhatsApp's **privacy/trusted-contact token** ("TC token", a.k.a. privacy token) is a small opaque blob (1–120 bytes) that the native VoIP/presence layer must attach to certain outgoing stanzas to prove a prior relationship with the peer. **JS owns the tokens** (they ride in over the encrypted protocol and are surfaced by the web client); the **native side is a cache + on-demand fetcher** that pulls a token from JS when it needs one and JS doesn't already have it cached natively. Doc 31 previously listed `TcTokenBridge` only with its purpose "unconfirmed"; this section documents the retrieval/caching/refresh mechanism and the presence-stanza wire usage.

**The cache (`WhatsApp.VoIP/WhatsApp/TrustedContactManager.cs`).** A process-global singleton (`Instance`, `:10`) holding `ConcurrentDictionary<ConvoJid, byte[]> _tokens` (`:8`). API: `SetTcToken(jid, byte[])` (`:12-15`) and `GetTcToken(ConvoJid)` → the cached bytes or **`Array.Empty<byte>()`** on miss (`:17-24`). No persistence — purely in-memory for the session.

**The controller / bridge adaptee (`WhatsApp.Root/WhatsApp.Bridge/TcTokenController.cs`).** `TcTokenController(TrustedContactManager)` implements `ITcTokenBridgeToNative` (`:10`); it owns the `_web` callback and a `Subject<string> _tokensUpdatedSubject` used to await JS replies (`:12-16`). Created in `AppModel` (`_tcTokenController = new TcTokenController(TrustedContactManager.Instance)`, `AppModel.cs:143`) and also exposed as a static `Instance` (`:16`).
- **`GetTcToken(ConvoJid)`** (`:18-21`): synchronous cache read (delegates to the manager).
- **`GetTcTokenAsync(ConvoJid)`** (`:23-41`) is the core retrieval mechanism: return the cached token if non-empty; **otherwise** subscribe to `_tokensUpdatedSubject.Where(key == convoJid.ToVoipString())`, fire **`_web?.RequestUpdate(convoJid.ToVoipString())`** to ask JS for it, and `await source.Timeout(5s).FirstOrDefaultAsync()` — a **5-second** budget. On `TimeoutException` it logs `FailuresService.Count("Failed to query trusted contact token within timeout", …)` (`:36-39`) and returns whatever the cache now holds (or empty).
- **`RequestTcToken(UserJid)`** (`:43-46`): fire-and-forget `_web?.RequestUpdate(jid.ToVoipString())` (prefetch / refresh, no await).
- **`Subscribe(ITcTokenBridgeToWeb web)`** (`:48-51`): JS hands its callback object; stored as `_web`.
- **`UpdateTcTokens(string[] jids, string[] tcTokens)`** (JS→native reply, `:53-68`): zips the two arrays, `ConvoJid jid = JidFactory.CreateNewConvoJid(jidString)`, `byte[] token = Convert.FromBase64String(s)` (**tokens cross the bridge base64-encoded**), `SetTcToken(jid, token)` into the cache, then the `GetTcTokenAsync` waiter resolves via the `_tokensUpdatedSubject`. Per-entry parse failures are counted, not fatal.

**The bridge triple (`WhatsApp.Root/WinRTAdapter/ITcTokenBridge*.cs`).**
- `ITcTokenBridgeToNative` (JS→native, GUID `649C16AA-…`, `:10-15`): `UpdateTcTokens(string[] jids, string[] tcTokens)`, `Subscribe(ITcTokenBridgeToWeb web)`.
- `ITcTokenBridgeToWeb` (native→JS, GUID `FD3BBE46-…`, `:10-13`): single callback `RequestUpdate(string ids)` — native asking JS "give me the TC token(s) for these jids".
- Plus internal `ITcTokenBridge`/`ITcTokenBridgeFactory`; RCW `WinRTAdapter/TcTokenBridge.cs` is the usual shim. **Registration:** `webView.AddWinRTBridge("TcTokenBridge", new TcTokenBridge(_tcTokenController), dispatchAdapter)` post-login on the shared adapter (`AppModel.cs:232`).

**Wire usage #1 — VoIP (`VoipBridgeJid.tcToken`).** When native builds a `VoipBridgeJid` for a user, it stamps the cached TC token onto it: `VoipBridgeJidFactory.ToVoipBridgeJid(this UserJid)` (`WhatsApp.VoIP/WhatsAppCommon/VoipBridgeJidFactory.cs:53-67`) does `byte[] tcToken = TrustedContactManager.Instance.GetTcToken(jid)` and sets `tcToken = tcToken` on the returned struct (alongside `str`/`type`/`primaryIdentifier`). So every user-JID handed to the native VoIP engine carries its TC token for the call-signaling layer.

**Wire usage #2 — Presence subscribe stanza (`tctoken` node).** The Smax presence builders attach the token to outgoing `<presence type=subscribe>` stanzas:
- `Presence.Outgoing.PrivacyTokenContentsMixin.Create(byte[] elementValue)` (`WhatsApp.Networking/WhatsApp.Smax.Generated.Presence.Outgoing/PrivacyTokenContentsMixin.cs:10-22`) validates the blob is **1–120 bytes** (`ValidateBinary(elementValue, 1, 120)`) and emits it as the **raw data** of a `smax:any` node.
- `TCTokenMixinTctoken.Create(long? tAttr, PrivacyTokenContentsMixin)` (`TCTokenMixinTctoken.cs:10-26`) wraps it in a **`<tctoken t=…>`** node (optional `t` timestamp attr, range-validated, then `Merge`s the privacy-token data).
- `TCTokenMixin.Create(tctokenChild)` (`TCTokenMixin.cs:10-21`) hosts it under a `smax:any`, and `SubscribeRequest.Create(toAttr, nameAttr, contextAttr, TCTokenMixin?)` merges the optional `TCTokenMixin.Node` into the presence-subscribe stanza (`SubscribeRequest.cs:11,27,31-33`). So the token rides out as `<presence type=subscribe to=…><tctoken t=…><…raw token bytes…></tctoken></presence>`.

> **Confirmed against the bundle — the native `TcTokenBridge` is NOT consumed by this bundle.** The complete list of host objects the bundle grabs (`grep -oh 'hostObjects\.[A-Za-z]*' U2j2EhR17gV.js | sort -u`) contains **every** registered bridge **except** `TcTokenBridge` — and there is **no** `WAWebWindowsHybridBridgeTcToken` wrapper module (the per-bridge wrappers are `…Contacts/…Voip/…Wam/…AbProps/…LinksPreview/…Ringtone/…TouchpadFix`, etc., but no `…TcToken`). So although C# registers `TcTokenBridge` (`AppModel.cs:232`) with a full interface triple, **this bundle version never calls `Subscribe`/`UpdateTcTokens` on it**; the native cache/round-trip described above is dormant for this build. The TC-token machinery the bundle *does* ship is its own protocol-level code: `TcToken` appears **104×**, `TrustedContact` **32×**, with `WAWebTrustedContactsUtils`, orphan-token tables (`getOrphanTcTokenTable`, `pruneExpiredOrphanTcTokens`), `sendTcTokenWhenDeviceIdentityChange`, `handleIncomingTcToken`, etc. The SMAX `tctoken` node is built **JS-side**: `o("WASmaxJsx").smax("tctoken",{t:…OPTIONAL…},…PrivacyTokenContentsMixin(…))` and a status provider `t.getTcToken(); return e ? wap("tctoken", null, e) : null` (21 `tctoken` hits); JS also exposes a `getTcToken` query (`frontendSendAndReceive("getTcToken", {wid})` feeding `WAWebVoipStackInterface`). So **the token's source of truth is JS** and the presence `tctoken` node is assembled entirely in JS; the native bridge would only matter if a future bundle wired it for the native VoIP-JID path.

> **TC-token `t` attribute semantics — confirmed in `WAWebTrustedContactsUtils` (`…TSxMupG87E6…js`).** `t` is the token's **issue time as a unix timestamp (seconds)**, and expiry/freshness is decided **entirely JS-side** via AbProp-configured, bucketed durations — the native cache having no TTL is by design (native just hydrates; JS owns staleness). The module:
> ```js
> __d("WAWebTrustedContactsUtils", ["$InternalEnum","WABase64","WATimeUtils","WAWebABProps"], function(...){
>   var e = 15552e3;                                   // 180 days (max duration, seconds)
>   var s = InternalEnum({Sender:"sender", Receiver:"receiver"});   // TcTokenMode
>   function u(t){ return Math.min(getABPropConfigValue(t===s.Receiver?"tctoken_duration":"tctoken_duration_sender"), e) }  // getTcTokenDuration
>   function c(e,t){ return e < d(t) }                 // isTokenExpired(t, mode): t older than cutoff
>   function d(e){ var n=getABPropConfigValue(e===s.Receiver?"tctoken_num_buckets":"tctoken_num_buckets_sender"),
>                  r=u(e), a=Math.floor(unixTime()/r), i=a-(n-1); return castToUnixTime(i*r) }   // tokenExpirationCutoff
>   function m(e){ if(e==null) return true; var t=getABPropConfigValue("tctoken_duration_sender");
>                  return Math.floor(unixTime()/t) > Math.floor(e/t) }   // shouldSendNewToken
> }
> ```
> So: token **duration** = `min(AbProp tctoken_duration[_sender], 180d)`; a token is **expired** when its `t` falls before the bucketed cutoff `(floor(now/duration) - (numBuckets-1)) * duration`; and a new token **should be sent** when `t`'s time-bucket is older than now's. `t` is therefore an issue/sender timestamp, not an explicit expiry, and the bucketing makes it stable within a duration window. The native `RequestTcToken` "refresh" has no TTL of its own — refresh cadence is governed by these JS rules. (Sender vs Receiver each have their own `tctoken_duration*` / `tctoken_num_buckets*` AbProps.)

> **Net data-flow:** JS holds TC tokens (delivered over the protocol). Native needs one (building a `VoipBridgeJid`, or a native presence-subscribe): cache hit → use it; cache miss → `GetTcTokenAsync` fires `ITcTokenBridgeToWeb.RequestUpdate(jidVoipString)` and awaits ≤5 s → JS answers `ITcTokenBridgeToNative.UpdateTcTokens(jids, base64Tokens)` → `TrustedContactManager.SetTcToken` caches it and resolves the waiter. The token is then stamped onto `VoipBridgeJid.tcToken` (VoIP) or emitted as a 1–120-byte `<tctoken t=…>` presence node (`PrivacyTokenContentsMixin`/`TCTokenMixin`). `RequestTcToken` is the no-await prefetch/refresh variant.

> **Port note (Electron):** `TcTokenBridge` is a tiny request/await bridge: implement `Subscribe(web)`, `UpdateTcTokens(jids, base64Tokens[])` (JS→native; base64-decode, cache by ConvoJid), and the `web.RequestUpdate(jidVoipString)` callback (native→JS). Mirror the in-memory `ConcurrentDictionary<jid, Buffer>` cache, the **5 s** await-with-timeout in `getTcTokenAsync`, and the empty-buffer-on-miss contract. Since the bundle already builds the presence `<tctoken>` node in JS, a port mainly needs the cache for the **VoIP** path (`VoipBridgeJid.tcToken`, 1–120-byte blob); if VoIP is stubbed the bridge can be a near-noop. Keep `ToVoipString()`-style JID stringification consistent between the `RequestUpdate` key and the `UpdateTcTokens` reply key, or the waiter never resolves.

### 5.2 The host-object bridge → Electron `contextBridge` + IPC
The single biggest porting task. The bundle expects `window.chrome.webview.hostObjects.<Name>` with:
- `hostObjects.options.defaultSyncProxy = true` (sync-by-default calls returning values inline),
- `forceAsyncMethodMatches = [/Async$/, /AsyncWithSpeller$/]` (only those return promises),
- `hostObjects.options.ignoreMemberNotFoundError = true`,
- a `hostObjects.sync.<Name>` synchronous proxy variant (`hostObjects.sync.ContactsBridge`, `hostObjects.sync.PopulatedContactsBridge`).

Two viable strategies:
1. **Shim `chrome.webview.hostObjects`** in a preload script so the unmodified bundle runs. Implement the Proxy contract over Electron IPC: synchronous methods via `ipcRenderer.sendSync` (note: blocks the renderer — acceptable since WebView2 sync proxy also blocks), `*Async` methods via `ipcRenderer.invoke`, and a callback registry for `Subscribe(webObj)` so native can call `Connect/Disconnect/UpdateNotificationsToken/AppStateChanged` back. This is the **highest-fidelity** path and likely required because the bundle hard-codes `chrome.webview.hostObjects` (and the legacy `chrome.webview.postMessage` comma-IPC, see below).
2. Patch the bundle to use a custom `contextBridge` API — fragile against bundle updates; avoid.

**Map each bridge** (`App.cs:315-318`, `AppModel.cs:217-242`) to a Node implementation:
- `SQLiteBridge.ExecuteSqlite(json) → Promise<json>` → **`better-sqlite3`** plus a **custom AES+HMAC page codec** for the at-rest DBs — note the native store is **not** stock SQLCipher (no `cipher_version`/`kdf_iter`/PRAGMA-key markers; 4096-byte pages + 16-byte per-DB salt + raw 32-byte key, `Sqlite::Open` delegating to internal codec callees — see storage doc 32 and doc 96). The `Async` suffix means it must return a promise.
- `ConnectionBridge` (lifecycle) → Node module managing window visibility ↔ web suspend, WNS-equivalent push token. Implement `Subscribe`, `ChangeSuspending`, `HandleConnectionState`, `AcknowledgeNotificationsToken`; call back `Connect/Disconnect/UpdateNotificationsToken`.
- `NativeAppStateBridge` → trivial pure-JS (`normal`/`minimizedToTray`, timestamps, `DetectNativeClockSkew = Date.now()-webTs`).
- `WebUpdateBridge` (`UpdateWebRevision`) → re-implement the `/spin?subplatform=win_hybrid&revision=` health poll + reload loop.
- `VoipBridge`/`VoipSignalingBridge` → **libwebrtc/pion** + SRTP (separate VoIP doc; biggest native effort).
- `MediaFilesBridge`/`MediaTranscodingBridge` → **ffmpeg/libopus**.
- `ContactsBridge`/`PopulatedContactsBridge` → choose name by initial-sync state, like `AppModel.cs:238` (§3.18). The native side is a validity-ledger + persistence layer over a JS-owned address book: implement `Subscribe`/`UpdateContacts`/`InvalidateContacts`/`AcknowledgeInitialSync`, the `RequestUpdate(idsJson)`/`RequestFrequentContacts(count,voipOnly)` callbacks, the 1 s drain loop, a durable `contactsState.db`-equivalent (`initialState` marker + invalid ids), and the `c.us→s.whatsapp.net` JID normalization. Preserve the DTO quirks (`int isAddressBookContact` compared `==1`, `bool? isHosted`).
- `ClientKeyBridge`/`ServerEncKeySaltBridge`/`TcTokenBridge`/`AdvBridge` → key-material + Curve25519 verify (**libsignal / @noble/curves**).
- `AbPropsBridge` (feature-flag gate, §3.15) → mostly pure-JS, but **must** implement the exposure round-trip: `GetVersion()`, `SetConfigs(hash, json)` (decode the `{configCode,configValue,configExpoKey}` triple and persist to the `abprops.db` equivalent), `Subscribe(web)` (replay un-acked exposures), `AcknowledgeExposure(key)`, and call back `web.NotifyExposureLogged(key)`. A typed `AbPropsValues`-style decode is optional for a JS port (the bundle already holds the values); the load-bearing native obligation is the **durable exposure ledger** (logged/acknowledged per key) so exposures survive restarts and re-attach. If the port stubs this, A/B exposure logging silently breaks but flag *values* still work (they come from JS).
- `LinksPreviewBridge.GetPreviewAsync(link) → Promise<json>` (§3.16) → **not thin**: a main-process `fetch(link,{signal:AbortSignal.timeout(5000), headers:{'User-Agent':'WhatsApp/<ver> W'}})`, scrape `<head>` (OG/Twitter/`<link rel=icon>` priority lists, ≥100×100 icon gate, 100 KiB cap), then fetch+decode the thumbnail (`sharp`: ≤168 then center-crop square ≤140) → `{title,description,thumbnail:Buffer}` / `{title:host,description:link}` / `""`. Run in main to match the native trust boundary; keep the 5 s timeout + size cap (SSRF/DoS guard).
- `WamBridge`, `PreferencesBridge`, `RateAppBridge`, `ScalingControlBridge`, `PicturesBridge`, `SharesheetBridge`, `AppActivationBridge`, `SeamlessMigrationBridge`, `SystemIntegrationsBridge`, `BrowserExtensionsBridge`, `DebugFeaturesBridge`, `TouchpadFix` → mostly thin/pure-JS or OS integration.

**Offline web-resource cache** (`CacheManager`, §3.17, gated by `Webview2EnableOfflineSupport`): re-implement with `session.webRequest`/`protocol.handle`. While online, mirror every 200-OK **same-origin `text/html`** doc to `userData/WebView2Cache/<sha256(stripQuery(url,'windows_offline'))>` + a `.meta` sidecar `{uri,contentType,cachedTs,expiresTs=+168h}`. While **offline** (`!net.isOnline()`), answer those keys from disk with a synthetic `200 OK` + `Access-Control-Allow-Origin: *` until the 7-day TTL lapses. Only HTML is cached here; Electron's own HTTP cache covers JS/CSS/media. Keep flipping `windows_offline=1` on the navigation so the cached shell boots and the bundle detects offline mode.

**Legacy comma-IPC fallback** (`AppModel.cs:249-401`): also re-implement, because the bundle ships the parser (`split(",")` / `NativeResponseToJS` / `handleNativeResponseEvent`) and negotiates via `CompatibilityGetSupportedNativeIpcs`. In Electron: listen on `ipcMessage`/a `postMessage` shim, parse `method,args…,txId`, dispatch by reflected method, reply `txId,NativeResponseToJS,0,<json>` (or `,-1,`). The `bridgeError=1` query param signals the bundle to tolerate/expect this fallback.

### 5.3 Origin / culture / activation
- Origin resolution (`WaWebEnvironmentProvider`): trivial — env var `WA_WEB_URL` or default `https://web.whatsapp.com/`.
- External links → `shell.openExternal`. Deep links: register `whatsapp://` via `app.setAsDefaultProtocolClient` + `second-instance`/`open-url`; rebuild the `whatsapp://send|newchat|newcall` deeplinks (`App.cs:585-692`) and replay to the renderer.
- Single-instance: `app.requestSingleInstanceLock()` ↔ `AppInstance.FindOrRegisterForKey`.
- Localization (`WhatsApp.WebView2`): only matters for **native chrome** (tray menu, dialogs). Use any i18n lib; RTL handled via CSS/`dir`. The web UI localizes itself.
- Tray/notifications: Electron `Tray` + `Notification`; on Linux use **libnotify/D-Bus** (`org.freedesktop.Notifications`) and a `StatusNotifierItem` tray.

### 5.4 Gaps / risks
1. **Sync host-object semantics** — `defaultSyncProxy=true` + `chrome.webview.hostObjects` is a WebView2-specific API. The preload shim must reproduce sync-return + `ignoreMemberNotFoundError` + the `/Async$/` async carve-out, or the bundle breaks subtly. Highest-risk item.
2. **WebContents suspend** — no Electron equivalent of `TrySuspendAsync`; tray power behavior will differ.
3. **`chrome.webview` namespace** — the bundle references `chrome.webview` (Edge-only). Must be polyfilled in preload, not just `contextBridge`.
4. **DevTools enabled in this build** (`IsDevEnvironment()==true`) — likely a debug artifact; the port should gate by build flag.
5. **WebView2 install dialog / `NewBrowserVersionAvailable`** — N/A on Electron (bundled Chromium), drop.
6. **Bundle freshness** — `WebContentUpdateManager` and the `?revision=` health check assume Meta’s server-side spin; reuse the JS bundle but keep the same reload/health loop or accept staleness.
