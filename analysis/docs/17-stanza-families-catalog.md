# 17 — Stanza Families Catalog (Smax.Generated.*)

> Target: Meta native WhatsApp for Windows (WhatsApp.Root.exe, WinUI 3 / Windows App SDK, v2.2607.106.0).
> All paths in §2–§3 are relative to `decompiled_source/`. Every non-trivial claim cites `path.cs:LINE` that was opened and read. Inferences are explicitly labelled.

## 1. Purpose & Scope

WhatsApp's native networking assembly (`WhatsApp.Networking.dll`) carries a large body of **machine-generated stanza classes** under the namespace tree `WhatsApp.Smax.Generated.<Domain>.{Outgoing,Incoming}`. These are the schema-bound, strongly-typed encoders/decoders for a fixed set of XMPP "FunXMPP" stanza families. "Smax" is WhatsApp's internal **stanza schema/codegen framework**: a `.smax` spec per domain is compiled into one C# class per stanza element (and per attribute-enum and per discriminated-union "mixin group"), all built on top of the in-memory `ProtocolTreeNode` model and the runtime helper `SmaxStandardLibrary`.

This document catalogs **all 11 generated domains** shipped in this build — **Groups, Presence, Chatstate, Bot, Blocklists, DirtyBits, Tos, PreKeys, Offline, Pings, UnifiedSession** — plus the two-direction split (`Outgoing` builders, `Incoming` parsers), the shared codegen idioms, the support runtime, and what each family is for. The catalog enumerates purpose, top-level tag, `xmlns`, key fields, child structure, and the C# class surface (namespaces and `Create`/`HandleResponse` factories) so that a Linux/Electron port can reproduce the exact wire shapes.

Scope boundary (confirmed from code): these classes only *build* and *validate* `ProtocolTreeNode` trees. They do not send, encrypt, or route. Serialization to bytes is `BinTreeNodeWriter`; encryption/transport is `StanzaWriter`/`FramesWriter` (see doc on networking-core). And crucially — **the only non-generated caller of any `WhatsApp.Smax.Generated.*` type in the entire dump is `WhatsApp.Networking.StanzaBuilders/ClearDirtyStanzaBuilder.cs`**, which constructs the **DirtyBits** `CleanRequest`/`CleanRequestClean`/`CleanTypeMixin` and writes them via `connection.Write(...)` (`ClearDirtyStanzaBuilder.cs:12-22`). That single builder proves the surface is *designed* to be native-originated — but it is itself **unreferenced** (`SendClearDirty` has no callers), and **no other family** (`Pings`/`Presence`/`PreKeys`/`Bot`/`Blocklists`/etc.) is constructed anywhere outside the generated tree. The native binary node format is shared with the WebView2 JS bundle, which independently carries the same wire vocabulary (`urn:xmpp:whatsapp:dirty`, `tctoken`, `composing`/`paused`) — and, as the round-2 beautified bundle now shows, the **same `.smax` codegen compiled to JS**: the bundle ships the full parallel `WASmax{Out,In}<Domain>*` module tree for all 11 domains, with *live* call-sites (e.g. `WAComms`→`WASmaxOutPingsClientRequest.makeClientRequest()` for the `w:p` ping; `WAWebClearDirtyBitsJob` for the dirty `<clean>`) where the C# twins have none. So these generated families are a (dormant) parallel native surface, with production traffic driven by the JS bundle. This is **now confirmed** — see §6 Q1.

## 3. How It Works

### 3.1 The two generated shapes

**Outgoing** — a `sealed class` with a public `ProtocolTreeNode Node { get; }` and a static `Create(...)` factory wrapping a `private` constructor that builds the node with `ProtocolTreeNodeBuilder`. Example, the Ping (`w:p`) request:

```csharp
// Pings.Outgoing/ClientRequest.cs:10-19
ProtocolTreeNodeBuilder b = new ProtocolTreeNodeBuilder("iq");
b.AddStringAttribute("id", idAttr);
b.AddAttribute(new ProtocolKeyValue("type", "get"));
b.AddAttribute(new ProtocolKeyValue("xmlns", "w:p"));
b.AddAttribute(new ProtocolKeyValue("to",
    (DomainJid)JidFactory.CreateNewJid("s.whatsapp.net"), ProtocolKeyValue.KVType.Jid));
Node = b.Build();
```

(`Pings.Outgoing/ClientRequest.cs:13-18`.) Attribute *order is fixed by the spec* and reproduced literally; this matters because `BinTreeNodeWriter` preserves attribute order on the wire.

**Incoming** — a `sealed class` with read-only typed properties and a static `Create(ProtocolTreeNode node, [request], bool logFailure = true)` returning `Either<T, SmaxError>`. It instantiates `new SmaxStandardLibrary()`, checks the tag, extracts/validates each attribute and child, and returns the populated object as `Left` or a `SmaxError` as `Right`. Parse exceptions are `CorruptStreamException`, caught and reported through `FailuresService.Investigate(...)` with the original Windows build path baked in (e.g. `Pings.Incoming/ClientResponseServerResponse.cs:59-65`).

### 3.2 Reference (cross-stanza) validation

Response parsers receive the originating request node and assert that `from`/`id` echo back. The pattern (`Pings.Incoming/ClientResponseServerResponse.cs:48-52`):

```csharp
smax.TryGetRequiredStringAttributeValue(requestNode, ["id"], isReference:false, null,null,null, out string reqId);
smax.TryGetRequiredStringAttributeValue(node,        ["id"], isReference:true,  reqId, null,null, out string id);
```

`isReference:true` means "must equal `expectedValue`" (`SmaxStandardLibrary.cs:177-185, 331-337`). For `from`, the request's `to` JID is the expected value (`Blocklists.Incoming/GetBlockListResponseSuccessWithMatch.cs:36-40`; `DirtyBits.Incoming/IQErrorResponseMixin.cs:30-39`). This is how IQ correlation is enforced *inside* the typed layer on top of the `IqRequestsTracker` id matching.

### 3.3 Response "Handler" dispatch (the union over result types)

For IQ-style families, an `<X>Handler` class fans an inbound result node across every possible response class in priority order, calling the matching `Action<T>` and returning `bool`. From `DirtyBits.Incoming/CleanHandler.cs:11-33`: try `CleanResponseSuccess.Create(...)`; if `IsLeft`, invoke the success callback and return true; else record the error string and try `CleanResponseError.Create(...)`; if all fail, `FailuresService.Investigate(...)` logs every per-type error and returns false. `PreKeys.Incoming/SetHandler.cs:11-52` does the same across 4 response types; `Bot.Incoming/BotListHandler.cs:11-61` across 5 (V1–V4 success + error); `Tos.Incoming/QueryAcceptanceHandler.cs:11-43` across 3 (success / client-error / server-error). This is a generated "tagged-union dispatch": the spec lists allowed responses, codegen emits one `Create` attempt per response and short-circuits on first parse success.

**How a typed `Handler` actually gets driven (the id-correlation glue, now fully traced — see §6 Q6).** The typed `<X>Handler` is *not* itself registered with the connection; the glue is the generic `IqResultHandler`. On send, a builder calls `connection.MakeId(prefix)` then `connection.AddIqHandler(id, new IqResultHandler(...))` before `connection.Write(node)` (`ClearDirtyStanzaBuilder.cs:50-54`); `Connection.AddIqHandler`/`MakeId` delegate to `IqRequestsTracker` (`Connection.cs:28-36`), which stores the handler in a lock-guarded `Dictionary<string,IqResultHandler> _pendingServerRequests` keyed by id (`IqRequestsTracker.cs:23-33`). On receive, `WAProtocol.ProcessIq` reads the inbound `id`/`type`/`from`, and for `type=="result"`/`"error"` calls `_requestsTracker.PopIqHandler(id)` (which removes+returns the entry, `IqRequestsTracker.cs:45-55`) and invokes `iqResultHandler.Parse(node, from)` / `.ErrorNode(node)` (`WAProtocol.cs:158-186`). The typed `Handler` (e.g. `CleanHandler`) is meant to run *inside* that `Parse` callback. In this build, the one live native sender passes `new IqResultHandler(null)` — a no-op parse (`ClearDirtyStanzaBuilder.cs:53`) — so the typed `CleanHandler` is registered-but-never-driven here; a port must wire the typed parser into the `IqResultHandler.Parse` delegate. (Note also that `ClearDirtyStanzaBuilder` ships **four** send methods: the two Smax-typed ones above, a `[Obsolete]` string-keyed shim (`:24-28`), and a fourth that hand-builds the `<iq xmlns=urn:xmpp:whatsapp:dirty>` node directly, *bypassing the Smax layer* (`:30-48`) — none of the four has any caller in the dump; see §6 Q1.)

### 3.4 MixinGroup = discriminated union; Mixin = reusable attribute/child bundle

Two codegen concepts share the "Mixin" name:

- **`MixinGroup` (interface) + `<Name>Extensions.Match`** — a *closed sum type*. The interface is just `ProtocolTreeNode Node { get; }` (`Chatstate.Outgoing/MixinGroup.cs:5-8`); each variant `: MixinGroup`; the extension provides exhaustive `Match(Action<A>, Action<B>, …)` / `Match<T>(Func<A,T>, …)` that `throw new InvalidOperationException("Unexpected type in match function.")` on an unknown type (`Chatstate.Outgoing/MixinGroupExtensions.cs:7-33`; `Presence.Incoming/MixinGroupExtensions.cs:7-60` — 5 variants). Incoming `MixinGroup` is resolved by `SmaxStandardLibrary.TryGetRequiredMixinGroup`, which tries each variant's `Create` and returns the first `Left` (`SmaxStandardLibrary.cs:463-484`); `Presence.Incoming/ServerUpdateServerRequest.cs:29-39` wires 5 candidate `Create` lambdas via `SelectLeft`.
- **plain `<Name>Mixin`** — a *named, reusable fragment* of attributes/children shared by several stanzas (e.g. `IQErrorResponseMixin`, `GroupInfoMixin`, `TCTokenMixin`, `VerifiedNameMixin`). On Outgoing it is `Merge`d into a parent builder (`ProtocolTreeNodeBuilder.Merge(mixin?.Node)`, e.g. `PreKeys.Outgoing/SetRequest.cs:23`); on Incoming it is pulled via `TryGetRequiredMixin`/`TryGetOptionalMixin` against the *same* node (empty path), so its attributes/children co-exist on the parent (`Groups.Incoming/GroupInfoMixin.cs:183-190`). A mixin may itself implement a `MixinGroup` interface (multiple inheritance of generated interfaces), e.g. `GroupInfoMixin : BatchGetGroupInfo1MixinGroup, GroupInfoMixinGroup` (`Groups.Incoming/GroupInfoMixin.cs:9`).

### 3.5 Child elements, leaf data, and enums

- **Child element classes** (`<Parent><Child>`) are themselves Outgoing/Incoming classes with their own tag; parents compose via `AddChild(child.Node)` (`PreKeys.Outgoing/SetRequest.cs:18-22`) and decode via `TryGetRequiredChild`/`TryGetChildren` (`Bot.Incoming/BotListResponseSuccessV2Bot.cs:35-42`).
- **Leaf binary data** uses `AddData(byte[])` with a fixed length check: PreKey public value `ValidateBinary(elementValue, 32L, 32L)` → `<value/>` data (`PreKeys.Outgoing/SetRequestListKeyValue.cs:14-15`); signed-prekey signature `ValidateBinary(…, 64L, 64L)` → `<signature/>` (`PreKeys.Outgoing/SetRequestSkeySignature.cs:14-15`). On read, `#`-prefixed path segments mean "element data, not attribute" (`SmaxStandardLibrary.cs:176, 316`).
- **Attribute enums** are emitted as a nested `enum Type { … }` plus a `…EnumExtension` static class holding a `BiDictionary<Type,string>` and `GetValue`/`GetEnum` (wire-string ↔ enum). Example DirtyBits clean type: `AccountSync="account_sync"`, `Groups="groups"`, `NewsletterMetadata="newsletter_metadata"` (`DirtyBits.Outgoing/CleanTypeMixinTypeEnumExtension.cs:7-21`). Incoming enums validate against the allowed `cases[]` via `TryGetRequiredStringEnumValue`/`TryGetOptionalStringEnumValue` (`SmaxStandardLibrary.cs:505-560`), e.g. `state ∈ {false,true}` (`Tos.Incoming/TosNoticeWithStateMixin.cs:34-38`), `last ∈ {deny,error,none}` (`Presence.Incoming/LastSeenWithOtherValueMixin.cs:45-49`).

### 3.6 The SmaxStandardLibrary contract (the validation primitives)

All extraction goes through `SmaxStandardLibrary` (`WhatsApp.Networking.Smax/SmaxStandardLibrary.cs`). Key primitives actually used by the families:

- `TryCheckNodeTag(node, tag)` — tag guard (`:86-94`).
- `TryGetRequired/OptionalStringAttributeValue(node, path[], isReference, expected, lowerBound, upperBound, out)` — string attr with reference-equality, length bounds; path's last segment may be `#` for element data (`:143-208`).
- `TryGetRequired/OptionalLongAttributeValue` — integer parse + numeric bounds (`:283-358`).
- `TryGetRequired/OptionalByteArrayAttributeValue` — binary element value, must be `#`-path, length bounds (`:210-281`).
- `TryGetRequired/OptionalJidAttributeValue<T>` and `CastJid<T>` — typed JID extraction; `CastJid` collapses `DeviceJid{DeviceId:0}`→`UserJid` and promotes `UserJid`→`DeviceJid(.,0)` (`:807-822`).
- `TryGetRequired/OptionalJidEnum(castFns[])` — JID that must match one of several JID subtypes (`:580-630`); used for `to`/`from` that may be `DomainJid` or `UserJid` (`Pings.Incoming/ClientResponseServerResponse.cs:40-43`).
- `TryGetRequiredChild` / `TryGetChildren(min,max)` / `TryGetOptionalChild` — child traversal by `path[]` with count bounds (`:45-141`). PreKey `<list>` allows 0..20000 keys (`PreKeys.Outgoing/SetRequestList.cs:17`); group `<participant>` 0..19999 (`Groups.Incoming/GroupInfoMixin.cs:99`); offline `<item>` 0..50 (`Offline.Incoming/…ThreadMetadata.cs:29`).
- `TryGetRequired/OptionalMixin` and `TryGetRequired/OptionalMixinGroup` — mixin and union resolution (`:430-503`).
- `CopyKeyValue(node, path[], newName, required)` — copies an attribute (preserving `kvType`) or `#`-data under a new name, for echo-style acks (`:632-673`).
- Outgoing-side `Validate*` (`ValidateString`, `ValidateLong`, `ValidateBinary`, `ValidateChildren`, `ValidateJidEnum`) assert via `SmaxAssert.DebugFail` (non-throwing in release) before building (`:710-805`).

On any failure the helper sets `ParseError` (a human path-qualified message, e.g. `"Required child <tos/> missing for tag <notification/> in the path 'notification.tos'"`, `:35`), and the caller wraps it in `SmaxError`.

### 3.7 Family-by-family catalog

Notation: **Tag** = top-level `ProtocolTreeNodeBuilder` tag; **xmlns**/**type** as set in the builder; → child elements; ⊕ merged mixins.

#### Pings — keepalive ping/pong (`w:p`)
- **Outgoing `ClientRequest`** (`Pings.Outgoing/ClientRequest.cs`): `<iq type=get xmlns=w:p to=s.whatsapp.net id=…/>`. Factory `Create(string idAttr)`.
- **Incoming `ClientResponseServerResponse`** (`Pings.Incoming/ClientResponseServerResponse.cs`): parses `<iq>` with `from` (Domain|User JID enum), `type="result"`, `id`==request id, required `long t` (server time). Props `FromAttr/TypeAttr/IdAttr/TAttr`. `Create(node, ClientRequest request, logFailure)`.
- Purpose: liveness probe; `t` is the authoritative server clock echoed back.

#### Chatstate — typing indicators (send-only)
- Top-level tag `<chatstate>`; **no xmlns/type** (it is a top-level stanza, not an IQ).
- **`ClientNotificationRequest`** (`Chatstate.Outgoing/ClientNotificationRequest.cs`): `to` must be `GroupJid|UserJid` (validated `:15`), `Merge(MixinGroup.Node)`, optional `Merge(InternalTestMixin?.Node)`.
- **`MixinGroup`** union = `ComposingMixin | PausedMixin` (`Chatstate.Outgoing/MixinGroupExtensions.cs`). `ComposingMixin` wraps child `<composing>` (`ComposingMixin.cs:13-15`); `ComposingMixinComposing` adds `media="audio"` only when `hasMediaAttr` (`ComposingMixinComposing.cs:14-17`). `PausedMixin`/`PausedMixinPaused` mirror it with `<paused>`.
- `InternalTestMixin`/`InternalTestMixinTest` is a dev-only `<test>` fragment: it wraps a `smax:any`→`<test>` child (`InternalTestMixin.cs:13-15`) and the leaf carries a single *optional* `config` string (≤100 chars), emitting a bare `<test/>` when absent (`InternalTestMixinTest.cs:13-19`). It is merged only as the nullable `Merge(InternalTestMixin?.Node)` in `ClientNotificationRequest`, and a full-tree grep finds **no caller** of `ClientNotificationRequest`/`InternalTestMixin` outside the generated tree — so in this build it is **never emitted** (see §6 Q5).
- Purpose: "typing…"/"recording audio…"/"stopped typing" presence within a chat or group.

#### Presence — availability + subscription + last-seen
- **Outgoing**
  - `AvailabilityRequest` (`Presence.Outgoing/AvailabilityRequest.cs`): `<presence>` with optional `type ∈ {Available→"available", Unavailable→"unavailable"}` (`AvailabilityRequestTypeEnumExtension`) and optional `name`.
  - `SubscribeRequest` (`SubscribeRequest.cs`): `<presence type=subscribe to=…>`; `to` is Group|User; optional `name`; optional `context` (GroupJid); `Merge(TCTokenMixin?)`.
  - `TCTokenMixin` (`TCTokenMixin.cs`): builds a `smax:any` wrapper carrying child `<tctoken>` (`TCTokenMixinTctoken`); `PrivacyTokenContentsMixin` holds privacy-token element data. **`smax:any` is a codegen-only sentinel that is stripped at merge time** — when its node is `Merge`d into a parent builder, `ProtocolTreeNodeBuilder.CheckTag` re-tags the builder to the child's real tag (`if ("smax:any" == tag) tag = node.tag;`, `ProtocolTreeNodeBuilder.cs:178-185`), so the wrapper never reaches `Build()`/the wire. Confirmed absent (0 occurrences) from the waweb JS bundle (see §6 Q2).
- **Incoming `ServerUpdateServerRequest`** (`Presence.Incoming/ServerUpdateServerRequest.cs`): `<presence>` whose body is the `MixinGroup` union of **5 variants** (`:29-36`): `GroupAvailableMixin`, `GroupUnavailableMixin`, `LastSeenWithOtherValueMixin`, `UserUnavailableMixin`, `AvailableMixin`.

- Purpose: online/offline presence broadcast and per-contact subscription; trusted-contact (`tctoken`) attached on subscribe.

#### DirtyBits — "clean" server-side dirty flags (`urn:xmpp:whatsapp:dirty`)
- **Outgoing `CleanRequest`** (`DirtyBits.Outgoing/CleanRequest.cs`): `<iq to=s.whatsapp.net xmlns=urn:xmpp:whatsapp:dirty id=… type=set>` → child `<clean>` (`CleanRequestClean` merges a `MixinGroup`). `CleanTypeMixin`/`CleanTypeWithTimestampMixin` carry `type` (enum: account_sync/groups/newsletter_metadata, `CleanTypeMixinTypeEnumExtension.cs`) and optional `timestamp`.
- **Incoming**: `CleanHandler` dispatches `CleanResponseSuccess | CleanResponseError` (`CleanHandler.cs`). `CleanResponseSuccess` requires `IQResultResponseMixin` (id/from/type=result) (`CleanResponseSuccess.cs:31`). `IQErrorResponseMixin` validates `type="error"`, id-ref, from-ref (`IQErrorResponseMixin.cs:25-44`). Error sub-mixins: BadRequest/FeatureNotImplemented/InternalServerError/ServiceUnavailable.
- Purpose: tell the client which app-state collections are stale and need re-sync; client acks by clearing them.

#### Tos — Terms-of-Service notices (`xmlns=tos`)
- **Outgoing**
  - `QueryAcceptanceRequest` (`Tos.Outgoing/QueryAcceptanceRequest.cs`): `<iq to=s.whatsapp.net xmlns=tos id=… type=get>` → `<request>` → `<notice>` (`QueryAcceptanceRequestRequestNotice`).

#### PreKeys — Signal key upload (`xmlns=encrypt`)
- **Outgoing `SetRequest`** (`PreKeys.Outgoing/SetRequest.cs`): `<iq type=set id=… xmlns=encrypt to=s.whatsapp.net>` with required children, in order:

#### Bot — AI-bot discovery list (`xmlns=bot`)
- **Outgoing `BotListRequest`** (`Bot.Outgoing/BotListRequest.cs`): `<iq to=s.whatsapp.net xmlns=bot id=… type=get>` → `<bot>` (`BotListRequestBot` → nested `<bot v=…>` with `BotListRequestBotVEnumExtension` version).

- Purpose: fetch the catalog of Meta AI / third-party bots the user can chat with, with server-driven section theming. Multiple `V*` classes = forward/backward schema negotiation.

#### Blocklists — block/unblock + sync (`xmlns=blocklist`)
- **Outgoing**
  - `GetBlockListRequest` (`Blocklists.Outgoing/GetBlockListRequest.cs`): `<iq to=s.whatsapp.net xmlns=blocklist type=get id=…>` + optional `<item>` (sync hint).

- Purpose: read and mutate the server block list; the "match/mismatch/migrated" variants reconcile the client's local hash against server state (PN↔LID addressing migration).

#### Offline — offline thread-metadata delivery (receive-only)
- **Incoming `ThreadMetadataServerRequest`** (`Offline.Incoming/ThreadMetadataServerRequest.cs`): tag `<ib>`, `from`==s.whatsapp.net (`:31`), required `<thread_metadata>` child. `ThreadMetadataServerRequestThreadMetadata` (`…ThreadMetadata.cs`) holds 0..50 `<item>` + optional `ThreadMetadataDelayedMixin` (counts of delayed `<notifications>` / `<status_msgs>` queued while offline).
- Purpose: on reconnect, the server's `<ib>` (info-broadcast) summarizes per-thread metadata and how many notifications/status messages were withheld during the offline window.

## 5. Linux/Electron Port Mapping

The Smax layer is the *easiest* part to port because it is data-only and deterministic.

- **Node model** → a TypeScript `ProtocolTreeNode` (tag, attrs ordered, children | data). The port already needs this for FunXMPP framing; the Smax classes sit directly on top. Reuse the **waweb JS bundle's** existing WAP node builders if present (the bundle almost certainly contains the JS equivalents of these specs) rather than re-deriving from C#.
- **Codegen vs hand-write**: 510 classes is a lot. Recommended path — **do not hand-port all 510**. Port the *runtime* (`SmaxStandardLibrary` → ~400 lines of TS validators returning a `Result<T, SmaxError>` analog of `Either`) and then port only the families the Electron shell actually originates natively. In this hybrid design the renderer (waweb JS) handles message/receipt/group UI traffic itself; the native side historically only originates a few low-level stanzas. Concretely, the **must-have** native-origin families for a from-scratch client that does its own networking are: **Pings** (keepalive), **PreKeys** (X3DH upload), **DirtyBits clean**, **Presence/Chatstate** (if not delegated to JS). Groups/Bot/Blocklists/Tos can be delegated to the reused JS bundle.
- **Either/Result** → `neverthrow`'s `Result`/`ok`/`err`, or a hand-rolled discriminated union. The exhaustive `Match` extensions map cleanly to a TS tagged-union `switch` with an `assertNever` default (mirrors the `InvalidOperationException` default).
- **Enums + BiDictionary** → a `const` object plus a reverse `Map`; the wire strings (`"account_sync"`, `"newsletter_metadata"`, presence `"deny|error|none"`, etc.) must match byte-for-byte.
- **JID typing** → a TS `Jid` union with `DomainJid|UserJid|GroupJid|DeviceJid`; replicate `CastJid`'s device-0↔user collapse (`SmaxStandardLibrary.cs:807-822`) exactly, as it affects equality in reference checks.
- **Binary length invariants** are load-bearing for crypto interop: PreKey value 32B, signed-prekey value 32B, signature 64B (`PreKeys.Outgoing/Set*.cs`). A libsignal-based port (`@signalapp/libsignal-client`) produces exactly these sizes.
- **Risks/gaps**: (1) attribute *order* must be preserved on the wire — TS object key order is insertion-ordered, so build attrs as an ordered array, not a plain object. (2) The `smax:any` transparent-wrapper tag (`Presence.Outgoing/TCTokenMixin.cs:13`) is a codegen sentinel that must be *merged away*, never serialized literally — **confirmed** by `ProtocolTreeNodeBuilder.CheckTag` re-tagging on merge (`ProtocolTreeNodeBuilder.cs:178-185`) and by its 0 occurrences in the waweb JS bundle (§6 Q2); replicate this collapse in the TS builder's `merge`. (3) The 5 Bot `V1..V4` response shapes mean the port must tolerate server schema drift; prefer parsing leniently. (4) `from`-path strings in errors are Windows artifacts — drop them.
