## 1. Purpose & Scope

This document covers the **VoIP calling subsystem** of the native Windows client: how 1:1 and group voice/video calls are placed, received, signaled, keyed, and torn down; how call signaling tunnels through the same FunXMPP/XMPP stanza layer as messaging; and how the call media engine itself lives in native code reached through a WinRT projection.

The architecture is a **three-layer hybrid**:

1. **WebView2 JS bundle** (waweb) — owns the *signaling transport* (it actually sends/receives `call`/`message`/`receipt`/`ack` stanzas on the wire via the Signal-encrypted XMPP connection), the call UI surfaces wired to the model, the multi-device/device-JID resolution, and the E2E (Signal) encryption of the `offer`/`enc_rekey` payloads. It drives native via two host objects: `VoipBridge` and `VoipSignalingBridge`.
2. **Managed C# shell** (`WhatsApp.VoIP` + `WhatsApp.Root/Voip`) — a thin marshaling/orchestration layer: it decodes base64 WAP into native `XmlNode`s, converts JID strings into `VoipBridgeJid`, hops everything onto a dedicated VoIP worker thread, derives SRTP/P2P keys via Axolotl/HKDF, owns the `CallModel`/call-log/missed-call/toast logic, and surfaces native call events back to JS as JSON.
3. **Native call engine** (`WhatsAppNative.dll`, C++/Rust) — `IVoip` (74 members): the real state machine, ICE/STUN/TURN relay negotiation, SRTP transport, Opus/video codecs, jitter buffer, BWE, renderer. Not present in the C# dump; reached only through the CsWinRT projection `WhatsAppNativeProjection`.

**Out of scope** (covered elsewhere): the Signal/X3DH crypto that protects the `offer` payload (doc 20), the FunXMPP binary node format (doc 12), the connection/Noise transport (docs 10–11), and the WebView2 host-object bridge mechanics in general (doc 31).

## 3. How It Works

### 3.1 The big picture — who actually sends call stanzas

A crucial and counter-intuitive fact established by the message-receive path (doc 16) and reconfirmed here: **the native layer does NOT put call stanzas on the socket.** `WAProtocol.ProcessStanza` only natively handles `iq`/`success`/`failure`; `message`/`receipt`/`notification`/`call` top-level stanzas are dispatched in the **JS bundle**. So for VoIP:

- **Inbound**: the JS bundle receives the Signal-decrypted `call` stanza, base64-encodes the inner WAP node, and pushes it into native via `VoipSignalingBridge.HandleIncomingSignalingOffer/...Message/...Ack/...Receipt`.
- **Outbound**: native produces a signaling node, hands it back to JS as base64 WAP via `IVoipBridgeToWeb.HandleSignalingXmpp(...)`; JS wraps it in the proper `call`/`message` envelope, applies Signal E2E encryption when required, and actually transmits it.

The native engine therefore owns *call logic and media*, while JS owns *transport and E2E crypto of the signaling payload*. The C# shell is the marshaler in between.

### 3.3 Native engine bring-up — `VoipManaged`

The engine is created lazily by `VoipManaged.ManagedInstance` (`VoipManaged.cs:53-91`):

```
IVoip voip = new VoipFactory().GetVoip(
    deviceManagerStatefulEnumeration: true, voipAsyncOperationManager: true,
    VoipLogLevel.Info, UseCachedMediaManager, UseWinRTRenderer, 3000);   // :60
VoipCallbacks voipCallbacks = new VoipCallbacks(voip, new VoipSignaling()); // :67
voip.SetCallbacks(voipCallbacks);                                          // :68
VoipManaged result = new VoipManaged(voip, voipCallbacks);                 // :69
OnVoipInstanceCreated(voip);                                               // :70  -> subscribes network-type changes
```

After activation it pushes initial state to the engine on the worker thread (`VoipManaged.cs:71-85`): `SetScreenSize(1920,1080)` (`DefaultVideoSize`, :31), `SetDevicePerformanceClass(...)` from `MachineSpec` (the **local** perf class — see below), `SetMicMuteKeyEnabled(true)`, `SetResetAfterActivate(...)`, a background image for the swap chain, `SetHealthCheckDelayAndPeriod(max(CameraHealthCheckDelay,6000), max(...,3000))`, `EnableWinRTRenderer(AbProps.Props.WinrtRenderer)`, and `SetRendererScaleInterpolation(HighQualityCubic)`.

The engine is **warmed up at app start** even before any call: `AppModel.Warmup` dispatches `() => VoipManaged.Instance` onto the worker (`AppModel.cs:182`), so `IVoip` is hot when the first offer arrives.

**Device performance class (the `VoipDevicePerformanceClass` enum) is derived in C# from CPU cores + RAM**, not opaque. `VoipDeviceInfo.DevicePerformanceClass` (`VoipDeviceInfo.cs:7-63`) buckets `(CpuCores, MemoryGb)` → `Low` (1 core), `ModLow` (2 cores, or 3 cores & <4 GB), `Mid` (≤7 GB at higher tiers), `ModHigh` (8 GB / 6-8 cores & ≤7 GB), `High` (≥6 cores or >8 GB), else `NotDefined`. This is pushed to native via `SetDevicePerformanceClass(deviceClass, deviceClass.ToString())` (`VoipManaged.cs:185-187`); native validates it (`wa_set_device_performance_class: Invalid device class (%d)`) and uses it in rate-control decisions (it appears as `peer_device_class`/`android_device_class` in the `vid_rc_dyn` decision log). Note this enum is **distinct** from the wire `device_class="2016"` numeric in the offer (§3.5): that attribute is the peer's **device-year-class** integer — the Meta `device-year-class` performance metric expressed as the model-year of an equivalent-performance flagship (so `"2016"` = the 2016 year-class bucket), confirmed by the native string `peer_year_class_2016` and the year-class codec gate `high_sample_rate_year_class_threshold` (native disables Opus super-wideband below a year-class threshold), and by the WAM fields `yearClass` (id 689) / `yearClass2016` (id 2617) present in both the web bundle and Baileys/whatsmeow schemas. It is consumed natively in rate-control via `cond_range_peer_device_class`/`cond_device_class_mask` and is **not** the `VoipDevicePerformanceClass` enum (see §6.7).

Network type is mapped from `NetworkStateMonitor` and pushed via `IVoip.SetNetworkType` whenever it changes, deduped with `DistinctUntilChanged` (`VoipManaged.cs:167-178`, `ConvertConnectionTypeToVoipNetworkType` :134-144 → `Wifi`/`Cellular`/`None`/`Unknown`).

**Threading.** All `IVoip` calls go through `VoipManaged.Worker`, a `VoipDispatcher` (`VoipDispatcher.cs`) backed by a single `ConcurrentQueueDispatcher` tagged `Log.DispatcherId.Voip`. The pattern everywhere is `IVoip voip = await dispatcher.Capture("MethodName", "<src>")` (e.g. `VoipWebCore.cs:27`). `Capture` returns the captured `IVoip` (i.e. `VoipManaged.Instance`, see `VoipDispatcher.TryGetCaptured` :17-24) once execution is on the VoIP thread; native objects are STA/thread-affined, so this serial hop is mandatory. **The affinity is confirmed from the binary** (not just inferred from the dispatcher): `strings WhatsAppNative.dll` contains the WinRT wrong-thread machinery `?__abi_WinRTraiseWrongThreadException@@YAXXZ` and the type `.?AUhresult_wrong_thread@winrt@@` — i.e. calling `IVoip` off the captured thread raises `RPC_E_WRONG_THREAD`, which is exactly why the dispatcher serializes every call. Internally the engine runs its own named worker threads (`p2p_worker_thread`, `start_call_timer_worker_thread`/`call_timer_worker_thread`, `event_worker_thread`, a dedicated **network event thread**, one-side-BWE) and posts results back over a callback queue (`enable_post_net_health_status_to_cb_queue`, `post kSendConnectStat to cb queue`) under a prioritized recursive lock-scope guard (`check_locking_order re-entrance into scope=%d, priority=%d, recursive=%d`).

**The inbound (native→managed) up-call contract is now established** (see §6.8 for the full evidence chain). The `VoipCallbacks` ComWrappers vtable exposes **only** `IVoipCallbacks` — no agile/free-threaded marshaler (`WhatsApp_VoipCallbacksWinRTTypeDetails.cs:8-18`) — so the callback object is a thread-affined WinRT object, and `SetCallbacks` is wired inline at bring-up, not on the worker (`VoipManaged.cs:67-68`). `VoipManaged.Worker` is *not* a fixed OS thread: `ConcurrentQueueDispatcher` drains its serial queue on rotating ThreadPool threads (`ConcurrentQueueDispatcher.cs:35,82-83,98`). Consequently the engine delivers `OnEvent`/`OnSignalingXMPP` **synchronously on its own event thread**, and the managed handler proves it is not already on the worker by **re-marshaling** deferred work back via `VoipManaged.Worker.Dispatch(CleanUpMuteState/PlayEndTone)` (`VoipCallbacks.cs:1185-1186`) and `await Worker.ContinueIn(...)` (`:1411`). A managed callback *may* re-enter `IVoip` synchronously (e.g. `_operations.MediaManagerTearDown()` at `:1176`), but that re-entry is governed by the same WinRT thread-affinity check (`hresult_wrong_thread`), which is why heavier follow-up actions are pushed onto the serial worker instead.

### 3.4 Inbound offer flow (line-by-line)

`VoipSignalingBridge.HandleIncomingSignalingOffer` → `VoipWebCore.HandleIncomingSignalingOffer` (`VoipWebCore.cs:25-58`):

1. `voip = await dispatcher.Capture("HandleIncomingSignalingOffer", …)` (:27) — hop to VoIP thread; bail if null.
2. `XmlNode xmlNode = DecodeWapOrLogError(xmlNodeBase64, …)` (:33). `DecodeBase64Wap` (:543-557): `Convert.FromBase64String` → `new BinTreeNodeReader(TokenMap)` → `binTreeNodeReader.ParseTreeNode(stream)` (a `ProtocolTreeNode`) → `VoipSignaling.ConvertFromStanzaElement(...)` to turn it into a native `XmlNode`. `TokenMap` is a single static `TokenDictionary` (:23).
3. Build `VoipNativeMsgInfo {platform, version, t, e, offline}` (`CreateMsgInfo`, :643-653).
4. Build `VoipNativeOfferInfo { isNotContact = isOfferNotContact }` (:39-42).
5. `VoipBridgeJid jid = CreateVoipBridgeDeviceJid(peerJid, tcTokenB64)` (:43).
6. `await voip.OnIncomingSignalOffer(xmlNode, msgInfo, offerInfo, jid)` (:44) — hands the offer to the native state machine.

`HandleIncomingSignalingMessage` / `…Ack` / `…Receipt` are structurally identical, calling `OnIncomingSignalMsg` / `OnIncomingSignalAck` / `OnIncomingSignalReceipt` respectively (`VoipWebCore.cs:60-152`). `Ack` carries a `VoipNativeAckInfo {error, type}` (:104-108) — this is how nacks/acks of the offer reach the engine.

The native engine then fires events up through `IVoipCallbacks.OnEvent`, ultimately driving `CallState.ReceivedCall` and a toast (see §3.8).

### 3.5 The offer stanza — confirmed wire layout (ground truth)

From `android/decoded-offer-stanza.json` (Android capture, same protocol). Root `<offer>` attrs:

| attr | example | meaning |
|---|---|---|
| `call-creator` | `91134994993240@lid` | device/user that created the call (LID form here) |
| `call-id` | `AC504134D682E01388604A6B4173B541` | 16-byte hex call id |
| `device_class` | `2016` | caller **device-year-class** (Meta `device-year-class` metric; `2016` = 2016 year-class bucket — §3.3, §6.7) |
| `caller_pn` | `918810868228@s.whatsapp.net` | caller phone-number JID (PN↔LID pairing) |
| `joinable` | `1` | joinable group/link call |
| `caller_country_code` | `IN` | used by `caller_country_code`/401/405 gating (see §3.9) |

Children (the call-engine payload):
- `<audio rate="16000" enc="opus"/>` and `<audio rate="8000" enc="opus"/>` — offered Opus capabilities at 16 kHz and 8 kHz.
- `<capability ver="1">` raw bytes `0105f709e4fa13` — feature bitmap.
- **`<enc v="2" type="pkmsg">`** 32 bytes — the **Signal-encrypted call cipher key** material (`pkmsg` = a Signal PreKeyMessage; would be `msg` on an established session). This is the E2E payload the JS Signal layer decrypts; its plaintext yields the per-call `cipherKey` that feeds `Axolotl.CallKeysFromCipherKeyV2` (§3.6).
- `<encopt keygen="2"/>` — key-generation scheme selector (matches `voip_settings_version` keygen).
- `<metadata peer_abtest_bucket … />`, `<net medium="3"/>` (a network-medium hint; native consumes it via `wa_call_update_network_medium_2`/`send_network_medium_info_async` and gates features on it — e.g. `weak wifi is skipped as network medium is cellular`. **Value space (now resolved, §6.4):** the engine-facing medium enum is the CsWinRT projection `WhatsAppNative.VoipNetworkType {Unknown=0, Cellular=1, Wifi=2, None=3}` (`VoipNetworkType.cs:7-13`), and both the local stack enum `WAWebVoipNetworkMedium` and the **peer**-reported `WAWebWamEnumPeerCallNetworkMedium` are `CELLULAR=1, WIFI=2, NONE=3` (`research/waweb-unmin/jEpDguw_pz8.js:3614-3618,459-465`). So the captured `<net medium="3">` = **NONE** (not cellular), i.e. the peer advertised no/unknown network at offer time; the wire space is `1=Cellular, 2=Wifi, 3=None`), `<rte>` 18 bytes (`240940e331dc89ca2c0b709e5bde0c92e77e` in the capture) — a packed **reflexive transport address**: 18 bytes is exactly the IPv6 `te2` form (`addr(16) ‖ port(2)`, §6.1), so `<rte>` uses the same `pj_sockaddr`-compatible `addr(4|16)‖port(2)` packing as `te2` (6 bytes for an IPv4 reflexive address). Native role: `p2p_add_rte_reflexive_addr`, `Adding reflexive address as candidate`, `dyn_alloc_default_rtt_estimate`; cross-reference corroborated by the WAM fieldstat pair `signalingReflexiveIpPeer`/`signalingReflexiveIpSelf` string-IP fields in `jEpDguw_pz8.js:2201-2202`, i.e. `<rte>` carries each side's STUN-reflexive transport address), `<uploadfieldstat/>`.
- **`<voip_settings uncompressed="1">`** — a 12,685-byte JSON blob of engine tunables. The decoded blob's **complete** top-level key set (confirmed by decoding the `__buffer_hex` in `android/decoded-offer-stanza.json:90`) is: `aec`, `agc` (`{enable_uwp_agc2:true}`), `bwe`, `encode` (`{complexity:5, frame_ms:60}`), `fs`, `history_storage`, `init_bwe`, `ns`, `options` (by far the largest — hundreds of `enable_*`/relay/transport knobs), `rc`, `rc_dyn`, `re`, `sfu`, `transport_splitter`, `transport_srtp`, `vid_rc`, and `voip_settings_version {release_type:1, version_number:143382}`. This is the server-pushed engine config; the native engine consumes it directly.
- **`<relay attribute_padding="1" peer_pid="1" self_pid="3" uuid="lOw8Bt0pXJozfhY6">`** — the TURN/relay bootstrap:
  - `<participant pid="1" jid="91134994993240@lid"/>` — participant↔pid map.
  - `<token id="0|1|2">` (182 bytes each) — **per-relay credential tokens** (TURN allocation tickets).
  - `<key>` `JXoUve9WZ98BXDbxP3ByUQ==` (base64, 16 bytes decoded) — relay/HBH key seed.
  - **`<te2 relay_name="fvns6c01" c2r_rtt="32" relay_id="0" token_id="0">`** entries — **relay endpoints** (ICE candidates): `relay_name` (a datacenter code like `fvns6c01`, `del3c02`, `ccu2c01`), `c2r_rtt` (client→relay RTT in ms, used for relay election; native confirms `relay`/`c2r_rtt` election logic — `strings WhatsAppNative.dll`: `relay_measured_c2r_rtt_list`, `elected_relay_idx`, `get_relay_idx_and_proto_from_addr`/`…_from_relay_name`), `relay_id`, `token_id` (which `<token>` authorizes it), optional `protocol="1"` (the native protocol enum is `kTransportProtocolUDP`/`kTransportProtocolTCP`). The raw `<te2>` content encodes a packed IP:port (e.g. `2a032880f3150136faceb00c000000b10d96`). **The byte layout is now decoded** (cross-reference: the web bundle's own `te2` rewriter `research/waweb-unmin/DXbR99V6DKN.js:3063-3081`/`:3906-3924`): the content is a `Uint8Array` whose length selects the family — **6 bytes = IPv4** (`addr[0..3]` ‖ `port[4..5]`), **18 bytes = IPv6** (`addr[0..15]` ‖ `port[16..17]`) — i.e. a packed `addr(4|16) ‖ port(2)` `pj_sockaddr` blob. The bundle also keeps a hardcoded `relay_name`→{v4,v6} datacenter-IP table and rewrites the address bytes in place (under gate `justknobx._("360")`) before re-encoding the offer to base64-WAP for the engine. The captured `2a032880f3150136faceb00c000000b10d96` is a **plain 18-byte IPv6 `te2`** of exactly that form: `addr(16)` = `2a032880f3150136faceb00c000000b1` (a Meta `2a03:2880::/32` block) ‖ `port(2)` = `0x0D96` = **3478** (the canonical STUN/relay port). The `faceb00c` substring sits at **byte offset 8, *inside* the 16-byte IPv6 address** — it is a **placeholder mask** that the client overwrites with the real relay IPv6 from its datacenter table, **not** an appended cookie or a separate header field. This is exactly why the bundle's `length === 18` rewriter blindly replaces *all 16* address bytes (`DXbR99V6DKN.js:3073-3074`, `for (i=0;i<16;i++) n[i]=a.v6[i]`): the bytes after `faceb00c` (`000000b1`) are still address, and the final `0d96` is the port — there are **no trailing flag bytes** appended to the `te2` blob. (This reconciles with doc 92 §6 item 6 / line 219, which states the same thing oppositely-phrased.) The native engine is **PJSIP-based** and parses these via `pj_sockaddr_parse`/`pj_sockaddr_get_addr`/`pj_sockaddr_get_port` (it supports both IPv4 and IPv6: `Error: relay address must include at least one type of ipv4 or ipv6`). **Separately** — at the STUN/P2P *packet-framing* layer, not inside the `te2` address blob — `faceb00c` is also the WhatsApp **"warp" magic cookie**: the engine validates it on every inbound P2P packet (`wa_stun_verify_magic_cookie`, `Error: Dropping incoming P2P message (wrong magic cookie 0x%X)`, `has_warp_header`/`strip_warp_header_for_packets_on_p2p`, all from `strings WhatsAppNative.dll`). These warp strings describe how received P2P datagrams are framed/validated; they do **not** add a second `te2` field form. So `te2` has a single packed-address layout (`addr(4|16) ‖ port(2)`); the warp cookie is a runtime packet-framing artifact, and the literal `faceb00c` seen in the address region is the placeholder mask, not a wire header.
  - `<hbh_key>` `sw7dkuU5+wLV1BRcjEXHWpHT1wr7EkQ6ZLaUNDIh` (base64) — **hop-by-hop SRTP key** for the relay leg.

This single offer thus carries: identity (creator/caller), the E2E key handshake (`enc`), the codec/feature negotiation (`audio`/`capability`), the full engine config (`voip_settings`), and the complete relay candidate set with credentials (`relay`/`te2`/`token`/`hbh_key`). The native engine needs no separate "ICE gathering from a STUN server" round-trip for relay candidates — they are server-pushed in-band.

### 3.6 Call-key derivation — Axolotl SRTP/P2P keys (confirmed)

Once the JS Signal layer decrypts the `<enc>` payload to a per-call `cipherKey`, the native engine calls **up** into managed code via `IVoipCallbacks.CallKeysFromCipherKeyV2` to derive media keys. The managed impl is `VoipCallbacks.CallKeysFromCipherKeyV2` (`VoipCallbacks.cs:243-251`):

```
if (string.IsNullOrEmpty(jid))
    jid = Settings.Instance.GetMyPnUserJidOrLidUserJid(_currentCall?.IsLidCall == true).GetRawString(); // :248
Axolotl.CallKeysFromCipherKeyV2(jid, cipherKey, out srtpBytes, out p2pBytes);                            // :250
```

`Axolotl.CallKeysFromCipherKeyV2` (`WhatsApp.VoIP.VoIP.Temp/Axolotl.cs:10-17`):

```
byte[] okm = HkdfSha256.Perform(46, cipherKey, salt: null, info: UTF8(jid));  // :12  -> 46-byte OKM
srtpBytes = okm[0..30];   // 30-byte SRTP key block   :13,15
p2pBytes  = okm[30..46];  // 16-byte P2P key            :14,16
```

So **HKDF-SHA256(IKM=cipherKey, salt=zero[32], info=UTF8(peerJid), L=46)**, split 30/16. `HkdfSha256.Perform` (`HkdfSha256.cs:7-9`) → `Hkdf.Perform` (`Hkdf.cs:10-13`) with `hashLen=32`. `Hkdf.Extract` (`Hkdf.cs:15-22`): `PRK = HMAC-SHA256(salt||zeros, IKM)`, salt defaulting to 32 zero bytes when null. `Hkdf.Expand` (`Hkdf.cs:24-44`): standard counter-based `T(n)=HMAC(PRK, T(n-1)||info||n)`, counter starts at 1, single-byte.

**Secure SSRC.** `VoipCallbacks.GetSecureSSRC(callId, identifier, ssrcTag)` (`VoipCallbacks.cs:253-259`):
```
HKDF-SHA256(IKM=UTF8(callId), salt=LE32(ssrcTag), info=UTF8(identifier), L=4) -> uint32 (LE)
```
i.e. the RTP SSRC for each stream is cryptographically derived, not random, so both ends compute the same SSRCs from `callId`.

Other crypto primitives the native engine calls up for:
- `GetRandomBytes(length)` → `Axolotl.GenerateRandomBytes` → `CryptographicBuffer.GenerateRandom` CSPRNG (`Axolotl.cs:19-26`).
- `ComputeHmacSha256(data, key)` → `HMACSHA256` (`VoipCallbacks.cs:319-330`).
- `ComputeHkdfSha256(salt, ikm, info, okmLen)` → generic HKDF (`VoipCallbacks.cs:332-348`).

The engine also queries `GetIsKnownContact(userJid)` (`:261-272`, used for "forbid p2p for non-contact" — note `forbid_p2p_for_non_contact:1` in the captured `voip_settings`) and `GetVoipAbProps()` (`:63-69`, currently only `WinRTRenderer`).

**Three distinct media-crypto layers (confirmed in native).** The 30-byte SRTP block from `CallKeysFromCipherKeyV2` is **not** the only media-crypto layer. `strings WhatsAppNative.dll` shows the engine composes three:
1. **End-to-end media E2EE = SFrame** — the per-participant content key (`enable_sframe_tx`/`enable_sframe_rx`, `sframe_cipher_suite`, `generate_sframe_keys_for_participant`, `update_sframe_key_for_participant`, `wa_call_participant_get_sframe_key`). This is the layer that survives across relays/SFU, derived per participant from the e2e key material. Source path `xplat\wa-voip\wacall\foundation\src\e2ee\cipher\wa_sframe_cipher_aes_impl.cc` (AES+HMAC, per shared brief).
2. **Hop-by-hop SRTP on the relay leg** — keyed from the offer's `<hbh_key>`; native `fill_hbh_key`/`fill_hbh_srtp_crypto`, SRTP suites `AES_CM_128_HMAC_SHA1_80` and `AES_CM_128_HMAC_SHA1_32`, with `hbh_master_key_len`/`hbh_master_salt_len`/`hbh_warp_mi_tag` (warp message-integrity tag). HBH index signaling is negotiated (`enable_hbh_srtp_rtp_index_signaling_req_bitmap`, `kTransportMsgTypeHBHSRTPIndexReq`).
3. **P2P transport key** — the 16-byte `p2pBytes` from `CallKeysFromCipherKeyV2`, used for the direct P2P leg (`set_transport_p2p_keys`).
So the 30-byte SRTP block + per-relay `hbh_key` protect the **transport** (client↔relay), while SFrame protects the **media end-to-end** through any relay or SFU. (See §4 and §6.3.)

### 3.7 Outbound signaling — native → wire (line-by-line)

When the engine wants to send (offer/accept/terminate/rekey/relay-latency/etc.), it calls `IVoipCallbacks.OnSignalingXMPP(peerJid, callId, payload)`. Managed impl `VoipCallbacks.OnSignalingXMPP` (`VoipCallbacks.cs:841-851`) just forwards to `SignalingHandler.OnSignalingXMPP`. The real work is `VoipSignaling.OnSignalingXMPP` (`VoipSignaling.cs:49-72`):

1. Validate `callId`/`payload` (special-case `peerJidStr == "call"` for server group-call jid).
2. `ProtocolTreeNode ptn = ConvertToStanzaElement(payload)` (:56) — convert native `XmlNode` → `ProtocolTreeNode`. `ConvertToStanzaElement` (:218-308) recursively maps attributes, JID-typing each via `xmlAttr.value.jid.type` (`PhoneNumberDeviceJid`/`LidDeviceJid` → `DeviceJid`; `…UserJid` → `UserJid`; `GroupCallJid`/`GroupJid` → those JID classes). A node may have data **or** children, never both (:222-226).
3. Validate the peer bridge jid is resolvable (:59-63).
4. Serialize: `new BinTreeNodeWriter(new TokenDictionary()).Write(new PassthroughTarget(memoryStream), ptn)` → `Convert.ToBase64String(...)` (:64-66). This produces the same FunXMPP binary as on the wire (but un-Noise-encrypted — JS adds the envelope + Noise/Signal layers).
5. Determine E2E flag: `bool shouldEncrypt = (tag == "offer" || tag == "enc_rekey")` (:67-69). Only those two node types carry the Signal `<enc>` payload that must be E2E-encrypted by JS before sending.
6. `SendSignalingXmpp?.Invoke(peerJidStr, callId, base64, shouldEncrypt)` (:70) → `_voipRoot.SendSignalingXmpp` (`VoipWinRTWebBridge.cs:32-37`) → `_web.HandleSignalingXmpp(VoipSignalingXmppEventArgs.Create(peerJid, callId, xmlPayloadBase64, shouldEncryptBeforeSending))`. JS then envelopes + transmits.

`ConvertFromStanzaElement` (`VoipSignaling.cs:310-353`) is the exact inverse used on ingress (used by `VoipWebCore.DecodeBase64Wap`).

### 3.8 Native → managed events — `VoipCallbacks.OnEvent`

The native engine pushes ~160 `VoipEvent` codes (`WhatsAppNative/VoipEvent.cs`, enum 0..163) into `VoipCallbacks.OnEvent(VoipEvent ev, object data)` (`VoipCallbacks.cs:350-829`). This giant switch is the heart of managed call handling. **These up-calls arrive synchronously on the engine's own event thread, not on the VoIP worker** (the callback object is a non-agile WinRT object — §3.3, §6.8): the handler reacts inline but pushes any deferred follow-up back onto `VoipManaged.Worker` (`Worker.Dispatch(...)` at :1185-1186, `Worker.ContinueIn(...)` at :1411). Notable branches:

- **`CallStateChanged`** (:427-434) → `OnCallStateChanged(e)` (:1109-1207) updates/creates the `CallModel`, arms state timers, and forwards `JsonSerializer.Serialize(e.CallInfo)` to JS via `VoipSignaling.HandleCallEvent?.Invoke((int)ev, "", json)` (:432). The JS side receives every event as `(eventType, userData, eventDataJson)` through `VoipBridge.HandleCallEvent` (`VoipWinRTWebBridge.cs:19-24`).
- **`CallEnding`** (:367-372) → `HandleCallEnding` (:1282-1357): call-end rating gating, `AddOrUpdateCallRecord`, missed-silenced detection, uploads call-replayer time-series.
- **`CallMissed`** (:494-568): builds a `MissedCallRecord`, reports it, rejects the active call.
- **`CallOfferNacked`** (:587-627): maps `AckErrorCode` (e.g. `NackGroupCallMaximumLimit`, `NackCalleeAppVersion`, `NackCalleeLidIncompatible`, `NackGroupCallNotEnabled`, `NackAICallUnavailable`) to user dialogs and possibly ends the call.
- **`CallRejectReceived`** (:416-426) and `CallFatal` (:392-415): map reason codes; `CallFatal==2` → `OnCallAcceptFailed()`.
- **`RejectedDecryptionFailure`** (:487-493): serializes `{CallId, PeerDeviceJid, Registration, RetryCount}` to JS so JS can drive `ResendOfferOnDecryptionFailure` / rekey.
- **`SyncDevices`** (:477-486): pushes a device list to JS for multi-device fan-out.
- **`LidCallerDisplayInfo`** (:771-797): packs PN/LID/displayName/userName CSV and calls `HandleLidCallerDisplayInfo`.
- **Call-link events**: `CallLinkStateChanged` (:435-440 → :1089-1107), `LinkCreateAcked` (:682-687 builds `https://call.whatsapp.com/{video|voice}/{token}`), `LinkCreateNacked`, `LobbyNacked`/`LobbyTimeout`/`LinkQueryNacked`/`LinkJoinNacked` → user dialogs.
- **`HandleGroupCallReminder`** (:799-820): drives the group-call/link-creator reminder toast.
- **Media/UX**: `SpeakerStatusChanged` (audio levels per participant), `ScreenShare`/`ScreenContentType`, `VideoStateChanged`/`PeerVideoStateChanged`, `VideoCodecMismatch` → audio fallback, `MutedByOthers`/`MuteRequestFailed`, `UserRemoved`/`RemoveUserFailed`, `WaitingRoomDenied`.
- **Other handled branches** (also dispatched by this switch): `RelayBindsFailed` → `CallEndReason.RelayBindFailed` (:388-391), `RTCPByeReceived` (:441-443), `HandleAcceptAckFailed` (:471-476), `GroupInfoChanged`/`GroupParticipantLeft` → `UpdateCallParticipantStates` (:467-470), `LonelyStateTimeout` (:668-681), `UpdateJoinableCallLog` (:373-387), and `Update1to1CallLog` (:762-770) — call-log syncs.

`OnCallStateChanged` arms **timeouts via `BeginTimer`** (the call state machine's watchdogs), constants from `VoipManaged`:
- `ReceivedCall` → 45 s (`CalleeIncomingTimeout`, :1145-1149).
- `Calling`/`PreacceptReceived` → 90 s (`CallerAcceptWaitingTimeout`, :1158).
- `AcceptReceived` → 30 s (`CallerAcceptReceivedTimeout`, :1193).
- On `CallState.None`: cleanup, `MediaManagerTearDown` (if cached), play hangup tone `Sounds/whatsapp_windows_hangup_01_705kbps.wav` (:1250), `ClearUniverstalMuteState`.

`CallState` enum (`WhatsAppNative/CallState.cs`): `None, Calling, PreacceptReceived, ReceivedCall, AcceptSent, AcceptReceived, CallActive, CallActiveElseWhere, ReceivedCallWithoutOffer, Rejoining, Link, ConnectedLonely, PreCalling, CallStateEnding, CallBCallStarting, Max`.

### 3.9 Outgoing 1:1 call (`StartCall`)

`VoipWebCore.StartCall(peerJid, deviceJids[], callId, useVideo, phoneNumberJid, isLidCall, isFromDialer, tcTokenB64, callFromUI, lobbyEntryType, username)` (`VoipWebCore.cs:154-186`):

1. If `ActiveCall.Id == callId` → just `ReturnToCall.OnNext` (re-focus). If another call active → "in another call" dialog. Else continue (:156-164).
2. `await ValidateDevicePermissionsOrWarn(useVideo)` (:164, →`CheckDevicePermissions` :618-627 → `DevicePermissionUtils.TryGetDevicePermissions(webView, hasVideo)`).
3. `voip = await dispatcher.Capture("StartCall", …)` (:171).
4. Build `StartCallMDParams` (:172-182): `PeerJid = CreateVoipBridgeJid(peerJid, tcTokenB64)`, `DJids = deviceJids.Select(CreateVoipBridgeDeviceJid)`, `CallId`, `HasVideo`, `PhoneNumberJid`, `IsLidCall`, `IsFromDialer`, `Username`. ("MD" = multi-device; the JS-supplied `deviceJids` are the peer's device fan-out set.) The **per-device fan-out is done natively, not in the C# StartCall path**: `strings WhatsAppNative.dll` shows `call_offer_fanout_count`, `find_device_index_in_fanout_offer`, `fanout_terminate_if_peer_is_md`, and for rekeys `make_and_send_rekey_fanout_msg to %zu peers, transaction id: %d, ver: %d, len: %zu` with a `rekey_fanout_format` knob — i.e. the engine builds one `<enc>` per peer device (one offer/rekey per `DJid`) and tracks them by device index. So `StartCallMDParams.DJids` is the input set; the native engine emits the per-device encrypted offers (which surface as repeated `OnSignalingXMPP` calls, §3.7).
5. `await voip.StartCallMD(offerParams)` (:183) → engine builds + signals the `offer` (which comes back out through `OnSignalingXMPP`, §3.7).
6. `RecordOutgoingCall(callFromUI, lobbyEntryType)` (:184).

JID parsing is non-trivial. `CreateVoipBridgeJid` (`VoipWebCore.cs:391-452`) regex `^(.+?)(?:\.0)?(:\d+)?@` to split primary id and optional device id; classifies `@lid`→`LidUserJid`/`LidDeviceJid`, else `PhoneNumberUserJid`/`PhoneNumberDeviceJid`; special-cases `"call"`→`ServerGroupCallJid` and `…@call`→`GroupCallJid`. `tcToken` (trusted-contact token) is base64-decoded into bytes. `CreateVoipBridgeDeviceJid` (:454-504) splits `user:device@server`. `VoipBridgeJidType` enum (`WhatsAppNative/VoipBridgeJidType.cs`): `PhoneNumberUserJid, GroupJid, PhoneNumberDeviceJid, GroupCallJid, ServerGroupCallJid, LidDeviceJid, LidUserJid`.

## 5. Linux/Electron Port Mapping

The hybrid split is actually **favorable** for an Electron port, because the JS bundle already owns signaling transport and Signal E2E. The native `IVoip` engine is the hard part. Concrete mapping:

| Windows piece | Electron/Linux equivalent | Notes / risk |
|---|---|---|
| `VoipBridge` / `VoipSignalingBridge` host objects | `ipcMain.handle('voip:*')` + a typed `contextBridge` API in preload; renderer (= the waweb bundle) calls `window.voip.*`. | Mechanical. Mirror the exact method set in §3.2 so the **unmodified** waweb bundle keeps working. |
| `VoipWebCore` / `VoipWinRTWebBridge` (marshaling, JID parsing, base64-WAP decode) | Pure TS in the main process; reuse the project's existing `BinTreeNodeReader/Writer` + `TokenDictionary` port (doc 12) for `DecodeBase64Wap`/serialize. `VoipBridgeJid` parsing → a small TS struct + the regex from `VoipWebCore.cs:406`. | Reusable as-is; no native dep. |
| `VoipDispatcher` (serial VoIP thread) | A serial async queue (e.g. a promise chain) or a dedicated worker; the native engine library will dictate its own threading. | Whatever engine lib you pick must be called from one thread/queue. |
| **`IVoip` native engine** (ICE/TURN/SRTP/Opus/video) | **`pion/webrtc` (Go)**, **libwebrtc** (via a native node addon), or **`@roamhq/wrtc`/`werift` (TS WebRTC)**. SRTP/ICE/Opus/jitter-buffer all come "for free." | **Biggest gap.** WhatsApp's media protocol is **not standard WebRTC SDP** — it's the custom `te2`/relay/SRTP-with-derived-keys scheme. You cannot point libwebrtc at a `<te2>` relay directly. Two routes: (a) re-implement the WhatsApp media/relay framing on top of a UDP/ICE library (large), or (b) port/FFI the original native engine. Standard WebRTC libs give you DTLS-SRTP + standard ICE, which do **not** match WA's HBH-SRTP-over-relay model. |
| **SRTP/P2P key derivation** (`Axolotl.CallKeysFromCipherKeyV2`, `GetSecureSSRC`) | `@noble/hashes` HKDF-SHA256: `hkdf(sha256, cipherKey, zeros(32), utf8(jid), 46)` then slice `[0..30]`/`[30..46]`. SSRC: `hkdf(sha256, utf8(callId), le32(ssrcTag), utf8(identifier), 4)` → read LE uint32. | **Trivial, exact** — see §3.6. Implement these in TS and feed your media engine. |
| HMAC/HKDF/random up-calls | `@noble/hashes` (`hmac`, `hkdf`) + `crypto.randomBytes`. | Exact. |
| Curve25519 for `<enc>` | Already handled by waweb's libsignal in the renderer; native side only needs `Verify` → `@noble/curves` `ed25519`/`x25519` if you must. | The `<enc>` decrypt stays in JS; minimal native need. |
| Relay candidates from `<relay>/<te2>` | Parse `te2` content as a packed `addr(4\|16) ‖ port(2)` blob — length-discriminated (6 bytes = IPv4, 18 bytes = IPv6), **no trailing flags** (§3.5, §6.1); the `<token>`/`<hbh_key>` feed your relay auth + HBH-SRTP. | **Byte layout already decoded** (§3.5/§6 item 1, cross-ref doc 92): `faceb00c` is a placeholder mask *inside* the IPv6 address (not an appended cookie), and the warp magic cookie is a separate STUN packet-framing artifact. Implement the same `pj_sockaddr`-style parse. |
| Incoming-call toast + ringtone (`VoipToastController`) | Electron `Notification` (no native incoming-call scenario on Linux) + a custom always-on-top BrowserWindow for the ring UI; play ringtone via renderer `<audio loop>` or `node-speaker`. | Mic/camera permission + enumeration (`IMediaManager`/`IDeviceManager`) | `navigator.mediaDevices.enumerateDevices()` / `getUserMedia` in the renderer, or the chosen media engine's device API. | On Linux, PipeWire/PulseAudio + V4L2; Chromium (Electron) handles this. |
| Screen share (`ShareScreen(GraphicsCaptureItem)`) | `desktopCapturer` + PipeWire screencast portal (`xdg-desktop-portal`). | Wayland needs the portal; X11 simpler. |
| Network-type push (`SetNetworkType`) | `os`/`systeminformation` or NetworkManager D-Bus; feed Wifi/Cellular/None to engine. | Minor. |
| Call-log/time-series (`VoipStanzasStorage`, time-series upload) | better-sqlite3 (per doc 32) + the existing media/HTTP upload path. | Reusable. |

**Reuse from the waweb bundle:** the entire signaling transport, the Signal session/`<enc>` crypto, device-JID/PN↔LID resolution logic, and the call UI/model are already in `waweb-source-bundle/`. A port can keep all of that and only needs to supply (1) the host-object IPC surface and (2) a media engine. That is the right boundary to target first.

**Top risks, in order:** (1) the media engine — WA's relay/HBH-SRTP scheme is non-standard, so off-the-shelf WebRTC won't interoperate without re-implementing the transport; (2) implementing the (already-decoded, §3.5/§6.1) `te2`/HBH-SRTP/SFrame framing correctly on top of your chosen UDP/ICE library; (3) Linux incoming-call UX (no OS incoming-call affordance). Audio-only 1:1 is the smallest viable first milestone.

1. **[RESOLVED] `te2` content byte layout** — *Original Q: the relay endpoint blobs (e.g. `2a032880f3150136faceb00c000000b10d96`) encode IP/port/flags but the exact field packing is not decoded in any C# I read; it lives in the native engine.* **Resolved via the web bundle's own `te2` rewriter** (cross-reference: beautified waweb bundle `research/waweb-unmin/DXbR99V6DKN.js:3063-3081` function `F`, dup at `:3906-3924` function `O`). The JS reads `te2.content` as a `Uint8Array` and branches on its length: **`length === 6` ⇒ IPv4** (bytes `[0..3]` = 4-byte address, bytes `[4..5]` = 2-byte port) and **`length === 18` ⇒ IPv6** (bytes `[0..15]` = 16-byte address, bytes `[16..17]` = 2-byte port). So the base `te2` endpoint is a packed **`addr(4|16) ‖ port(2)`** blob — exactly the `pj_sockaddr`-compatible layout the native PJSIP `pj_sockaddr_parse`/`pj_sockaddr_get_addr`/`pj_sockaddr_get_port` path (confirmed in `strings WhatsAppNative.dll`) consumes. The rewriter substitutes the address bytes in place from a hardcoded `relay_name`→{v4,v6} datacenter IP table (e.g. `mia3c01.v4 = [57,144,23,57]`), leaving the trailing 2 port bytes intact; it runs under feature gate `justknobx._("360")` and is applied to the offer node **before** the stanza is re-encoded to base64-WAP and handed to the engine (`DXbR99V6DKN.js:2552-2557`). The captured longer blob `2a032880f3150136faceb00c000000b10d96` is a **plain 18-byte IPv6 `te2`** of this single form — `addr(16)` = `2a032880f3150136faceb00c000000b1` (a Meta `2a03:2880::/32` block) ‖ `port(2)` = `0x0D96` = **3478** (canonical STUN/relay port). The `faceb00c` substring is **not** an appended cookie: it sits at **byte offset 8, inside the 16-byte IPv6 address**, and is a **placeholder mask** the client overwrites with the real relay IPv6 from its datacenter table. The byte math is unambiguous — the blob is exactly 18 bytes, the bytes after `faceb00c` (`000000b1`) are still part of the address, and the final `0d96` is the port, so there are **no trailing flag bytes between `faceb00c` and the end**. The bundle's own `length === 18` branch proves this: it replaces *all 16* address bytes (`DXbR99V6DKN.js:3073-3074`, `for (i=0;i<16;i++) n[i]=a.v6[i]`), which only makes sense if `faceb00c` is part of the address. (This matches doc 92 §6 item 6 / line 219 verbatim — `faceb00c` is "a placeholder mask the client overwrites… not a separate header field… which is why F(e) blindly replaces all 16 address bytes". The two docs now agree.) The WhatsApp **"warp" magic cookie** machinery (`faceb00c` as a magic cookie validated on inbound P2P: `wa_stun_verify_magic_cookie`, `Error: Dropping incoming P2P message (wrong magic cookie 0x%X)`, `has_warp_header`/`strip_warp_header_for_packets_on_p2p`, all from `strings WhatsAppNative.dll`) is **real but operates at the STUN/P2P packet-framing layer**, on received datagrams — it does **not** define a second `te2` field form and does not appear inside the `te2` address blob. So there is one `te2` layout (`addr(4|16) ‖ port(2)`); the warp cookie is a runtime packet-framing artifact, separate from the field encoding. (Folded into §3.5.)
