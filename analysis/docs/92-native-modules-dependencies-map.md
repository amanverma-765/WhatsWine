## 1. Purpose & Scope

A single index that answers: **"for each native thing the Windows client ships, what do I use on Linux/Electron?"**

In scope:
1. The four custom native/bridge binaries (`WhatsAppNative.dll`, `WhatsAppRust.dll`, `WinRTAdapter.dll`, `WhatsAppNativeProjection.dll`) and the third-party native libs shipped in `x64/`.
2. The ~40 `WinRTAdapter.*Bridge` host-object classes exposed to JS (`x64/AppxManifest.xml:153-196`) and the ~75 `WhatsAppNative.*` activatable classes (`:197-274`).
3. A column-by-column Linux mapping: `libsignal` / `@noble/curves`, `better-sqlite3`/SQLCipher, `ffmpeg`/`libopus`, `libwebrtc`/`pion`, `libnotify`/D-Bus, Electron `Notification`/`Tray`/`protocol`/`safeStorage`.
4. Gaps, risks, and what is **reusable as-is from the waweb JS bundle**.

Out of scope: the wire-protocol/Signal/storage/media/VoIP *internals* — those have dedicated docs (`10-17`, `20-21`, `32`, `40`, `41`, `42`). This doc is the cross-cutting dependency map only.

### 3.3 Cryptographic constants the managed callers reveal (transcribe verbatim — do not paraphrase)

These are the exact numbers a port must match; all read from managed code (the native side only does Curve25519):

- **Noise/frame AEAD**: `AesGcmProvider` uses WinRT `SymmetricKeyAlgorithmProvider.OpenAlgorithm(SymmetricAlgorithmNames.AesGcm)`, `public const int TagSize = 16;` (`AesGcmProvider.cs:11`); decrypt rejects ciphertext `< 16` bytes and splits the trailing 16-byte tag (`AesGcmProvider.cs:41-49`). → Node `crypto.createCipheriv('aes-256-gcm')`, 16-byte tag.
- **HKDF**: `HkdfSha256.Perform(keyLen,key,salt,info)` → `Hkdf.Perform(key,salt,info,keyLen, 32, PerformHmac)` with `HMACSHA256` (`HkdfSha256.cs:7-14`); hash-len `32`. → `@noble/hashes/hkdf` `sha256`.
- **VoIP SRTP/P2P key derivation** (`Axolotl.CallKeysFromCipherKeyV2`, `decompiled/WhatsApp.VoIP/WhatsApp.VoIP.VoIP.Temp/Axolotl.cs:10-13`):
  ```
  sourceArray = HkdfSha256.Perform(46, cipherKey, null, UTF8(jid));
  srtpBytes   = first 30 bytes;   // p2PBytes = remaining 16
  ```
  → `hkdf(sha256, cipherKey, /*salt*/undefined, utf8(jid), 46)`, slice `[0..30]` / `[30..46]`.
- **CSPRNG**: `Axolotl.GenerateRandomBytes(n)` → `CryptographicBuffer.GenerateRandom((uint)n)` (`Axolotl.cs:19-25`). → `crypto.randomBytes`.

### 5.2 The decisive architectural fact

**The renderer payload (WhatsApp Web JS bundle) is reusable as-is** — it runs in Chromium either way and already owns: Signal X3DH/double-ratchet/sender-keys/message-AEAD, media `mediaKey` HKDF + `mmg` HTTP up/download, all signaling transport, presence/receipts/contacts model, and the QR pairing UI. The native side only handles `iq`/`success`/`failure` and the host-object capabilities. So the port's native surface collapses to:

1. **Preload shim** of `window.chrome.webview.hostObjects.*` (sync-by-default, `/Async$/` carve-out, `ignoreMemberNotFoundError`) + the legacy comma-IPC fallback. **(highest-value, doc 31 §5.2)**
2. **`SQLiteBridge.ExecuteSqlite`** over `better-sqlite3`/SQLCipher (keep the JSON `string[][]`→`SqliteResult` contract verbatim).
3. **Crypto host methods**: `AdvBridge.verify`, `ClientKey.set/get/clear`, `ServerEncKeySalt.set/get/clear`, `TcToken.*` over libsignal/`@noble` + `safeStorage`.
4. **Media services**: transcode/Mp4 over `ffmpeg`+`libopus`.
5. **VoIP engine**: the one genuinely large native rewrite.
6. **OS integration**: tray, notifications, badge, protocol, autostart, persistent-connection (replacing WNS suspend/resume).

### 5.3 Gaps & risks (ranked)

1. **VoIP media engine** — WA's `te2`/relay/HBH-SRTP-with-derived-keys is **not** standard WebRTC; off-the-shelf libs (`pion`/`libwebrtc`) give DTLS-SRTP + standard ICE that won't interoperate. Either re-implement WA media framing on a UDP/ICE lib or FFI the original engine. The `te2` relay **address** byte layout is now fully decoded — flat length-discriminated `[IPv4(4)|IPv6(16)][BE-port(2)]`, port `0x0D96`=3478, `faceb00c` warp cookie is an overwritten placeholder inside the v6 region (§6 item 6, bundle `DXbR99V6DKN.js:3063-3081`); the remaining VoIP unknowns are the relay *credential/STUN-framing* path and SFrame keying, not the candidate address parse. **Audio-only 1:1 is the smallest first milestone.**
2. **XEdDSA exactness** — `Curve25519.Sign` is XEdDSA, not Ed25519. Use `@signalapp/libsignal-client` to guarantee bit-compat; a tweetnacl Ed25519 substitute will fail server-signed verification (doc 20 §5).
3. **Sync host-object semantics** — `defaultSyncProxy=true` + the `chrome.webview` namespace are Edge-specific; the preload shim must reproduce them exactly or the bundle breaks subtly (doc 31 §5.4).
4. **SQLite cipher opacity (downgraded — mostly moot for a port)** — the at-rest **key chain is now resolved** end-to-end (decompiled-C# + live dump, docs 94/95, §6 item 2): all 7 native DBs use a **custom AES+HMAC page codec (NOT stock SQLCipher** — no `cipher_version`/`kdf_iter`/`PRAGMA key` markers anywhere, doc 96 §3) with 4096-byte pages and a 16-byte salt prefix; secrets are DPAPI(`LOCAL=user`)+`StaticKeyBytes` (session.db), PBKDF2-SHA256/AES-CBC of `ClientKey`+`SystemIdForPublisher` (nativeSettings.db), and a random `LegacyDbSecret` (data DBs). Only the page codec's exact AES mode/HMAC variant stays inside `Sqlite::Open`'s codec callees in `WhatsAppNative.dll` (`Sqlite::Open` located at `0x1807326d9`, doc 96 §3/§5). **Decisive insight:** the native DBs are a *machine-bound cache/mirror*; the real Signal/app-state keys live in **plaintext** in the WebView IndexedDB, so a renderer-hosted port inherits the store and **never needs to port the cipher at all** (doc 94 §4).
5. **Key-at-rest re-anchoring** — DPAPI + `SystemIdForPublisher` have no exact Linux analogue and are **not offline-decryptable** from an app-folder dump (machine/user-bound); but moving to `safeStorage`/`machine-id` is fine because the port re-derives nothing from the Windows DBs — it protects its own `userData` dir instead (doc 32 §5, doc 94 §4).
6. **Push/wake** — no WNS on Linux; replace suspend/resume-on-push with a persistent connection, which deletes ~60% of the notification subsystem (doc 42 §5).
7. **`WhatsAppRust.dll` identified (no longer a gap)** — it is the Meta **wamedia** media-parsing/MP4-operations engine (`strings`: `xplat\whatsapp\wamedia\rust\...`, crates `binrw`/`memchr`/`hashbrown`, zero crypto crates; §6 item 1). It maps onto the existing **`ffmpeg`** media tier (mp4 demux/repair/edit + H.264/H.265/AMR/Opus parsers) and adds **no separate port surface**. Risk: low.
8. **Notification feature parity** — quick-reply text and incoming-call scenarios are not portable to GNOME/Freedesktop; build a custom ring `BrowserWindow` and fall back to click-to-open for reply (doc 42 §5).
