# 15. Message Send Pipeline

> Target: Meta NATIVE WhatsApp for Windows (`WhatsApp.Root.exe`, WinUI 3 / Windows App SDK 1.6, v2.2607.106.0), hybrid WebView2 + native C#/C++/Rust.
> All `decompiled/...` paths below are relative to `decompiled_source/`. Every `:LINE` citation was opened and read.

## 1. Purpose & Scope

This document traces the path a user-composed chat message takes from the compose box to the wire: text/media compose → Signal (libsignal) per-device E2E encryption → serialized `WebMessageInfo` plaintext → `<message>` `ProtocolTreeNode` with per-device `<enc>` ciphertexts → FunXMPP binary serialization → optional zlib (flag `0x02`) → AES‑GCM frame crypto → length‑prefixed socket write → server `<ack>` / delivery `<receipt>` → retry on `<receipt type="retry">`.

**The single most important architectural fact, confirmed from code:** in this native Windows client the **live message-send pipeline is NOT in the native C# stack**. The compose UI, the libsignal session/double‑ratchet, `WebMessageInfo` construction, the `<message>`/`<enc>` stanza build, the WebSocket to `web.whatsapp.com`, ack tracking and retry are all in the **WhatsApp Web JS bundle** (`decompiled_source/waweb-source-bundle/`) running inside WebView2. The native C# FunXMPP send pipeline that the hints describe (`StanzaWriter` → `AesGcmEncrypt` → socket) physically exists and is complete, but it is instantiated by **exactly one** caller — Seamless Migration — to send **one** `removecompanion` IQ during logout/migration. It never carries chat `<message>` stanzas.

Evidence for that claim, established up front because it reframes the whole pipeline:

- The native `WAProtocol` receive router *drops* every top-level stanza except `iq`/`success`/`failure`: a `message`/`receipt`/`notification`/`presence`/`chatstate` node is logged `"Unrecognized top-level stanza"` and discarded (`decompiled/WhatsApp.Root/WhatsApp/WAProtocol.cs:80`, `:154`). A real chat client cannot drop inbound `message`/`receipt`; therefore native is not the chat transport.
- The *only* `new ConnectionManager(...)` (which builds the whole socket→Noise→`StanzaWriter` chain) is in `SeamlessMigrationManager.StartLogOut` (`decompiled/WhatsApp.Root/WhatsApp.SeamlessMigration/SeamlessMigrationManager.cs:563`), and the only thing it writes is `connection?.SendRemoveCompanion(...)` (`:572`).
- The host-object bridge registry exposed to the JS bundle (`AppModel.SetWebView`, `decompiled/WhatsApp.Root/WhatsApp/AppModel.cs:218-241`) contains SQLite, Voip, Wam, Contacts, MediaFiles, ClientKey, etc. — **no socket / stanza / noise / connection bridge**. JS therefore opens and owns its own message socket.

So this document covers two distinct things and labels each:
1. **(JS-owned, primary)** The real send pipeline as it exists in the waweb bundle + the native bridges it leans on (SQLite persistence, ClientKey, media, contacts). Mostly inferred from bridge contracts + WhatsApp Web protocol knowledge; the bundle is minified so exact JS symbols are not citable.
2. **(Native C#, confirmed line-by-line)** The FunXMPP serialize/compress/encrypt/frame send stack — fully readable, and the authoritative reference for the *wire format* a Linux port must reproduce, even though here it only ships the migration IQ.

## 3. How It Works

### 3.0 Two-pipeline overview

```
COMPOSE (JS, WebView2)
  └─ build Message protobuf (Conversation/ExtendedText/Image/…)
  └─ wrap as WebMessageInfo {Key{remoteJid,fromMe,id}, Message, MessageTimestamp}
  └─ for each recipient DEVICE jid:
        libsignal session encrypt(plaintext = MessageProto-or-padded) → (type, ciphertext)
            type ∈ {"msg"(2 whisper), "pkmsg"(3 prekey), "skmsg"(group sender-key)}
  └─ assemble <message to=.. id=.. type=text/media> [<enc v=2 type=..>ct</enc> ...]
  └─ JS WebSocket: FunXMPP-serialize → (zlib?) → Noise/AES-GCM → web.whatsapp.com
  ─────────────── server routes; emits <ack> then later <receipt> ───────────────
  └─ on <receipt type="retry">: re-encrypt for that device, resend

SEAMLESS MIGRATION ONLY (native C#)  ← the code the hints point at
  ConnectionManager → SocketAdapter → Noise handshake → StanzaWriter(writeKey)
  └─ SendRemoveCompanion → <iq xmlns=md type=set><remove-companion-device jid=../></iq>
  └─ BinTreeNodeWriter → (zlib flag 0/2) → AesGcmProvider.AesGcmEncrypt → FramesWriter → socket
```

### 3.1 The message envelope: `WebMessageInfo` and `MessageKey` (confirmed protobuf)

`MessageKey` identifies a message; its 4 fields and wire tags are explicit in the serializer (`decompiled/WhatsApp.Protobuf/WhatsApp.ProtoBuf/MessageKey.cs:159-183`):

| field | # | wire tag | type |
|---|---|---|---|
| `RemoteJid` | 1 | `10` | string |
| `FromMe` | 2 | `16` | bool |
| `Id` | 3 | `26` | string (the client message id, hex) |
| `Participant` | 4 | `34` | string (group sender) |

`WebMessageInfo` wraps it (`decompiled/WhatsApp.Protobuf/WhatsApp.ProtoBuf/WebMessageInfo.cs:181-255`, serializer at `:1285+`):

- field 1 tag `10` = `Key` (`MessageKey`, **required** — `:1283` throws if null) — `WebMessageInfo.cs:1285-1290`.
- field 2 tag `18` = `Message` (the content union) — `:1291-1299`.
- field 3 tag `24` = `MessageTimestamp` (uint64) — `:1300-1302`.
- field 4 tag `32` = `Status` (`ERROR=0, PENDING=1, SERVER_ACK=2, DELIVERY_ACK=3, READ=4, PLAYED=5`) — enum at `WebMessageInfo.cs:11-19`.

`Status` is the local ack state machine: a freshly composed message is `PENDING`, becomes `SERVER_ACK` on `<ack>`, `DELIVERY_ACK` on delivery `<receipt>`, `READ`/`PLAYED` on read receipts. `WebMessageInfo` is the **stored / app-state-sync** form persisted to `genericStorage.db` via `SQLiteBridge`; the bytes E2E-encrypted to a peer are the inner `Message` protobuf (§3.3), not the full `WebMessageInfo`.

### 3.2 The content union: `Message` (confirmed)

`Message` (`decompiled/WhatsApp.Protobuf/WhatsApp.ProtoBuf/Message.cs:9+`) is the polymorphic body — `Conversation` (plain text), `ExtendedTextMessage` (text + context/quote), `ImageMessage`/`VideoMessage`/`AudioMessage`/`DocumentMessage` (media: carry `mediaKey`, `url`, `directPath`, `fileEncSha256`, `fileSha256`), `ReactionMessage`, `LiveLocationMessage`, etc. Notably it also contains `SenderKeyDistributionMessage { GroupId, AxolotlSenderKeyDistributionMessage }` (`Message.cs:19-23`) — the inline SKDM piggybacked on the first group message so peers can decrypt subsequent `skmsg` sender-key payloads. For media, the bytes are uploaded to `mmg.whatsapp.net` out of band (see Media bridge) and only the keyed/hashed reference travels inside `Message`. The on-send field-population order (cross-reference: Baileys `prepareWAMessageMedia`, `Baileys/src/Utils/messages.ts:124-308`, mirroring the same upload protocol the native client uses) is: encrypt the file → derive `{mediaKey, fileEncSha256, fileSha256}`, POST the *encrypted* bytes to the mmg host, and read back `{url, directPath}` from the upload response (`messages.ts:241-244`; uploader `messages-media.ts:830-874`). The media sub-message is then filled with the keyed-reference quartet **`mediaKey` + `fileEncSha256` + `directPath` (or absolute `url`) + `mediaKeyTimestamp`** plus `fileSha256`, `fileLength`, `mimetype` and per-kind fields (`messages.ts:302-308`). `directPath` is the server-relative path; the receiver rebuilds the download URL as `https://${host}${directPath}` (`messages-media.ts:519-520`), so `directPath` and `url` are interchangeable references. The native bundle carries the same field names ([bundle] `"directPath","filehash","mediaKey","mediaKeyTimestamp","mediaStage","mimetype",...`, with a richer `"directPath","encFilehash","filehash","mediaKey","mediaKeyTimestamp",...` variant alongside a `url` slot), and these media references persist in the live WebView IndexedDB `model-storage` `message`/`message-info` stores ([live-appdata] `research/idb_schema.txt`; storage docs 94/95), with cached encrypted buffers in `lru-media-storage-idb` — the IndexedDB, not the native at-rest cache (a custom AES+HMAC page codec, **not** stock SQLCipher; docs 32/94/96), is the authoritative store (docs 94/95).

### 3.3 Per-device Signal encryption (JS/libsignal; persisted state is C#)

For a 1:1 send the JS layer runs, per recipient *device* JID:
- if a session exists → Whisper message → `<enc type="msg">` (FunXMPP token `"msg"` = primary index **77**, `WAPDefaultTokenDictionary.cs:14`).
- if no session yet → X3DH using the peer's prekey bundle → PreKeyWhisperMessage → `<enc type="pkmsg">`.
- group fan-out → Sender Keys → `<enc type="skmsg">` (token `"skmsg"` = index **50**, `WAPDefaultTokenDictionary.cs:12`), with an SKDM `<enc type="msg">`/`pkmsg` per device on first send.

In the bundle this Signal layer is the (minified-but-named) module set `WASignalCipher` / `WASignalSessions` (1:1 whisper/prekey), `WASignalGroupCipher` / `WASignalGroupSession` (sender-key groups), `WASignalSignatures` (Curve25519 sign/verify), and `WASignalLocalStorageProtocol` (persistence facade), driven from the `WAWebBackendJobs` send path. The ratchet math runs in JS / native libsignal, but the **persisted session state is the C# protobuf set** the bridge serializes to SQLite: `SessionStructure` (root key, sender/receiver chains, `MessageKeys {Index, CipherKey, MacKey, Iv}` — `decompiled/WhatsApp.Protobuf/WhatsApp.ProtoBuf/SessionStructure.cs`), `RecordStructure` (current + archived previous sessions), and `SenderKeyRecordStructure` for groups. Curve25519 ECDH/sign used by X3DH is the native `WhatsAppNative.Curve25519` (see §4). The note in the hints about “MessageKey” in the crypto sense (`{Index, CipherKey, MacKey, Iv}`) is the **chain message key inside `SessionStructure`**, distinct from the protobuf `MessageKey` of §3.1 — both names exist; do not conflate.

### 3.4 Assembling the `<message>` `ProtocolTreeNode`

The stanza model is `ProtocolTreeNode` (`decompiled/WhatsApp.Networking/WhatsApp.Networking.Nodes/ProtocolTreeNode.cs:11-19`): `string tag`, `ProtocolKeyValue[] attributes`, `ProtocolTreeNode[] children`, `byte[] data` — a node has children *or* data, never both (the writer in §3.5 emits at most one of them). Attributes carry a `KVType` JID-type flag (`ProtocolKeyValue.cs:9-33`) that drives both validation (`IsValidForType`, `:119-143`) and compact JID encoding on the wire. `ProtocolKeyValue.ToServer` is the canonical `to="s.whatsapp.net"` attr (`:35`).

The outbound message node (built in JS, mirrored by the native node model) is shaped:

```
<message to="<peer>@s.whatsapp.net" id="<hexId>" [participant="<deviceJid>"] type="text|media" [edit=".."]>
   [<enc v="2" type="pkmsg|msg|skmsg" [count=".."] [mediatype=".."] [decrypt-fail=".."]>CIPHERTEXT</enc> ... one per device]
   [<device-identity>..</device-identity>]   // adv identity — attached exactly when type=="pkmsg"
</message>
```

`<message>` is FunXMPP token `"message"` = primary index **19**, `<enc>` = index **29**, `s.whatsapp.net` = index **3**, `to` = index **17**, `id` = index **8**, `type` = index **4** (`WAPDefaultTokenDictionary.cs:7-9`). The ciphertext lives in `ProtocolTreeNode.data` of each `<enc>` and is emitted as a length-prefixed binary blob (§3.5). The client message id (`MessageKey.Id`) is generated in JS — typically uppercase-hex/random — and echoed by the server in the `<ack>`.

**Exact builders (read from the bundle, confirmed this session).** The chat-message node has no native builder (native has no chat-message path), but the JS builder is readable by module name. `WAWebBackendJobs` builds the `<message>` node via `o("WAWap").wap("message",{ id: CUSTOM_STRING(c.id), to: WAWebCommsWapMd.JID(u) /* or CHAT_JID */, participant: WAWebCommsWapMd.DEVICE_JID(l), type: WAWebE2EProtoUtils.typeAttributeFromProtobuf(a), edit: WAWebSendMsgCommonApi.editAttribute(...) }, <enc-children>)`. The bundle actually ships **two** `<message>` builder shapes (both confirmed by grep this session): a **per-device** form (`to: JID(u), participant: DEVICE_JID(l)`) and a **chat/fan-out** form (`to: CHAT_JID(L)`) that adds the current LID/PN dual-addressing attributes — `peer_recipient_pn`, `peer_recipient_lid`, `peer_recipient_username`, `recipient_pn`, `device_fanout` (set to `"false"` on resend), and `addressing_mode` — each emitted as `WAWap.DROP_ATTR` when not applicable. A Linux port that talks to current servers must emit `addressing_mode`/`peer_recipient_*` where the JS does, not just the minimal `to`/`id`/`type`/`participant` set. Each `<enc>` is `wap("enc",{ v: CUSTOM_STRING(WAWebBackendJobsCommon.CIPHERTEXT_VERSION.toString()) /* "2" */, type: CUSTOM_STRING(s), count: d===0 ? DROP_ATTR : INT(d), mediatype: WAWebBackendJobsCommon.encodeMaybeMediaType(R), "decrypt-fail": WAWebBackendJobsCommon.encodeMaybeDecryptFail(...) }, ciphertext)`. So the **exact `<enc>` attribute order is `v, type, count, mediatype, decrypt-fail`** (`count` dropped when 0; `mediatype`/`decrypt-fail` only when applicable), and the builder returns `shouldHaveIdentity = (s === CiphertextType.Pkmsg)` — the `<device-identity>` adv node is attached exactly on `pkmsg`. The per-device vs fan-out split is `WAWebSendMsgCreateDeviceStanza` / `WAWebSendMsgCreateFanoutStanza`, called from `WAWebSendMsgJob` ← `WAWebSendMsgChatAction`.

### 3.5 FunXMPP binary serialization — `BinTreeNodeWriter` (confirmed, line-by-line)

`StanzaWriter.Write(node, compress)` → `BinTreeNodeWriter.Write` (`decompiled/WhatsApp.Networking/WhatsApp/BinTreeNodeWriter.cs:44-67`), which under a lock calls `WriteInternal(node)` then `FlushBuffer` (`:69-78`). `WriteInternal` (`:80-97`) is the recursive node encoder:

1. **List header** = `1 (tag) + 2*attrCount + (data?1:0) + (children?1:0)` via `WriteListStart` (`:82`, `:420-436`): `0`→empty (`BinTag.LIST_EMPTY=0`); `<256`→`248`(`LIST_8`)+int8; else `249`(`LIST_16`)+int16 big-endian.
2. **Tag** string (`:83`, `WriteString` `:153-173`): if in token dict → 1 byte (or subdict-selector byte `236..239` + index byte); else UTF‑8 via `WriteBytes`.
3. **Attributes** (`:84`, `WriteAttributes` `:99-109`): for each, write key (token-or-bytes) then value via `WriteValueString` (`:111-151`) which special-cases JID types → `WriteJid`.
4. **Data** if present (`:85-88`, `WriteBytes`) — this is where `<enc>` ciphertext goes.
5. **Children** (`:89-96`): a nested `WriteListStart(children.Length)` then recurse.

`WriteBytes` (`:264-387`) chooses the tightest string encoding:
- len ≥ `1048576` → `254` (`BINARY_32`) + 31-bit length (`WriteInt31`).
- len ≥ `256` → `253` (`BINARY_20`) + 20-bit length.
- len < `128` and all chars are `0-9 - .` → `255` (`NIBBLE_8`) packed two nibbles/byte (`:282-318`); odd length padded with low nibble `0xF` (`:360-363`); high bit of the length byte flags the odd pad (`:366`).
- else all chars `0-9 A-F` → `251` (`HEX_8`) packed hex (`:319-356`).
- else → `252` (`BINARY_8`) + int8 length + raw UTF‑8.

`WriteJid` (`:186-254`): suffix `s.whatsapp.net`/`lid` → if a device id is present (`user.deviceSep`), emit `247` (`JID_U`) + type byte (`UserJidType`) + device byte + user-string (`:229-236`); else `250` (`JID_PAIR`) + user + server (`:243-253`). `bot` server and agent-jids have their own handling; `JID_FB` (246) is write-unsupported here. This is the compact JID encoding that makes per-device `<message to=..>` cheap.

After encoding, `FlushBuffer` rewinds `_out` to 0 and hands the `MemoryStream` to `target.WriteStanza` (`:73-77`) — `target` is the `StanzaWriter`.

### 3.6 Compression flag + AES‑GCM frame crypto — `StanzaWriter` (confirmed)

`StanzaWriter.WriteStanza(stanzaStream, useCompression)` (`decompiled/WhatsApp.Root/WhatsApp/StanzaWriter.cs:35-72`) prepends the **1‑byte frame flag** and optionally zlib-deflates:

- If `useCompression` and `len < 4 GiB`: write byte `2` (the compressed flag — `WAProtocol.FlagCompressed = 2`, `WAProtocol.cs:11`), deflate the stanza into the buffer (`SharpZipLib DeflaterOutputStream`, `:46-49`). **Only keep the compressed result if it is actually smaller** (`:50-54`); otherwise discard and fall through to uncompressed (`:56-59`).
- Uncompressed path: write byte `0`, copy the raw stanza (`:61-66`).
- Hard cap: a frame `≥ 33554432` (32 MiB) throws `IOException("Buffer too large")` (`:67-70`).

So the byte at offset 0 of the *plaintext frame* is `0x00` (raw) or `0x02` (zlib). The receiver mirrors this: `EncryptedBytesReceiver.ReceiveFrame` inflates when `(array[0] & 2) != 0` (`decompiled/WhatsApp.Root/WhatsApp/EncryptedBytesReceiver.cs:33-45`).

Then `WriteEncrypted` (`StanzaWriter.cs:74-78`):
```
frame = AesGcmProvider.AesGcmEncrypt(_writeKey, WAProtocol.LongToByteArray(_writeNonce++, 12), null, frame, 0, streamLength);
_framesWriter.WriteFrameInternal(frame);
```
Key facts:
- **Nonce** = the post-handshake write transport key’s monotonically incrementing counter `_writeNonce` (starts 0), rendered big-endian into a 12-byte buffer by `WAProtocol.LongToByteArray(value, 12)` (`WAProtocol.cs:53-66`: writes the 8-byte value right-aligned, leaving the leading 4 bytes zero).
- **AAD = `null`** for transport frames (`:76`) — unlike the Noise *handshake* frames which use the transcript hash as AAD.
- `AesGcmEncrypt` (`decompiled/WhatsApp.Root/WhatsApp/AesGcmProvider.cs:13-34`) uses WinRT `SymmetricKeyAlgorithmProvider(AesGcm)` and **appends the 16-byte auth tag to the end** of the ciphertext (`:28-32`; `TagSize = 16`, `:11`). The optional `offset/length` slice (`:17-22`) lets it encrypt just the flag+stanza region in place.

### 3.7 Length-prefix framing + socket write — `FramesWriter` (confirmed)

`FramesWriter.WriteFrameInternal(frame)` (`decompiled/WhatsApp.Networking/WhatsApp/FramesWriter.cs:21-29`):
- `MediumToByteArray(frame.Length)` = **3-byte big-endian length** (`:31-38`: `[len>>16, len>>8, len]`).
- Writes `[len3][frame]` into a scratch `WaMemoryStream`, then `_socket.Send(buffer, length)` and resets the stream (`:24-28`).

So the on-wire unit is: `[3-byte BE length] [ AES-GCM( [flag byte][FunXMPP stanza] ) ‖ 16-byte tag ]`. The 3-byte length covers the ciphertext+tag. `ISocket.Send` is the transport (SystemSocket TCP / ChunkedHttpSocket on port 80).

### 3.8 IQ id allocation and ack correlation — what the native stack *does* track (confirmed)

Outbound IQs (used by the migration path, and structurally identical to how JS tracks its own) get an id from `IqRequestsTracker.MakeId(prefix)` (`decompiled/WhatsApp.Networking/WhatsApp/IqRequestsTracker.cs:35-43`): a thread-safe `Interlocked`-incremented counter rendered `num.ToString("X")` (uppercase hex), or `prefix+num` when verbose. The handler is stored one-shot in a lock-guarded `Dictionary<string, IqResultHandler>` (`AddIqHandler`, `:27-33`). Inbound `iq result/error` pops the handler (`PopIqHandler`, `:45-55`) and dispatches via `WAProtocol.ProcessIq` (`WAProtocol.cs:158-187`: `type=="result"`→`handler.Parse`; `type=="error"`→`handler.ErrorNode`; missing `type`→`CorruptStreamException`; one-shot, so duplicates are ignored).

> For chat `<message>` the equivalent ack/retry logic lives in JS (the native router would drop `message`/`receipt`/`ack` — `WAProtocol.cs:80`, `:154`). The native `IqRequestsTracker` only correlates **IQ** stanzas.

### 3.9 The actual native send that ships: `removecompanion` (confirmed)

When this native send stack runs at all, it sends exactly this (`decompiled/WhatsApp.Root/WhatsApp/ConnectionExtensions.cs:10-56`):
```
<iq to="s.whatsapp.net" type="set" id="removecompanion_<N>" xmlns="md">
   <remove-companion-device jid="<deviceJid>" [reason="client_version_outdated"]/>
```

`SendCompanionIq` (`:10-22`) builds the node via the 4-attr ctor, registers the IQ handler, then `connection.Write(node)` (no compression) → `StanzaWriter` → §3.5/3.6/3.7. `SendRemoveCompanion` (`:24-56`) wires the result handler: child `<error>` absent → `onComplete`; present → `onError(code)`. `SeamlessMigrationManager.StartLogOut` (`SeamlessMigrationManager.cs:563-589`) drives it, retrying via `DispatchConnect(10s)` unless error is 401/403 (`:578-585`).

### 3.10 Server ack / retry (JS-owned; protocol summary)

After the JS WebSocket sends `<message>`, the server replies (per WhatsApp Web protocol; not in native C# here):
- `<ack class="message" id=.. from=.. t=..>` → JS flips `WebMessageInfo.Status` `PENDING → SERVER_ACK` and persists via `SQLiteBridge`.
- later `<receipt type="delivery"|"read"|"played" ...>` → `DELIVERY_ACK`/`READ`/`PLAYED`.
- `<receipt type="retry" .. count=N>` (peer couldn’t decrypt — usually a stale/mismatched session): JS fetches the peer’s prekey bundle, rebuilds the Signal session, re-encrypts the same `Message` for that device (often as `pkmsg`), and resends a new `<message>` with the same `MessageKey.Id`. The local message stays `PENDING` until a fresh `<ack>`.

The bundle code for this (confirmed by module name this session) is `WAWebHandleRetryRequest.handleRetryRequest({stanzaId, originalMsgId, participant, retryCount, ...})`, which pulls the peer prekey bundle via `WAWebProcessRetryKeyBundle`, re-encrypts and resends. The outbound side increments `retryCount` (`t.retryCount+=1`) and **delays the resend with exponential backoff** `WAPromiseDelays.delayMs(WABackoffUtils.expBackoff(t.retryCount, 12e4, 1e3, .1))` — base **1 s**, ceiling **120 s**, **10% jitter** ([bundle], all `expBackoff(...,12e4,1e3,.1)` call-sites). A guard module `WAWebPostMessageHighRetryCountMetric` reports once `retryCount >= MAX_RETRY`, and **`MAX_RETRY` is a clean bundle literal `var e = 5; … l.MAX_RETRY = e`** — i.e. the Windows client's configured outbound cap is **5** ([bundle] `SjCAw3j6BfscMiCaVlE8ws3ouPY_oSLXNFbdc6aC1yv_NiDGbhIdl5zyHAaImr0WiG.js:119508,119532`), not an opaque injected/A-B value as earlier supposed. This matches the interoperating-protocol cap: whatsmeow stops sending retry receipts at `retryCount >= 5` ([protocol-cross-ref] `whatsmeow/retry.go:482`) and caps inbound retry-request handling at 10 per message (`retry.go:238`); Baileys defaults `maxMsgRetryCount: 5` ([protocol-cross-ref] `Baileys/src/Defaults/index.ts:72`). A Linux port should use **5 outbound resends / 10 inbound retry-requests** as the protocol ceiling.

The native `WebMessageInfo.Status` enum (`WebMessageInfo.cs:11-19`) is the on-disk representation of exactly this state machine, which is why it is defined natively even though the transitions are driven from JS.

## 5. Linux/Electron Port Mapping

Because the live send pipeline is JS in WebView2, the **highest-leverage port strategy is to reuse the waweb bundle** under Electron’s `BrowserWindow`/`<webview>` and re-implement only the *native bridges* it expects (SQLite, ClientKey, ServerEncKeySalt, MediaFiles, Contacts, Voip, Wam) as Electron `contextBridge` host objects mirroring the `I*BridgeToNative`/`ToWeb` contracts. Under that approach the JS owns its own WebSocket to `web.whatsapp.com`, libsignal runs in JS/WASM, and you do **not** re-implement the FunXMPP send pipeline at all for chat.

If instead you build a *headless native client* (no waweb bundle), you must reproduce the wire format documented in §3.5–3.7:

| Windows piece | Linux/Electron/native-module equivalent | Notes / risk |
|---|---|---|
| `BinTreeNodeWriter`/`BinTag`/`WAPDefaultTokenDictionary` | Port the encoder + token tables verbatim to TS (`Buffer`); `whiskeysockets/Baileys` already has a faithful FunXMPP impl. | The token table is build-versioned; copy index→string exactly (`WAPDefaultTokenDictionary.cs`). |
| `StanzaWriter` flag byte (0/2) + compress | `zlib.deflateRawSync`/`inflateRawSync`; only keep compressed if smaller. | Match SharpZipLib’s raw-deflate framing; first byte `0x00`/`0x02`. |
| `AesGcmProvider` (tag appended, 12-byte BE counter nonce, AAD=null) | Node `crypto.createCipheriv('aes-256-gcm', key, nonce)`, append `getAuthTag()`. | Nonce = big-endian incrementing counter, leading 4 bytes zero; AAD `null` for transport frames (≠ handshake). |
| `FramesWriter` 3-byte BE length | `buf.writeUIntBE(len, 0, 3)` prefix. | Trivial. |
| `MessageKey`/`WebMessageInfo`/`Message` protobufs | `protobufjs` with WhatsApp `.proto`; or reuse Baileys protos. | Field tags confirmed in §3.1 (Key=1/10, FromMe=2/16, Id=3/26). |
| Signal session encrypt | `libsignal-node` / `@signalapp/libsignal-client`. | Produces `msg`/`pkmsg`; group sender-keys for `skmsg`. |
| Curve25519 | `@signalapp/libsignal-client` or `@noble/curves` (x25519 + XEdDSA). | XEdDSA sign/verify must match WhatsApp’s variant. |
| `WebMessageInfo.Status` + ack/retry | App-level state machine in TS reacting to `<ack>`/`<receipt type=retry>`. | Re-encrypt + resend same `Id` on retry. |
| genericStorage.db | `better-sqlite3` (the native at-rest layer is a **custom AES+HMAC page codec, NOT stock SQLCipher** — 4096-byte pages + 16-byte per-DB salt + raw 32-byte key, no `cipher_version`/`kdf_iter`/`PRAGMA key` markers; docs 32/94/96 — so do not replicate it, just use an app-managed key in the OS keychain if you want at-rest encryption). | Native client owns the schema; reuse-bundle approach keeps the waweb schema. | Media upload | `undici`/`fetch` to `mmg.whatsapp.net` + ffmpeg/opus transcode. | Keyed-ref (`mediaKey`,`fileEncSha256`) goes in `Message`. |

**Gaps/risks:** (1) the waweb bundle expects the exact host-object names and `windows=1`/build query args to enter native mode (`App.BuildQueryParameters`) — under Electron you must spoof or adapt that handshake or the JS may not open its socket. (2) The native FunXMPP stack here is migration-scoped, so it is *not* a complete chat client to copy from — it lacks `message`/`receipt`/`presence` handling (`WAProtocol.cs:80`); a headless port must add all of that (Baileys is the better reference). (3) AAD differs between handshake (transcript hash) and transport (null) frames — easy to get wrong.

The AES‑GCM *transport-frame* crypto is separate and goes through managed WinRT `SymmetricKeyAlgorithmProvider` (`AesGcmProvider`, §3.6) — not this native DLL. **The signature *scheme* is NATIVE-CONFIRMED as Curve25519 XEdDSA (64-byte) via radare2 (doc 96 §1):** the X25519 ladder constant `a24=121665` and SHA‑512 K[0] are statically present, and `Sign`/`Verify` live on the same `Curve25519` class that does X25519 `Derive` — the XEdDSA fingerprint. This is independently corroborated by two open impls of the same protocol: whatsmeow signs prekeys/ADV with libsignal's XEdDSA — `(*KeyPair).Sign` prepends the DjbType byte (`ecc.DjbType` = 0x05) to the 32-byte pubkey and calls `ecc.CalculateSignature(...)` returning `[64]byte` ([protocol-cross-ref] `whatsmeow/util/keys/keypair.go:48-54`, dep `go.mau.fi/libsignal`), verified with a hard 64-byte length check (`whatsmeow/handshake.go:156-168`, `prekeys.go:222-265`); Baileys signs identically via libsignal `curve.calculateSignature` ([protocol-cross-ref] `Baileys/src/Utils/crypto.ts:2,27`). So `Curve25519::Sign` is **XEdDSA over Curve25519**, not Ed25519 — byte-evidenced, not interop-only. **The only residual (and it is MOOT for the port):** the *native byte-impl* of `Curve25519::Sign` (private-key clamping, the random nonce/`Z` source fed by `BCryptGenRandom`, hash-to-scalar) is not statically readable at the instruction level. The DLL exports only `DllCanUnloadNow`/`DllGetActivationFactory` ([native-binary] `rabin2 -E`) and carries no FUNC symbol table (`objdump -t` = "no symbols"), so the `Sign` *body* is reachable only by resolving the activation factory and walking the `__ICurve25519PublicNonVirtuals` vtable into the static blob (`.text` ~9.2 MB, ~57k `.pdata` unwind entries) — but this is not needed: `@signalapp/libsignal-client` produces interoperable, bit-compatible XEdDSA signatures regardless of the native byte-impl (doc 96 §1).
