# 00. Architecture Overview

> Target: Meta **native** WhatsApp for Windows (`WhatsApp.Root.exe`, full-trust WinUI 3 / Windows App SDK 1.6, package `5319275A.WhatsAppDesktop` v`2.2607.106.0`, x64).
> This is the **synthesis** document. It builds on the per-mechanism docs in `docs/` and the primary artifacts (`x64/AppxManifest.xml`, `x64/AppxBlockMap.xml`, the decompiled C# assemblies, the `ghidra-output/` dumps, and `waweb-source-bundle/`).
> All `path:LINE` citations are **relative to `decompiled_source/`** and were read directly. "Confirmed" = read in code. "Inferred" = deduced, not directly visible. Per-mechanism detail is deferred to the cited sibling docs (e.g. doc 31 = `31-webview2-host-native-js-bridge.md`).
> A standing caveat about Ghidra: both `ghidra-output/` native dumps are empty (`WhatsAppNative-functions.txt` is 0 lines; `WhatsAppRust-functions.txt` is a PyGhidra export-script error). That tooling failure is no longer load-bearing — the native binary was **re-analyzed directly via `radare2`/`rabin2` (doc 96)** plus `strings`/`objdump`/`nm`, which together supply native **provenance** (statically-linked third-party libraries, algorithm/feature names, source paths) and the crypto **scheme** from on-binary constants and the `.WAobs` function-name section. Concretely (doc 96): the **signature scheme is now native-confirmed as X25519 ECDH + XEdDSA(SHA-512)** — the X25519 Montgomery-ladder constant `a24 = 121665` (`0x0001DB41`) and SHA-512 (`K[0]=0x428a2f98d728ae22`) are statically present, and the class exposes `Curve25519::{Derive,Sign,Verify,GenKeyPair}` (no longer "inferred by interop"; the bundle `WASignalSignatures` and Baileys/whatsmeow libsignal merely corroborate it). `@signalapp/libsignal-client` is bit-compatible. Only the *instruction-level* `Sign` body (clamping/nonce) stays unread, and that is moot for a port. The at-rest storage scheme is **pinned by a live appdata dump + decoded IndexedDB** (docs 94/95) and confirmed by radare2 (doc 96): the native DBs are a **custom AES+HMAC page codec — NOT stock SQLCipher** (no `cipher_version`/`kdf_iter`/`PRAGMA key`/`sqlcipher` markers anywhere; doc 96 §3), with 4096-byte pages + a 16-byte per-DB salt and the raw 32-byte key used directly, behind a fully-traced machine-bound key chain (DPAPI `LOCAL=user` + `GetSystemIdForPublisher`-salted PBKDF2 + random `LegacyDbSecret`); the **authoritative** Signal/app-state store is the *plaintext* WebView2 IndexedDB — so the encrypted-DB mirror is moot for a renderer-hosted port. The only residuals are (a) the page codec's exact AES mode (CTR vs CBC) / HMAC variant (moot for the port) and (b) the byte-for-byte `Curve25519::Sign` body. Also confirmed: native crypto (AES/HMAC/SHA-1/256/512/KDF) is **statically linked** (BoringSSL-from-WebRTC likely), **not** Windows CNG — `WhatsAppNative.dll`'s only `bcrypt.dll` imports are RNG (`BCryptGenRandom`/`Open`/`CloseAlgorithmProvider`) (doc 96 §2).

## 1. Purpose & Scope

This document is the top-level map of the whole client: the **hybrid model**, the **process/thread model**, the **module map** (12 C# assemblies + `WhatsAppNative` C++ + `WhatsAppRust` Rust + the `WinRTAdapter` projection), the **native↔web split**, and the **end-to-end request lifecycle** that ties the sibling docs together. It is deliberately a bird's-eye view: each subsystem has a dedicated deep-dive doc, cited inline.

The one architectural fact that governs everything below, confirmed independently in multiple sibling docs:

> **The native Windows app is a thin host. The chat/business logic is the WhatsApp Web JS bundle running inside WebView2.** Native C#/C++/Rust provides process/window ownership, OS integration, encrypted local storage, media transcode, VoIP media engine, and a handful of crypto/key primitives — exposed to JS through ~27 WinRT "bridge" host objects. The native FunXMPP/Noise/Signal stack physically exists and is fully readable, but at runtime the **chat socket, message stanzas, Signal double-ratchet, and ack/retry all live in JS**; the native transport stack is instantiated by exactly one caller (Seamless Migration) to send one `removecompanion` IQ (doc 15 §1, `WhatsApp.Root/WhatsApp.SeamlessMigration/SeamlessMigrationManager.cs:563,572`; doc 20 §1; doc 10 §1).

## 3. How It Works — the hybrid model

### 3.1 Two runtimes, one app

```
 ┌──────────────────────── WhatsApp.Root.exe (single full-trust process) ────────────────────────┐
 │                                                                                                │
 │  C# managed shell (.NET 8 / WinUI 3)                  WebView2 (Chromium render + GPU procs)    │
 │  ───────────────────────────────────                 ──────────────────────────────────────    │
 │  Program.Main [STAThread]                            web.whatsapp.com bundle (waweb)            │
 │  App : Application                                     • chat UI + business logic               │
 │   ├ AppModel  (DI/composition root)   ◄── bridges ──►  • own WebSocket to web.whatsapp.com      │
 │   ├ MainWindow (hosts ONE WebView2)                    • Noise + FunXMPP + Signal in JS         │
 │   ├ ViewsController (tray/voip/notif)                  • ack/retry, presence, receipts          │
 │   └ ~27 bridge controllers           window.chrome.webview.hostObjects.<Bridge>                 │
 │        SQLiteBridge, VoipBridge, VoipSignalingBridge, ClientKeyBridge, ServerEncKeySaltBridge,  │
 │        ContactsBridge, MediaFilesBridge, MediaTranscodingBridge, WamBridge, AdvBridge, …        │
 │              │                                                                                  │
 │              ▼  (in-proc WinRT, WinRTAdapter.dll)                                               │
 │  WhatsAppNative.dll (C++)  ── Sqlite, Curve25519, Transcoder, IVoip (74-member call engine)     │
 │       (WebRTC + openh264 + Opus media; wacall/PJSIP VoIP + SFrame/SRTP; static crypto, bcrypt=RNG only) │
 │  WhatsAppRust.dll  (Rust)  ── Meta wamedia stream parsers + mp4operations (FFI from Native)      │
 └────────────────────────────────────────────────────────────────────────────────────────────────┘
        │ WNS push channel        │ COM out-of-proc (push BG task, CLSID 082f08a8…)
        ▼                          ▼
   Windows Push (WNS)        WhatsApp.Root.exe -RegisterForBGTaskServer /pushnotification (2nd process)
```

The shell's job is narrow: **own the process, own the window, route OS activations, register bridges, and keep the WebView alive** (doc 30 §1). Everything the user sees is rendered by the JS bundle.

### 3.2 Process model

- **One full-trust process.** `Application Id="App" Executable="WhatsApp.Root.exe" EntryPoint="Windows.FullTrustApplication"` (`AppxManifest.xml:87`); capability `runFullTrust` (`:279`). Entry is `[STAThread] Program.Main` (doc 30 §3.1, `Program.cs:19`).
- **Single-instance with redirection.** `AppInstance.FindOrRegisterForKey("61E9B470-92E6-482B-AFD8-38C1FB464FB5")` (`Program.cs:37`); a secondary instance `RedirectActivationToAsync`'s its activation to the primary **unless** the command line contains `/pushnotification` (`Program.cs:45`) (doc 30 §3.1).
- **WebView2 = Chromium multi-process.** The single embedded `WebView2` spawns its own browser/render/GPU/utility processes out of the managed process's control (standard Edge WebView2 model) (doc 31 §3.2). User-data folder = `ApplicationData.Current.LocalCacheFolder.Path` (`MainWindow.cs:509`).
- **A second managed process for push.** When WNS raw-push fires while the app is gone, the OS launches `WhatsApp.Root.exe` as an **out-of-proc COM server** (`ExeServer ... Arguments="-RegisterForBGTaskServer /nowindow /pushnotification"`, CLSID `082f08a8-6f51-4fa3-9a14-7563c83d7c49`, `AppxManifest.xml:110-116`). This launch is the one that the single-instance redirect skips (doc 42 §2; doc 30 §3.10).
- **Toast-activator COM server.** A separate COM CLSID `bebd6a5e-1b14-4a57-a195-c772d28b6d15` activates the app on notification click (`AppxManifest.xml:100-124`).

### 3.3 Thread model

The shell is **not** free-threaded; it uses explicit named dispatchers with an `await dispatcher.ContinueIn(...)` idiom (doc 30 §3.13, `WhatsApp.Core`):

| Dispatcher | Kind | Used by |
|---|---|---|
| **`Dispatcher`** (`Deployment.Current.Dispatcher`) | The WinUI UI `SynchronizationContext`; queues pre-`Initialize` actions then flushes | All UI/window work (`Dispatcher.cs`) |
| **`ConcurrentQueueDispatcher`** | Serial queue drained on thread-pool threads; each component gets its own | `LoginSessionManager`, `SqliteBridge`, `WebLifecycleController`, `WebContentUpdateManager`, `ConnectionManager` (`ConcurrentQueueDispatcher.cs`) |
| **`SingleThreadedDispatcher`** | A dedicated OS thread draining a `BlockingCollection` | Where one persistent thread is required |
| **`Deployment.ThreadPool`** | `LimitedThreadPool` of 10 `ConcurrentQueueDispatcher`s round-robined | General background work (`Deployment.cs:70-79`) |
| **`VoipManaged.Worker`** | `VoipDispatcher : ICapturingThreadDispatcher<IVoip>` — a worker thread that *owns* the native `IVoip` context | **All** VoIP native calls hop onto it (`VoipManaged.cs:45`; doc 41) |

The capturing-dispatcher pattern matters for the port: the native VoIP engine is single-threaded-affine and every call must be marshaled onto its owning thread (doc 41 §3).

### 3.4 Boot sequence (cold start)

Confirmed in doc 30 §3.2–§3.4:

1. `Program.Main` → `XamlCheckProcessRequirements`, `InitializeComWrappers`, single-instance decision, `Application.Start(new App())` (`Program.cs:22-30`).
2. `App` ctor (synchronous, no heavy I/O): singleton, dispatcher init, culture, exception sinks, `PushNotificationBackgroundTask.Register`, `ViewsController`, then `StartModel()` (`App.cs:111-152`).
3. `StartModel` hops off the UI thread, builds `AppModel` (the DI/composition root) and `await model.Initialize()` (`App.cs:175-213`). `AppModel.Initialize` opens the login session, sets `IsStartedRegistered = sessionData.ClientKey != null` (the logged-in flag, `AppModel.cs:175`).
4. `EnsureMainWindowInitialized` (UI thread): `new MainWindow()`, `await MainWindow.GetWebViewCore()`, register pre-login bridges, `await _modelSource.Task` (blocks on the async model), `model.SetWebView(...)` registers the ~23 model bridges, then `MainWindow.Start(args)` navigates the WebView (`App.cs:259-341`).
5. **Native runtime bootstrap.** `NativeInterfaces.Initialize()` runs `new WhatsAppNativeInit().Setup()` once under a lock (`NativeInterfaces.cs:47-57`, projection `WhatsAppNativeInit.cs:129-131`). What `Setup()` does inside `WhatsAppNative.dll` is **inferred** (logging/SQLite/crypto/VoIP registration) — native body opaque.

### 3.5 The JS↔native bridge (the seam)

The single contract between the two runtimes. Two channels (doc 31 §3.9–§3.10):

- **Primary: WinRT host objects.** Each C# controller is wrapped by `DispatchAdapter.WrapObject` and vended via `CoreWebView2.AddHostObjectToScript(name, …)` (`WinRTAdapterExtensions.cs:13-21`). JS reaches each as `window.chrome.webview.hostObjects.<Name>`. Registered in two phases: 4 **pre-login** bridges in `App.EnsureMainWindowInitialized` (`App.cs:315-318`: `ConnectionBridge`, `TouchpadFix`, `NativeAppStateBridge`, `WebUpdateBridge`) and ~23 **post-login** bridges in `AppModel.SetWebView` (`AppModel.cs:217-242`).
- **Proxy contract the bundle hard-codes** (`U2j2EhR17gV.js`, doc 31 §3.9): `hostObjects.options.defaultSyncProxy = true` (**calls are synchronous by default**, value returned inline), `forceAsyncMethodMatches = [/Async$/, /AsyncWithSpeller$/]` (only those return promises), `ignoreMemberNotFoundError = true`. A port must replicate these exact semantics. The bundle references the host objects **by the same names as the C# bridges, 1:1** (confirmed `grep` over `waweb-source-bundle/*.js`: `hostObjects.SQLiteBridge`, `ClientKeyBridge`, `ServerEncKeySaltBridge`, `VoipBridge`, `VoipSignalingBridge`, `ContactsBridge`, `MediaFilesBridge`, `MediaTranscodingBridge`, `WamBridge`, `AbPropsBridge`, `AdvBridge`, `LinksPreviewBridge`, `SeamlessMigrationBridge`, …), using both the async `hostObjects.<Name>` form and the `hostObjects.sync.<Name>` variant (e.g. `hostObjects.sync.ContactsBridge`). Call frequency is dominated by storage: `ExecuteSqlite` is the hottest host-object method across the bundle — i.e. the SQLite bridge is the hottest part of the seam. (Note: `getTcToken` is **not** a bridge method but a JS-side server IQ via `frontendSendAndReceive`, corrected in §6 item 4.) The full per-method legacy-IPC argument schemas are now legible in the beautified bundle's `WAWebWindowsMessagePassingBridgeArgsList` (§6 item 4).
- **Each bridge is a generated triple:** `I<X>BridgeToNative` (JS→native methods), `I<X>BridgeToWeb` (native→JS callbacks, handed over via `Subscribe`), `I<X>BridgeFactory` (doc 31 §3.9).
- **Legacy fallback: comma-delimited string IPC** over `WebMessageReceived`/`PostWebMessageAsString` — `method,arg1,…,transactionId` → reflected dispatch → `transactionId,NativeResponseToJS,0,<json>` (`AppModel.cs:249-415`, `[Obsolete]`). The `bridgeError=1` query param signals the bundle this fallback exists (doc 31 §3.10). The **JS side of this exact wire format is now readable** in the beautified bundle (`WAWebWindowsMessagePassingBridge`, `research/waweb-unmin/TSxMupG87E6…js:198158-198218`): it builds the outbound frame as `mappedMethod + "," + args.join(",") + "," + transactionId`, maps the JS method name to the native one via a table (`["ExecuteSqlite","RequestExecuteSqlite"]`, …), posts it with `postMessage`, then parses responses by splitting on `,`, reading `transactionId`, asserting the literal `NativeResponseToJS`, and reading a `"0"`/non-zero status field — i.e. a 1:1 mirror of the C# `AppModel` reflected-dispatch path. The number of leading args actually sent is trimmed to the count the native side advertises via `CompatibilityGetSupportedNativeIpcs` (forward-compat negotiation).

The mnemonic: **JS owns the protocol and the crypto-on-the-wire; native owns the device** (storage, media codecs, call engine, OS surfaces, key material at rest).

### 3.7 Request lifecycle — a sent text message (end-to-end)

The canonical path stitching the docs together (doc 15 §3.0, doc 32, doc 40):

```
1. User types in compose box                                  → JS (WebView2)
2. JS builds Message protobuf → WebMessageInfo                → JS
3. JS persists to genericStorage.db                           → SQLiteBridge.ExecuteSqlite(json)  [native SQLite]
4. JS libsignal-encrypts per recipient device (msg/pkmsg/skmsg) → JS (keys read from genericStorage.db via SQLiteBridge)
   - (media) JS asks MediaFilesBridge to upload to mmg + MediaTranscodingBridge to transcode  [native]
5. JS assembles <message><enc/></message>, FunXMPP-serializes, (zlib?), Noise/AES-GCM frames → JS WebSocket
6. Server routes; emits <ack> then later <receipt>            → JS handles
7. On <receipt type="retry">: JS re-encrypts for that device, resends → JS
```

The **native shell touches this path only for persistence (SQLite), key material (ClientKey/ServerEncKeySalt bridges), and media (MediaFiles/Transcoding bridges)** — never the socket or the stanza. An *inbound* message is the mirror image: the JS WebSocket receives it, Signal-decrypts, persists via `SQLiteBridge`, and if the window is hidden the native `NotificationsManager` raises a Windows toast (doc 16, doc 42).

A **VoIP call** diverges: signaling stanzas still flow through the JS WebSocket, but JS base64-hands the inner `call` node into native via `VoipSignalingBridge`, and the native `IVoip` engine runs the actual media (doc 41 §3.1).

The Windows app is **structurally already an Electron app**: Chromium + a thin native host + a JS bundle reached over an IPC bridge. The port's job is to re-implement the *host* faithfully; **the renderer payload (waweb) is reusable as-is** (doc 30 §5, doc 31 §5).

### 5.3 Native modules + gaps

- **Native runtime** (`WhatsAppNativeInit.Setup`): the shell only needs to init its replacements once at boot; the capabilities (SQLite/crypto/media/VoIP) map to the JS/native modules above (covered by docs 20, 32, 40, 41).

## 7. Rich-text formatting subsystem (`WhatsApp.RichTextFormatting`)

`WhatsApp.RichTextFormatting.dll` is a self-contained, pure-managed text-analysis library: it takes a plain message string and produces a list of typed "chunks" (plain / bold / italic / strikethrough / code / emoji / link / email / phone / mention / list-item). It holds the **native emoji catalog**, the **emoticon shortcut table**, the **skin-tone modifier logic**, and a large set of **URL/email/phone link-detection regexes**.

> **Scope caveat (confirmed).** This subsystem is consumed **only by the native WinUI chrome**, not by the WebView2 chat. Its single runtime entry point is `WhatsApp.Design`'s `RichTextSourceFactory.CreateWithoutCaching`, which calls `TextParser.SplitIntoChunks` (`WhatsApp.Design/WhatsApp/RichTextSourceFactory.cs:42`) and renders the chunks into WinUI `RichTextBlock`s. **No `*Bridge` exposes it to JS** (grep over `WhatsApp.Root/WhatsApp.Bridge` and `WinRTAdapter` finds no rich-text/emoji/link-detect bridge). In the running hybrid app the *chat bubbles* are formatted by the waweb JS bundle. The **live** native callers traced this pass are label-only: contact-list/call-status subtitles via the no-parse `CreateFromPlainText` (`WhatsApp.Design/WhatsApp.Design/ContactListCellView.cs:588`) and the recipient-picker VMs (`WhatsApp.Root/WhatsApp.Pages.Recipients/UserRecipientItemVm.cs`, `DisplayUserRules.cs`). A **full message-body** parse path *exists* — `MessageRichTextSourceExtensions.CreateRichTextSource(this Message,…)` feeds the real body text (`message.GetText()` / `Template.HydratedContentText`) plus `GetMentionedJids()`/`GetMentionedGroupJids()` into the parser (`WhatsApp.Design/WhatsApp.Data/MessageRichTextSourceExtensions.cs:31-44`) — but it has **no decompiled caller** (`grep "CreateRichTextSource\|GetRichTextSource"` matches only that file), so it is dormant code, not a confirmed live surface (§6 item 8). For the Electron port this whole subsystem is therefore **reference, not a required bridge** — but the catalogs and regexes are wire/UX-relevant if the native shell ever renders user text (doc 31 §1, §3.9).

### 7.1 Pipeline — `TextParser` and the processor chain

`TextParser.SplitIntoChunks(inputString, formatsToDetect = DefaultFormatting, jids, groupJids)` runs a **fixed, ordered** list of `IFormattedTextProcessor`s over a mutable `TextParsingState`, each gated by a bit in the `[Flags] RichTextFormat` enum, then exports immutable `RichTextSource.Chunk`s (`WhatsApp.RichTextFormatting/WhatsApp.RichTextFormatting/TextParser.cs:9-38`). Order is load-bearing (earlier passes claim spans that later passes must skip):

`MentionsProcessor → CodeBlocksProcessor → InlineCodeProcessor → EmojiProcessor → TextFormattingProcessor → LinksProcessor → EmailsProcessor → PhoneNumbersProcessor → NewLineProcessor → ListItemProcessor` (`TextParser.cs:11-20`).

- `RichTextFormat` (`WhatsApp.Core/WhatsApp/RichTextFormat.cs:6-28`) is a `[Flags]` enum: `Link=1, Emoji=2, Bold=4, Italic=8, Code=0x10, Strikethrough=0x20, Emoticon=0x40, Foreground=0x80, Mention=0x100, NewLines=0x200, Email=0x400, PhoneNumber=0x800, InternalLinks=0x1000, InlineCode=0x2000, ListItems=0x4000, GroupMention=0x8000, ExcludedCharacter=int.MinValue`; `TextFormattings=0x3C` (= Bold|Italic|Code|Strikethrough) and `DefaultFormatting=0xAD3F` (the production mask, which **omits** `Emoticon=0x40` and `InternalLinks=0x1000`).
- Cross-pass exclusion is explicit: `EmojiProcessor` skips any span already inside `Code|InlineCode` (`Processors/EmojiProcessor.cs:15-17`); `LinksProcessor` skips `ExcludedCharacter|Mention|GroupMention` spans and clamps every match window to **4096** chars (`Processors/LinksProcessor.cs:15-24`).
- Output `TextParserResult` carries `Index/Length/Type` plus an `AuxiliaryInfo` string (normalized URL / E.164 phone) or an `AuxiliaryJid` (mentions); it asserts that mention results never carry a string and vice-versa (`TextParserResult.cs:106-136`). Original FB source paths are still embedded (e.g. `…\WebView2\WhatsApp.RichTextFormatting\Formatting\TextParserResult.cs`, `:23`).

### 7.2 Emoji catalog & detection (`Emoji`, `EmojiDetector`, `EmojiChar`)

- **`Emoji`** (`WhatsApp/Emoji.cs`) is the static catalog + scanning surface. `GetAllEmojiMatches(str)` walks the string calling `EmojiDetector.TryGetEmojiIndex` and emits `RichTextFormat.Emoji` `TextParserResult`s (`Emoji.cs:168-193`). It also holds: `TopFifty` (50 most-used emoji, `:13-20`); a 4-entry `EmojisWithVariationSelectorCodepoints` table of ZWJ couple sequences (`:26-36`); a 6-page `SoftbankCodepoints` legacy SoftBank PUA→Unicode table (90/90/83/77/76/55 entries, `:38-166`) consumed by `TryConvertSoftbankCodepoint`/`ConvertToUnicode` to upgrade legacy encodings (`:342-402`); `ToCountryEmoji` (ASCII region code → 🇫🇷-style regional-indicator pair, `:268-280`); and `IsInvalidEmojiAndVariationPair`/`IsVariantSelectorCodepointPartOfEmoji`, which strip stray U+FE00–FE0F variation selectors not part of a known emoji (`:195-266`).
- **`EmojiDetector`** (`WhatsApp.TextUtils/EmojiDetector.cs`) is the **generated trie**: a 53,296-line hand-rolled `switch` over UTF-16 code units (`TryGetEmojiIndex`) that recognizes the full emoji set without regex. Header constants record catalog size: `EmojiMappingsCount = 4733`, `SkintoneEmojiMappingsCount = 2934`, `MultiSkinToneEmojiMappingsCount = 534` (`EmojiDetector.cs:5-7`). It also exposes `IsEmoji`/`IsSkintoneEmoji`/`IsMultiSkinToneEmoji` string extensions (`:18-44`). This is the single largest file in the assembly and the most expensive thing to port faithfully (a port would substitute a Unicode emoji-data table + segmenter rather than reproduce the switch).
- **`EmojiChar`** (`WhatsApp/EmojiChar.cs`) wraps one emoji and classifies it via `EmojiType {Regular, Skintones, Couples}` (`EmojiType.cs`): setting `Codepoints` auto-detects multi-skin (`Couples`) vs single-skin (`Skintones`) emoji (`EmojiChar.cs:21-35`). `DefaultCodepoints` strips variation back to the base glyph; `ApplyVariation`/`ApplyVariations`/`ApplyMultiSkinToneSelector` re-insert one or two skin-tone modifiers (`:93-153`).

### 7.3 Skin-tone modifiers (`EmojiSkinModifier`, `EmojiCharTools`, `MultiSkinToneEmojiTools`)

- `EmojiSkinModifier` enumerates `None, Light, MediumLight, Medium, MediumDark, Dark, Blur` (`WhatsApp/EmojiSkinModifier.cs`). The five real Fitzpatrick tones map to Unicode modifiers **U+1F3FB–U+1F3FF** as the surrogate pair `D83C` + (`DFFB`+index−1); `Blur` is the private-use char **U+E92A** (`EmojiCharTools.cs:7-13,48-61`). `ToSkinModifierEnum` reverses a 2-char modifier (or the 1-char blur) back to the enum (`:15-37`).
- `EmojiCharTools.RemoveSkinTones` deletes any `D83C` + `DFFB..DFFF` pair to recover the neutral emoji (`:78-96`); `GetAllEmojiVariants` builds the 6-element tone palette (default + 5 tones) for a picker (`:63-76`).
- **Couple/handshake emoji** (e.g. 🧑‍🤝‍🧑, 👩‍❤️‍👨) carry *two* independent tones; `MultiSkinToneEmojiTools` (`WhatsApp.TextUtils/MultiSkinToneEmojiTools.cs`) holds the `DefaultToMultiSkinEmojiUnicode` map of base→ZWJ-expanded sequences (`:9-24`) and converts between the simple "default" form and the per-person "complex" form so each side can take a different tone (`EmojiChar.ApplyVariations`, `EmojiChar.cs:130-153`).

### 7.4 Emoticon shortcuts (`EmojiShortcuts`) — table only, not `:smile:` shortcodes

`EmojiShortcuts` (`WhatsApp/EmojiShortcuts.cs`) is a static dictionary mapping **classic emoticons** to emoji, exposed via `TryGetEmoji(shortcut, out emoji)` plus `GetLongestShortcut`/`GetShortestShortcut` for an incremental keystroke matcher (`:33-46`). The 20 entries are punctuation emoticons, **not** colon-delimited shortcodes: `(y)`/`(Y)`→👍, `(n)`/`(N)`→👎, `:-)`→😊, `;-)`→😉, `:-(`→🙁, `:-p`/`:-P`→😛, `;-p`/`;-P`→😜, `:-|`→😐, `8-)`→😎, `:-\`→😕, `:-D`→😁, `:-*`→😘, `:-o`→😮, `<3`→❤, `^^`→🙙, `:-O`→😯 (`EmojiShortcuts.cs:7-29`).

> **Correction vs prior recon.** The gap note called these "`:smile:`-style shortcut expansion"; the actual table is **emoticon→emoji** (`:-)`, `(y)`, `<3`), with no `:word:` colon shortcodes. Also note this maps to the `RichTextFormat.Emoticon=0x40` bit, which is **absent from `DefaultFormatting`** (§7.1) — i.e. the production native parser does not run emoticon expansion by default; the table exists for opt-in/compose paths. **No external caller of `EmojiShortcuts` exists in the decompiled C#** (the `EmojiShortcutsEnabled` flag in `WhatsApp.Design/.../ViewGlobalProps.cs:199` is an unrelated settings toggle), so live use is **inferred** to be a compose/keyboard helper.

### 7.5 Link / email / phone detection (`LinkDetectTools`)

`LinkDetectTools` (`WhatsApp/LinkDetectTools.cs`) is the regex-driven detector. It embeds an Android-`Patterns`-derived **IANA TLD list** and a family of precompiled regexes — `AUTOLINK_WEB_URL`, `QUALIFIED_URL_PATTERN`, `STRICT_TLD_PATTERN`, `AUTOLINK_EMAIL_ADDRESS`/`EMAIL_ADDRESS`, `PHONE`, and `INTERN_URL` (`LinkDetectTools.cs:70-104`). Schemes recognized: `http/https/rtsp/ftp` plus bare `www.` (`:38,84`). Key behaviors:

- `MatchAllLinks(text, matchInternalLinks, …)` caps scanned length at `MAX_LINKIFIED_TEXT_LENGTH = 4096` (`:62,123`), validates the TLD group against `STRICT_TLD_PATTERN` (`:137`), then trims trailing punctuation and balances brackets/markup via `FindUpdatedUrlRegion`/`ResolvePairedChars` so `(see example.com)` and `*example.com*` don't swallow the delimiter (`:273-447`). Each hit is normalized with `MakeUrl` (prepends `http://` / preserves scheme, `:298-318`) and returned as a `RichTextFormat.Link` result whose `AuxiliaryInfo` is the resolved href.
- `MatchAllEmails` double-checks loose matches against `AUTOLINK_EMAIL_ADDRESS` and emits `RichTextFormat.Email` (`:166-189`); `MatchAllPhoneNumbers` delegates to libphonenumber (`PhoneNumbers.PhoneNumberUtil.FindNumbers`) and stores the **E.164** form as `AuxiliaryInfo` (`:191-206,108`). `InferUriScheme` is the helper that turns a typed string into a navigable URI or `mailto:` (`:230-242`).
- `INTERN_URL` + `BUNNY_SCHEMES` (`https://www.internalfb.com/intern/bunny/?q=`) are **Meta-internal** task/diff/hash autolinks (`#1234`, `T123`, `D123`, 8–40-hex revisions) gated behind the non-default `InternalLinks` bit (`:70,86-88,244-271`) — dead weight for a port, but a useful tell that this is lifted from FB's internal text stack.
- **Link-preview relationship (corrected).** On the native side, OpenGraph previews are fetched by a *separate* bridge, `LinksPreviewBridge.GetPreviewAsync(link)` → native (`WhatsApp.Root/WinRTAdapter/LinksPreviewBridge.cs:190-193`), which takes an already-chosen URL string; it does **not** call `LinkDetectTools`. So `LinkDetectTools` *feeds link detection for native-rendered text*, and only *indirectly* relates to previews. In the live chat UI both URL detection and preview selection are done by the waweb JS bundle (doc 31 §1); the native `LinksPreviewBridge` is the metadata fetcher JS can call.

### 7.6 Chunk consumers (`RichTextSourceExtensions`, `RichTextSource`)

`RichTextSource` (`WhatsApp.Core/WhatsApp/RichTextSource.cs`) is the immutable chunk container with bidi class + `EmojiSeriesLength`; `RichTextSourceExtensions` (`WhatsApp/RichTextSourceExtensions.cs`) are the accessors the renderer uses: `TryGetUri`/`TryGetEmail`/`TryGetPhoneNumber` pull `AuxiliaryData` off link/email/phone chunks (`:9-63`), `TryGetUserJid`/`TryGetGroupJid` resolve mention chunks to JIDs via `JidFactory` (`:27-52`), `GetMentionedJids` enumerates them (`:79-88`), and `GetEmojiSeriesLength` returns 1–3 for an all-emoji message so the UI can render the "jumbo emoji" large style, while `IsSingleHeartEmoji` special-cases a lone ❤ (`:65-107`). `RichTextSourceFactory.Create` memoizes results in a 1000-entry LRU (`WhatsApp.Design/WhatsApp/RichTextSourceFactory.cs:9,32`).

### 7.7 Port notes

For the Electron/Linux port this whole assembly is **renderer-reference, not a host requirement**: the reusable waweb bundle already linkifies, emoji-renders, and markdown-formats chat text in JS. A native shell only needs an equivalent if it renders user-authored strings itself (contact subtitles, recipient pickers). If so: the emoji trie maps to a Unicode emoji-data table + grapheme segmenter; the link regexes map to a `linkify`-class library (note the 4096-cap, trailing-punctuation trim, bracket-balancing, and E.164 phone normalization as behaviors to match); skin-tone/couple logic maps to inserting `U+1F3FB–1F3FF` modifiers and ZWJ-rewriting couple sequences; the `EmojiShortcuts`/SoftBank/Bunny tables are largely droppable (the last is Meta-internal). The `:smile:`-style shortcode expansion the JS client supports is **not** in this native table — do not assume parity.
